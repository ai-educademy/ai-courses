# System Design Fundamentals

The system design interview is where senior engineers are made or broken. Unlike coding interviews with clear right answers, system design tests your ability to navigate ambiguity, make trade-offs, and communicate architectural decisions. The good news? There's a repeatable framework that works for every question. The great news? Once you understand the building blocks, you can design anything from a URL shortener to a global-scale messaging platform. Let's build that toolkit.

## The 4-Step Framework

Every system design interview should follow this structure. Interviewers expect it, and it keeps you from spiralling into premature detail.

### Step 1: Requirements Clarification (5 minutes)

Never start designing without asking questions. This is where you demonstrate product thinking.

**Functional requirements** — what the system does:
- "Should users be able to send messages to groups, or just 1:1?"
- "Do we need real-time notifications?"
- "Should the feed be chronological or ranked?"

**Non-functional requirements** — how well it does it:
- "What's the expected read/write ratio?"
- "What latency is acceptable?"
- "Do we need strong consistency or is eventual consistency okay?"

<Callout>Asking clarifying questions isn't a sign of weakness — it's the most senior-engineer thing you can do. Interviewers intentionally leave requirements vague to see if you'll clarify or blindly start designing.</Callout>

### Step 2: Back-of-Envelope Estimation (3 minutes)

Quick math grounds your design in reality and prevents over- or under-engineering.

```
Example: Design Twitter

Daily Active Users (DAU): 300M
Tweets per day: 300M × 2 = 600M tweets/day
Reads per day: 300M × 100 = 30B reads/day (read-heavy)
Tweets per second: 600M / 86400 ≈ 7,000 writes/sec
Reads per second: 30B / 86400 ≈ 350,000 reads/sec
Storage per tweet: ~300 bytes text + metadata ≈ 500 bytes
Daily storage: 600M × 500B = 300GB/day ≈ 110TB/year
```

These numbers tell you: this system is extremely read-heavy, needs caching, and will accumulate massive storage.

### Step 3: High-Level Design (10 minutes)

Draw the architecture with major components. Start simple, then add complexity as needed.

```
Client → Load Balancer → API Servers → Service Layer
                                          ↓
                              ┌───────────┼───────────┐
                              ↓           ↓           ↓
                           Database     Cache      Message Queue
                                          ↓           ↓
                                       CDN        Workers
```

Walk through the main user flows:
- "When a user posts a tweet, the API server validates it, writes to the database, and pushes to the fan-out service..."
- "When a user opens their feed, we first check the cache..."

### Step 4: Deep Dive (15 minutes)

The interviewer will pick 1-2 areas to go deep. Be ready to discuss trade-offs in detail.

<ThinkAboutIt>If you were designing a notification system, would you push notifications to all followers immediately (fan-out on write) or compute the feed when each user requests it (fan-out on read)? What are the trade-offs for users with millions of followers?</ThinkAboutIt>

## The Building Blocks

Think of these as Lego pieces. Every system design is a different arrangement of the same blocks.

### Load Balancers

Distribute traffic across multiple servers. Know three strategies:
- **Round Robin** — simple, equal distribution
- **Least Connections** — sends to the server with fewest active connections
- **Consistent Hashing** — routes based on request content (useful for caching)

### Caches

The single most impactful performance optimisation. Remember these cache patterns:

```
Write-through:  App → Cache → DB  (consistent, higher write latency)
Write-back:     App → Cache ... → DB  (fast writes, risk of data loss)
Cache-aside:    App checks cache → miss → read DB → populate cache
```

**Cache eviction**: LRU (Least Recently Used) is the default answer in interviews. Mention TTL (Time-To-Live) for time-sensitive data.

<FunFact>According to a famous paper by Jeff Dean, a single cache miss that goes to disk instead of memory adds about 100x latency. This is why caching strategy is often the most impactful design decision — reading from cache (1ms) vs reading from disk (10ms) vs a cross-datacenter read (150ms) are different orders of magnitude.</FunFact>

### Databases

| Type | Use When | Examples |
|---|---|---|
| **Relational (SQL)** | ACID transactions, complex queries, structured data | PostgreSQL, MySQL |
| **Document Store** | Flexible schemas, nested data, rapid iteration | MongoDB, DynamoDB |
| **Wide-Column** | Massive scale, time-series, write-heavy | Cassandra, HBase |
| **Graph** | Relationship-heavy data, social networks | Neo4j, Neptune |

**Database scaling**:
- **Vertical scaling**: bigger machine (limited ceiling)
- **Read replicas**: separate read and write traffic
- **Sharding**: partition data across multiple databases (by user ID, geography, etc.)

### Message Queues

Decouple producers from consumers. Essential for:
- Asynchronous processing (send email after signup)
- Load levelling (absorb traffic spikes)
- Event-driven architectures

```
Producer → [Message Queue] → Consumer
           (Kafka, SQS,      (Workers that
            RabbitMQ)          process at
                               their own pace)
```

### CDNs (Content Delivery Networks)

Cache static content (images, CSS, JS) at edge servers close to users. Mention CDNs whenever the system serves media or has a global user base.

## Classic System Design Questions

### Design a URL Shortener (e.g., bit.ly)

**Key decisions**:
- Generate short codes using base62 encoding (a-z, A-Z, 0-9)
- Use a counter or hash-based approach for uniqueness
- Read-heavy → cache popular URLs aggressively
- Database: simple key-value store (short_code → original_url)

```
POST /shorten  { "url": "https://example.com/very/long/path" }
→ Returns: { "short_url": "https://bit.ly/abc123" }

GET /abc123
→ 301 Redirect to original URL (301 = permanent, cacheable by browser)
```

### Design a Chat System (e.g., WhatsApp)

**Key decisions**:
- WebSockets for real-time bidirectional communication
- Message storage: write-optimised database (Cassandra)
- Online presence: heartbeat mechanism with timeout
- Group messages: fan-out to all group members via message queue

### Design a News Feed (e.g., Twitter/Facebook)

**Key decisions**:
- **Fan-out on write** for users with < 10K followers (pre-compute feeds)
- **Fan-out on read** for celebrity users (compute on request to avoid writing to millions of feeds)
- Hybrid approach: most systems use both strategies
- Ranking: ML-based relevance scoring over pure chronological order

## Handling Estimation Questions

Interviewers love asking "how many X does Y need?" Use these anchors:

| Metric | Value |
|---|---|
| QPS for a single web server | ~1,000-10,000 |
| QPS for a single database | ~1,000-5,000 |
| Cache read latency | ~1ms |
| Database read latency | ~5-10ms |
| Cross-datacenter latency | ~100-150ms |
| 1 million seconds | ~12 days |
| 1 billion seconds | ~32 years |

## Communication Tips During System Design

Your communication matters as much as your design. Practise these habits:

1. **Structure your thoughts**: "Let me start with requirements, then estimation, then high-level design."
2. **State trade-offs explicitly**: "We could use SQL for consistency or NoSQL for scale. Given our read-heavy workload, I'd lean toward NoSQL with a caching layer."
3. **Acknowledge what you don't know**: "I'm not sure about the exact replication factor for Kafka, but I know it's configurable and typically set to 3."
4. **Check in with the interviewer**: "I've covered the high-level design. Would you like me to deep-dive into the database schema or the caching strategy?"

<Callout>The biggest mistake in system design interviews is going too deep too early. Resist the urge to design the database schema in the first 5 minutes. Start wide, then narrow based on what the interviewer wants to explore.</Callout>

## Key Takeaways

- **Follow the 4-step framework**: requirements → estimation → high-level design → deep dive. It works every time.
- **Know your building blocks**: load balancers, caches, databases, message queues, and CDNs are the vocabulary of system design.
- **Trade-offs are the point**: there are no perfect designs — only trade-offs. Articulate them clearly.
- **Estimation grounds your design**: quick back-of-envelope math prevents you from over-engineering or under-engineering.
- **Communication is half the grade**: structure your thoughts, state assumptions, and check in with the interviewer regularly.
- **Practise with real systems**: study how Twitter, WhatsApp, YouTube, and Uber actually work. Their engineering blogs are gold mines.
