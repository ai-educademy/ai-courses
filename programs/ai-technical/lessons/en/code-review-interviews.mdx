# Code Review Interviews

Code review interviews are one of the most underrated rounds in the hiring process — and one of the most revealing. While coding interviews test if you can write code, code review interviews test if you can read it, reason about it, and improve it. They reveal how you think about quality, how you prioritise feedback, and how you communicate with other engineers. Many candidates prepare for months on LeetCode but walk into a code review round completely unprepared. Don't be that person.

## What Code Review Interviews Actually Test

Interviewers evaluating your code review skills are looking for five things:

1. **Bug detection** — Can you spot logical errors, edge cases, and race conditions?
2. **Design sense** — Do you recognise violations of SOLID principles, poor abstractions, or code that will be hard to maintain?
3. **Security awareness** — Can you identify injection vulnerabilities, authentication flaws, or data exposure risks?
4. **Communication skill** — Can you explain issues clearly and constructively?
5. **Prioritisation** — Do you focus on what matters most, or do you bikeshed on variable names?

<Callout>The most common mistake in code review interviews is spending all your time on style issues (naming, formatting) while missing actual bugs. Interviewers deliberately plant critical bugs alongside style issues to see what you prioritise.</Callout>

## The Systematic Review Approach

Don't scan code randomly. Use a structured pass-based approach:

### Pass 1: Understand the Intent (2 minutes)

Before critiquing anything, understand what the code is trying to do:
- Read the function/class names and comments
- Identify the inputs and outputs
- Understand the business logic

### Pass 2: Correctness (5 minutes)

This is where you find bugs. Check systematically:
- **Edge cases**: empty inputs, null values, single elements, maximum values
- **Off-by-one errors**: loop boundaries, array indexing
- **Logic errors**: incorrect conditions, missing break statements, wrong operator
- **Concurrency issues**: shared mutable state, race conditions
- **Error handling**: uncaught exceptions, missing null checks

### Pass 3: Design and Maintainability (3 minutes)

Look for structural issues:
- Functions doing too many things (violating Single Responsibility)
- Tight coupling between components
- Magic numbers and hardcoded values
- Missing abstractions or unnecessary abstractions
- Code duplication

### Pass 4: Performance and Security (2 minutes)

- Unnecessary database queries in loops (N+1 problem)
- Missing input validation or sanitisation
- Exposed sensitive data in logs or responses
- Inefficient algorithms where better alternatives exist

## Common Bug Patterns to Spot

Train your eyes to catch these high-frequency bugs:

### Null Reference Errors

```java
// Bug: user could be null
public String getUserEmail(Long userId) {
    User user = userRepository.findById(userId);
    return user.getEmail();  // NullPointerException if user not found
}

// Fixed
public String getUserEmail(Long userId) {
    return userRepository.findById(userId)
        .map(User::getEmail)
        .orElseThrow(() -> new UserNotFoundException(userId));
}
```

### Off-by-One Errors

```python
# Bug: misses the last element
def find_max(numbers):
    max_val = numbers[0]
    for i in range(1, len(numbers) - 1):  # Should be len(numbers)
        if numbers[i] > max_val:
            max_val = numbers[i]
    return max_val
```

### Resource Leaks

```java
// Bug: connection is never closed if an exception occurs
public List<User> getUsers() {
    Connection conn = dataSource.getConnection();
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery("SELECT * FROM users");
    // process results...
    conn.close();
    return users;
}

// Fixed: use try-with-resources
public List<User> getUsers() {
    try (Connection conn = dataSource.getConnection();
         Statement stmt = conn.createStatement();
         ResultSet rs = stmt.executeQuery("SELECT * FROM users")) {
        // process results...
        return users;
    }
}
```

### SQL Injection

```python
# Bug: SQL injection vulnerability
def get_user(username):
    query = f"SELECT * FROM users WHERE username = '{username}'"
    return db.execute(query)

# Fixed: parameterised query
def get_user(username):
    query = "SELECT * FROM users WHERE username = %s"
    return db.execute(query, (username,))
```

<FunFact>A study of over 100,000 code reviews at Google found that about 25% of all review comments are about "correctness" — actual bugs or logic errors. Another 25% are about "comprehension" — making code easier to understand. Style and formatting accounted for less than 10% of substantive comments.</FunFact>

### Concurrency Bugs

```java
// Bug: race condition — check-then-act is not atomic
public void transferMoney(Account from, Account to, BigDecimal amount) {
    if (from.getBalance().compareTo(amount) >= 0) {
        // Another thread could modify balance here
        from.setBalance(from.getBalance().subtract(amount));
        to.setBalance(to.getBalance().add(amount));
    }
}
```

<ThinkAboutIt>How would you fix the concurrency bug above? Consider at least three approaches: synchronisation, optimistic locking, and database-level transactions. What are the trade-offs of each in a high-throughput system?</ThinkAboutIt>

## Prioritising Your Feedback

In a time-limited interview, not all feedback is equal. Use this priority framework:

| Priority | Category | Example |
|---|---|---|
| 🔴 **Critical** | Bugs, security vulnerabilities, data loss | SQL injection, race conditions, null dereference |
| 🟠 **High** | Design issues, performance problems | N+1 queries, missing error handling, tight coupling |
| 🟡 **Medium** | Maintainability, readability | Complex conditionals, missing documentation, magic numbers |
| 🟢 **Low** | Style, minor improvements | Variable naming, import ordering, minor refactors |

In an interview, spend 80% of your time on 🔴 and 🟠 items. Mention 🟡 items briefly. Only touch 🟢 items if you have time remaining and have already found the important issues.

## Communicating Review Feedback

How you deliver feedback matters as much as the feedback itself. Interviewers are evaluating your collaboration skills.

**Do this**:
- "I notice this query is inside a loop, which could cause N+1 performance issues. We might want to batch this into a single query."
- "This looks like it could throw a NullPointerException if the user isn't found. Could we add a null check or use Optional here?"
- "Nice use of the strategy pattern here — it makes adding new payment types straightforward."

**Not this**:
- "This code is wrong." (too blunt, no explanation)
- "I would never write it this way." (ego-driven)
- "Change this variable name." (bikeshedding on low-priority items)

<Callout>Always explain the *why* behind your feedback, not just the *what*. "This should use parameterised queries" is okay. "This should use parameterised queries because the current string concatenation is vulnerable to SQL injection attacks" is much better.</Callout>

## Example Code Review Walkthrough

Here's a function you might be asked to review in an interview:

```python
def process_orders(orders):
    results = []
    for order in orders:
        user = db.query(f"SELECT * FROM users WHERE id = {order['user_id']}")
        if user:
            total = 0
            for item in order['items']:
                price = db.query(f"SELECT price FROM products WHERE id = {item['product_id']}")
                total = total + price[0] * item['quantity']
                if item['quantity'] > 100:
                    total = total * 0.9
            if total > 0:
                results.append({'user': user, 'total': total, 'order_id': order['id']})
    return results
```

**A strong review would identify:**

1. 🔴 **SQL Injection** (lines 4, 7): String interpolation in SQL queries. Use parameterised queries.
2. 🔴 **N+1 Query Problem** (lines 4, 7): Two database queries per order, one per item. For 100 orders with 10 items each, that's 1,100 queries. Batch into single queries.
3. 🟠 **No error handling**: `price[0]` will crash if the product isn't found. `db.query` could fail.
4. 🟠 **Discount logic is hidden**: The 10% discount for quantity > 100 is a business rule buried in nested loops. Extract it.
5. 🟡 **Magic number**: `0.9` and `100` should be named constants (`BULK_DISCOUNT_RATE`, `BULK_THRESHOLD`).
6. 🟡 **Function does too much**: fetching users, calculating totals, applying discounts — violates Single Responsibility.

## Key Takeaways

- **Use a structured approach**: multiple passes (intent → correctness → design → performance/security) catch more issues than random scanning.
- **Prioritise ruthlessly**: critical bugs and security issues first, style last. Interviewers plant both to test your judgement.
- **Communicate constructively**: explain the *why*, suggest alternatives, and acknowledge good patterns you see.
- **Know the common bug patterns**: null references, off-by-one errors, resource leaks, SQL injection, and race conditions appear in nearly every code review interview.
- **Practice reading code**: spend time reading open-source pull requests on GitHub. The skill of reading code is distinct from writing it, and most engineers don't practise it enough.
