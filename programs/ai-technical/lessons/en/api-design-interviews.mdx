# API Design Interviews

APIs are contracts between systems, and designing them well is a skill that separates senior engineers from juniors. In an API design interview, you're not just writing endpoints — you're demonstrating that you understand how consumers will use your system, how it will evolve over time, and where the failure modes are. Whether you're asked to design a payment gateway or a social media platform API, the principles are the same. Let's master them.

## REST API Design Principles

REST remains the dominant API paradigm in interviews. The foundations you must demonstrate:

### Resource-Oriented Design

Think in nouns (resources), not verbs (actions). URLs should represent things, and HTTP methods should represent actions on those things.

```
✅ Good:
GET    /users/123              — Fetch user 123
POST   /users                  — Create a new user
PUT    /users/123              — Replace user 123
PATCH  /users/123              — Partially update user 123
DELETE /users/123              — Delete user 123

❌ Bad:
GET    /getUser?id=123
POST   /createUser
POST   /deleteUser/123
```

<Callout>A clean resource hierarchy tells the API's story. `/users/123/orders/456/items` reads naturally: "User 123's order 456's items." If your URL structure doesn't read like a sentence, reconsider your resource design.</Callout>

### HTTP Methods and Status Codes

Know these cold — interviewers will test your precision:

| Method | Idempotent? | Use Case |
|---|---|---|
| GET | Yes | Read a resource (never mutate) |
| POST | No | Create a resource |
| PUT | Yes | Full replacement of a resource |
| PATCH | No* | Partial update |
| DELETE | Yes | Remove a resource |

*PATCH is technically not guaranteed idempotent, though implementations often are.*

**Status codes you must know**:
- `200 OK` — Success with response body
- `201 Created` — Resource created (include `Location` header)
- `204 No Content` — Success with no response body (common for DELETE)
- `400 Bad Request` — Client sent invalid data
- `401 Unauthorized` — Authentication required
- `403 Forbidden` — Authenticated but not authorised
- `404 Not Found` — Resource doesn't exist
- `409 Conflict` — Conflict with current state (duplicate, version mismatch)
- `429 Too Many Requests` — Rate limited
- `500 Internal Server Error` — Something broke on our side

### Pagination

Never return unbounded lists. Three common patterns:

```json
// Offset-based (simple but poor for large datasets)
GET /users?offset=20&limit=10

// Cursor-based (preferred for feeds and large datasets)
GET /users?cursor=eyJ0aW1lIjoiMjAyNC0wMS0wMSJ9&limit=10

// Response includes pagination metadata
{
  "data": [...],
  "pagination": {
    "next_cursor": "eyJ0aW1lIjoiMjAyNC0wMS0wMiJ9",
    "has_more": true
  }
}
```

<ThinkAboutIt>Why is cursor-based pagination preferred over offset-based for large, frequently-updated datasets? Consider what happens when new items are inserted while a user is paginating through results.</ThinkAboutIt>

### Versioning

APIs evolve. Design for it from day one.

```
// URL versioning (most common, explicit)
GET /v1/users/123
GET /v2/users/123

// Header versioning (cleaner URLs, less discoverable)
GET /users/123
Accept: application/vnd.myapi.v2+json
```

In interviews, URL versioning is the safe answer unless the interviewer specifically explores alternatives.

### Error Responses

Consistent, informative error responses separate great APIs from frustrating ones:

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "The request body contains invalid fields.",
    "details": [
      {
        "field": "email",
        "message": "Must be a valid email address."
      },
      {
        "field": "age",
        "message": "Must be a positive integer."
      }
    ],
    "request_id": "req_abc123"
  }
}
```

Always include a `request_id` for debugging and a machine-readable `code` alongside the human-readable `message`.

## GraphQL in Interviews

GraphQL questions are increasingly common, especially at companies with complex frontend needs.

**Key advantages to articulate**:
- Clients request exactly the data they need (no over-fetching)
- Single endpoint reduces round trips
- Strongly typed schema serves as documentation

```graphql
# Schema definition
type User {
  id: ID!
  name: String!
  email: String!
  orders(first: Int, after: String): OrderConnection!
}

type Order {
  id: ID!
  total: Float!
  status: OrderStatus!
  items: [OrderItem!]!
}

# Query — client specifies exactly what it needs
query {
  user(id: "123") {
    name
    orders(first: 5) {
      edges {
        node {
          total
          status
        }
      }
    }
  }
}
```

**Trade-offs to discuss**: GraphQL adds complexity on the server side (resolver chains, N+1 queries, caching is harder). For simple CRUD APIs, REST is often the better choice.

<FunFact>GraphQL was developed internally at Facebook in 2012 to solve a specific mobile performance problem: the News Feed required data from many different backend services, and making multiple REST calls from mobile devices on slow networks was killing performance. It was open-sourced in 2015.</FunFact>

## API Security Basics

Every API design interview should touch on security. Cover these essentials:

### Authentication and Authorisation

```
Authentication (AuthN): WHO are you?
→ API Keys, OAuth 2.0, JWT tokens

Authorisation (AuthZ): WHAT can you do?
→ Role-based (RBAC), Scope-based, Resource-based
```

```
// JWT in Authorization header
GET /users/123/orders
Authorization: Bearer eyJhbGciOiJSUzI1NiIs...

// API Key (for server-to-server)
GET /analytics/reports
X-API-Key: sk_live_abc123...
```

### Rate Limiting

Protect your API from abuse and ensure fair usage:

```
// Rate limit headers in response
HTTP/1.1 200 OK
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 847
X-RateLimit-Reset: 1640995200

// When limit is exceeded
HTTP/1.1 429 Too Many Requests
Retry-After: 60
```

Discuss rate limiting strategies: per-user, per-IP, per-endpoint, and tiered limits for different API plans.

## Real Interview Scenarios

### Design a Payment API

Requirements to clarify:
- What payment methods? (Card, bank transfer, wallet)
- Do we need idempotency? (Yes — always for payments)
- What about refunds and disputes?

```
POST /v1/payments
{
  "amount": 4999,           // Always use smallest currency unit (cents)
  "currency": "GBP",
  "payment_method": "pm_card_visa_123",
  "idempotency_key": "order_789_attempt_1",
  "metadata": {
    "order_id": "order_789"
  }
}

Response: 201 Created
{
  "id": "pay_abc123",
  "status": "processing",
  "amount": 4999,
  "currency": "GBP",
  "created_at": "2024-01-15T10:30:00Z"
}
```

<Callout>For payment APIs, always discuss idempotency keys. Network failures happen, clients retry, and without idempotency, you'll charge customers twice. The idempotency key ensures that retrying the same request produces the same result.</Callout>

**Key design decisions to discuss**:
- Amounts in smallest currency unit (cents/pence) to avoid floating-point issues
- Idempotency keys for safe retries
- Webhook callbacks for async status updates
- PCI compliance considerations

### Design a Social Media API

```
// Post creation
POST /v1/posts
{
  "content": "Hello world!",
  "media_ids": ["media_123"],
  "visibility": "public"
}

// Feed retrieval with cursor pagination
GET /v1/feed?cursor=abc&limit=20

// Interactions
POST /v1/posts/{post_id}/likes
DELETE /v1/posts/{post_id}/likes
POST /v1/posts/{post_id}/comments
```

Discuss filtering, search, and how the feed is assembled (pull vs push model ties back to system design).

## Common Mistakes in API Design Interviews

1. **Exposing internal implementation** — Don't let database column names or internal IDs leak into the API. Use DTOs.
2. **Inconsistent naming** — Pick `snake_case` or `camelCase` and stick with it everywhere.
3. **Missing pagination** — Any list endpoint must support pagination.
4. **No error handling discussion** — Show you think about failure modes.
5. **Ignoring backwards compatibility** — Adding required fields to existing endpoints breaks clients.
6. **Using verbs in URLs** — `/createUser` instead of `POST /users`.

## Key Takeaways

- **Think in resources**: design URLs around nouns, let HTTP methods handle the verbs.
- **Know your status codes**: precise status codes show you understand HTTP semantics.
- **Always paginate**: unbounded list responses are a red flag in interviews.
- **Design for failure**: idempotency, error responses, and rate limiting show production-readiness thinking.
- **Security is not optional**: authentication, authorisation, and rate limiting should be part of every API design discussion.
- **Articulate trade-offs**: REST vs GraphQL, consistency vs flexibility, simplicity vs power — interviewers want to hear your reasoning.
