# Live Coding Best Practices

You can solve LeetCode mediums in your sleep, but put a human being on the other end of a screen share and suddenly your fingers forget where the semicolons go. Live coding interviews are a performance — one where you're judged not just on the final code, but on how you get there. The candidates who pass aren't always the fastest coders; they're the ones who communicate clearly, handle setbacks gracefully, and write code that's readable under pressure. This lesson teaches you to be that candidate.

## Setting Up Your Environment

First impressions matter. When you open CoderPad, HackerRank, or VS Code Live Share, the interviewer is already forming opinions.

### Platform-Specific Tips

**CoderPad** (most common):
- Choose your language from the dropdown before the interview starts
- Use the built-in Run button — know that output appears in the console pane
- Auto-indent and syntax highlighting are available

**HackerRank**:
- Read the problem statement carefully — inputs come from stdin
- The test cases run automatically; understand the expected format
- Custom test cases can be added in the left panel

**VS Code Live Share / Google Docs**:
- No auto-complete or syntax highlighting (possibly)
- Indent consistently by hand — it shows discipline
- For Google Docs: use a monospace font (Courier New)

<Callout>Before any live coding interview, do a 5-minute environment check. Open the platform, write a "Hello World", and run it. Fumbling with the platform during the interview wastes precious minutes and shakes your confidence.</Callout>

### Language Choice

Pick the language you're most fluent in, not the one you think sounds impressive. That said, here's what to consider:

| Language | Pros | Cons |
|---|---|---|
| **Python** | Concise, rich standard library, fast to write | Slower runtime, type ambiguity |
| **Java** | Strong typing, interviewers expect it for system design | Verbose, boilerplate |
| **JavaScript** | Good for frontend roles, flexible | Quirky type coercion, less common for algorithms |
| **C++** | Fast runtime, shows low-level skills | Error-prone, verbose STL |

## The Talk-Aloud Protocol

This is the single most impactful interview skill, and the one most candidates neglect. Your inner monologue needs to become an outer monologue.

### What to Say at Each Stage

**Reading the problem** (minutes 0-3):
- "Let me make sure I understand the problem: we need to find..."
- "A few clarifying questions: can the input be empty? Can there be negative numbers?"
- "Let me think of a few examples to verify my understanding."

**Planning your approach** (minutes 3-7):
- "My first thought is a brute-force approach using nested loops — that would be O(n²)."
- "I think we can do better with a hash map to track elements we've seen."
- "Let me think about edge cases: empty input, single element, all duplicates..."

**Writing code** (minutes 7-25):
- "I'll start by setting up the function signature and handling edge cases."
- "Now I'm iterating through the array, and for each element I'm checking..."
- "I'm choosing a hash map here because we need O(1) lookups."

**Testing** (minutes 25-30):
- "Let me trace through with the example: for input [2, 7, 11, 15] with target 9..."
- "I should also check the edge case of an empty array..."
- "Hmm, my output for this case doesn't match. Let me look at line 5..."

<ThinkAboutIt>Think about the last time you solved a problem silently. What was your internal decision process? How would you externalise each of those micro-decisions? Practise verbalising your thinking even when solving problems alone — it becomes natural during the interview.</ThinkAboutIt>

## Writing Clean Code Under Pressure

Under stress, code quality is the first casualty. Fight this with deliberate habits:

### Use Meaningful Names

```python
# Under pressure (what many candidates write)
def solve(a, t):
    d = {}
    for i, v in enumerate(a):
        if t - v in d:
            return [d[t - v], i]
        d[v] = i

# What you should write
def two_sum(nums, target):
    seen = {}
    for index, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], index]
        seen[num] = index
```

The second version takes 10 extra seconds to write but makes debugging 10x easier — and impresses the interviewer.

### Start with the Skeleton

Before filling in logic, write the structure:

```python
def merge_intervals(intervals):
    # Edge case
    if not intervals:
        return []

    # Sort by start time
    intervals.sort(key=lambda x: x[0])

    # Merge overlapping intervals
    merged = [intervals[0]]
    for current in intervals[1:]:
        # TODO: merge logic
        pass

    return merged
```

This shows the interviewer your plan before you implement it. It also means even if you run out of time, they see a correct high-level approach.

### Helper Functions Are Your Friend

```python
# Extract reusable logic into named functions
def is_valid_move(board, row, col):
    return 0 <= row < len(board) and 0 <= col < len(board[0])

def get_neighbours(row, col):
    return [(row + dr, col + dc) for dr, dc in [(0, 1), (0, -1), (1, 0), (-1, 0)]]
```

Helper functions reduce cognitive load, make your code self-documenting, and allow you to say "I'll implement this helper later — for now, assume it works."

<FunFact>Studies on pair programming (which live coding interviews simulate) show that pairs produce code with 15% fewer defects than solo programmers, but take about 15% more time. In an interview, the "pair" dynamic with the interviewer actually works in your favour — they can catch mistakes you'd miss alone, if you communicate openly.</FunFact>

## Handling Hints Gracefully

Interviewers give hints. It's not a sign of failure — it's a conversation. How you respond matters:

**Good responses to hints**:
- "That's a great point — I think you're suggesting I use a stack here. Let me think about how that helps..."
- "Ah, I see — if I sort the input first, the two-pointer approach becomes possible. Let me restructure."
- "Thanks, I hadn't considered that edge case. Let me add handling for negative numbers."

**Bad responses**:
- Ignoring the hint and continuing your approach
- "I know, I was just about to do that" (defensive)
- Looking defeated or flustered

<Callout>An interviewer who gives hints is trying to help you succeed. Many candidates get hired after receiving multiple hints — what matters is that you take the hint, adapt, and show you can collaborate. Stubbornly sticking to a failing approach after a hint is a red flag.</Callout>

## Time Management: The 35-Minute Plan

For a typical 35-minute coding problem:

| Phase | Minutes | What To Do |
|---|---|---|
| **Clarify** | 0-5 | Read problem, ask questions, write examples |
| **Plan** | 5-8 | Discuss approach and complexity, get interviewer buy-in |
| **Code** | 8-25 | Write clean, readable code with talk-aloud |
| **Test** | 25-30 | Trace through examples manually, check edge cases |
| **Optimise** | 30-35 | Discuss possible improvements, alternative approaches |

### Common Timing Mistakes

1. **Spending 15 minutes coding without a plan** — you end up rewriting
2. **Never testing** — submitting and saying "I think it works" is a red flag
3. **Optimising before getting a working solution** — solve it first, then improve
4. **Spending too long on one approach** — if it's not working after 10 minutes, ask the interviewer if there's a better direction

## Testing Your Solution Before Saying "Done"

Never say "I'm done" without testing. Here's a systematic approach:

```python
# Step 1: Trace through the provided example
# Input: [1, 2, 3, 4, 5], target = 9
# i=0: complement=9-1=8, not in seen, seen={1:0}
# i=1: complement=9-2=7, not in seen, seen={1:0, 2:1}
# ...
# i=3: complement=9-4=5, not in seen, seen={1:0, 2:1, 3:2, 4:3}
# i=4: complement=9-5=4, 4 IS in seen at index 3 → return [3, 4] ✓

# Step 2: Test edge cases
# Empty array: [] → should return [] or None
# Single element: [5], target=5 → no pair possible
# Negative numbers: [-1, 2, 3], target=1 → [-1, 2] at indices [0, 1]
# Duplicates: [3, 3], target=6 → should return [0, 1]
```

Verbalise this testing process. It shows thoroughness and catches bugs before the interviewer does.

## What to Do When You're Stuck

Getting stuck is not failure — staying stuck silently is. Here's your escape plan:

1. **Restate the problem** — sometimes saying it aloud reveals a new angle
2. **Work through a small example** — manually solving a tiny case can reveal the algorithm
3. **Think about related problems** — "This reminds me of the sliding window pattern..."
4. **Consider the constraints** — if n ≤ 1000, an O(n²) solution might be acceptable
5. **Simplify the problem** — solve a simpler version first, then add complexity
6. **Ask for a hint** — "I'm considering using a hash map to track frequencies, but I'm not sure how to handle the window shrinking. Could you point me in the right direction?"

```python
# When stuck, try the "brute force first" strategy:
# 1. Write the O(n²) solution — it's usually straightforward
# 2. Identify the bottleneck (usually the inner loop)
# 3. Ask: "Can I eliminate this inner loop with a data structure?"
# 4. This often leads naturally to the optimal solution
```

## The Post-Solution Discussion

After your solution works, the interview isn't over. Be ready for:

- **"What's the time and space complexity?"** — always have this ready
- **"Can you optimise it?"** — discuss trade-offs even if you can't implement in time
- **"How would this handle edge cases?"** — show you've thought about production scenarios
- **"What if the input was 1 billion elements?"** — this pivots to system design thinking

```python
# Example post-solution discussion:
# "My solution is O(n) time and O(n) space.
#  The space is used by the hash map which stores at most n elements.
#  If memory is a constraint, we could sort the array first and use
#  two pointers for O(n log n) time and O(1) space — but we'd need
#  to modify the input array or create a copy.
#  For 1 billion elements, we might need to partition the data
#  and process in chunks, or use external sorting."
```

## Key Takeaways

- **Talk aloud continuously**: silence is your enemy — narrate your problem-solving process from start to finish.
- **Clarify before coding**: 5 minutes of questions saves 15 minutes of wrong-direction coding.
- **Write clean code even under pressure**: meaningful names, helper functions, and skeleton-first approach make debugging easier and impress interviewers.
- **Test systematically**: trace through examples and edge cases before declaring "done."
- **Handle hints and setbacks gracefully**: getting stuck is normal — how you recover is what matters.
- **Manage your time**: plan (5min) → code (17min) → test (5min) → discuss (5min). Adjust based on the problem, but never skip testing.
