# Complexity Analysis Mastery

"What's the time complexity?" — four words that follow every solution you present in a coding interview. Get it right, and you sound like a confident engineer. Stumble, and the interviewer questions whether you truly understand your own code. Complexity analysis isn't just academic theory; it's the lens through which engineers evaluate whether a solution will work at scale. This lesson equips you to analyse any solution with confidence — from simple loops to tricky recursive algorithms.

## Big-O Notation Fundamentals

Big-O describes the upper bound of how an algorithm's runtime or memory usage grows as the input size grows. It strips away constants and lower-order terms to focus on the dominant factor.

| Complexity | Name | Example | 1M Elements |
|---|---|---|---|
| O(1) | Constant | Hash map lookup | Instant |
| O(log n) | Logarithmic | Binary search | ~20 operations |
| O(n) | Linear | Single loop through array | 1M operations |
| O(n log n) | Linearithmic | Merge sort, heap sort | ~20M operations |
| O(n²) | Quadratic | Nested loops (brute force) | 1 trillion operations |
| O(2ⁿ) | Exponential | Recursive subsets | Heat death of universe |
| O(n!) | Factorial | All permutations | Heat death × heat death |

<Callout>A practical rule of thumb: modern computers handle roughly 10⁸ simple operations per second. If your algorithm is O(n²) and n = 10⁵, that's 10¹⁰ operations — about 100 seconds. Too slow. This mental math helps you quickly judge whether your approach will pass within typical time limits.</Callout>

## Analysing Simple Loops

### Single Loop → O(n)

```python
def find_max(nums):
    max_val = nums[0]       # O(1)
    for num in nums:         # Loop runs n times
        if num > max_val:    # O(1) comparison
            max_val = num    # O(1) assignment
    return max_val           # O(1)
# Total: O(1) + O(n) × O(1) = O(n)
```

### Nested Loops → Multiply

```python
def has_duplicate_pair(nums):
    for i in range(len(nums)):           # n iterations
        for j in range(i + 1, len(nums)):  # ~n iterations
            if nums[i] == nums[j]:
                return True
    return False
# Total: O(n) × O(n) = O(n²)
```

### Sequential Blocks → Add

```python
def process(nums):
    # Block 1: O(n)
    for num in nums:
        print(num)

    # Block 2: O(n²)
    for i in range(len(nums)):
        for j in range(len(nums)):
            print(i, j)

# Total: O(n) + O(n²) = O(n²)  — drop the lower-order term
```

### Loops with Non-Obvious Iterations

```python
# This is O(n), not O(n²)!
def two_pointer_sum(nums, target):
    left, right = 0, len(nums) - 1
    while left < right:    # Each pointer moves at most n times total
        total = nums[left] + nums[right]
        if total == target:
            return True
        elif total < target:
            left += 1
        else:
            right -= 1
    return False
```

<ThinkAboutIt>Consider a loop where the variable doubles each iteration: `i = 1; while i < n: i *= 2`. How many times does this loop execute? What's the complexity? (Hint: if i doubles each time, how many doublings does it take to reach n?)</ThinkAboutIt>

## Analysing Recursion

Recursion is where complexity analysis gets interesting. The key tool is the **recursion tree**.

### Linear Recursion → O(n)

```python
def factorial(n):
    if n <= 1:
        return 1
    return n * factorial(n - 1)

# Recursion tree: single chain of n calls
# factorial(5) → factorial(4) → factorial(3) → factorial(2) → factorial(1)
# Time: O(n), Space: O(n) — n stack frames
```

### Binary Recursion → O(2ⁿ)

```python
# Naive Fibonacci — the classic exponential example
def fib(n):
    if n <= 1:
        return n
    return fib(n - 1) + fib(n - 2)

# Recursion tree: each call spawns TWO calls
#                fib(5)
#              /        \
#          fib(4)       fib(3)
#         /     \       /    \
#      fib(3)  fib(2) fib(2) fib(1)
#      ...
# Nodes at each level double → total nodes ≈ 2ⁿ
# Time: O(2ⁿ), Space: O(n) — max depth of tree
```

### Memoised Recursion → O(n)

```python
def fib_memo(n, memo={}):
    if n in memo:
        return memo[n]
    if n <= 1:
        return n
    memo[n] = fib_memo(n - 1, memo) + fib_memo(n - 2, memo)
    return memo[n]

# Each subproblem computed only once → O(n) unique calls
# Time: O(n), Space: O(n)
```

<FunFact>The difference between naive and memoised Fibonacci is staggering at scale. Computing fib(50) naively requires about 2⁵⁰ ≈ 1 quadrillion operations — roughly 100 days on a modern computer. The memoised version requires exactly 50 operations — done in microseconds. This is why complexity analysis matters.</FunFact>

### The Master Theorem (Simplified)

For divide-and-conquer recurrences of the form **T(n) = aT(n/b) + O(nᵈ)**:

| Condition | Complexity |
|---|---|
| d > log_b(a) | O(nᵈ) |
| d = log_b(a) | O(nᵈ log n) |
| d < log_b(a) | O(n^(log_b(a))) |

**Practical examples**:
- **Merge Sort**: T(n) = 2T(n/2) + O(n) → a=2, b=2, d=1 → d = log₂2 → **O(n log n)**
- **Binary Search**: T(n) = T(n/2) + O(1) → a=1, b=2, d=0 → d = log₂1 → **O(log n)**
- **Strassen Matrix Mult**: T(n) = 7T(n/2) + O(n²) → a=7, b=2, d=2 → d < log₂7 → **O(n^2.81)**

## Space Complexity

Space complexity is just as important as time complexity but often overlooked. Count:

1. **Auxiliary space** — extra data structures you create
2. **Stack frames** — each recursive call adds a frame

```python
# O(1) space — only a few variables
def reverse_in_place(nums):
    left, right = 0, len(nums) - 1
    while left < right:
        nums[left], nums[right] = nums[right], nums[left]
        left += 1
        right -= 1

# O(n) space — creating a new list
def reverse_copy(nums):
    return nums[::-1]

# O(n) space from recursion — n stack frames
def reverse_recursive(nums, left, right):
    if left >= right:
        return
    nums[left], nums[right] = nums[right], nums[left]
    reverse_recursive(nums, left + 1, right - 1)
```

<Callout>In interviews, always state both time AND space complexity. A common follow-up is "Can you do it in O(1) space?" This often means converting a recursive solution to iterative, or finding an in-place algorithm.</Callout>

## Amortised Analysis Basics

Some operations are expensive occasionally but cheap on average. Amortised analysis accounts for this.

**Classic example: Dynamic Array (ArrayList)**

```python
# Appending to a dynamic array
# Most appends are O(1) — just place the element
# Occasionally (when full), the array doubles: O(n) copy
# Over n operations: n × O(1) + log(n) × O(n) = O(n) total
# Amortised cost per operation: O(n) / n = O(1)
```

Other amortised O(1) examples:
- Hash map insertions (occasional rehashing)
- Splay tree operations
- Two-stack queue implementation

## Common Complexity Traps

### Trap 1: Hidden Loops in Built-in Operations

```python
# Looks like O(n), but string concatenation in a loop is O(n²)
def build_string(chars):
    result = ""
    for char in chars:
        result += char   # Each += creates a new string → O(current length)
    return result
# Fix: use ''.join(chars) → O(n)

# list.index() and 'in' on lists are O(n)
for item in big_list:
    if item in another_big_list:  # O(n) × O(n) = O(n²)
        process(item)
# Fix: convert to set for O(1) lookup
```

### Trap 2: Sorting Dominates

```python
def find_kth_largest(nums, k):
    nums.sort()       # O(n log n) — this dominates
    return nums[-k]   # O(1)
# Total: O(n log n), not O(1)
# Better: use a heap for O(n log k)
```

### Trap 3: Recursion Depth ≠ Total Work

```python
# Depth is O(log n), but total work is O(n)
def merge_sort(arr):
    if len(arr) <= 1:
        return arr
    mid = len(arr) // 2
    left = merge_sort(arr[:mid])     # T(n/2)
    right = merge_sort(arr[mid:])    # T(n/2)
    return merge(left, right)        # O(n) work at each level
# log n levels × O(n) work per level = O(n log n)
```

This is the arithmetic series sum, which simplifies to O(n²). It's a common trap because the inner loop doesn't always run n times, but the total work is still quadratic." />

## From Brute Force to Optimal

In interviews, you'll often start with a brute-force solution and optimise. Here's the common optimisation path:

| Current | Optimisation Technique | Improved To |
|---|---|---|
| O(n²) | Hash map for lookups | O(n) |
| O(n²) | Sorting + two pointers | O(n log n) |
| O(2ⁿ) | Memoisation / DP | O(n²) or O(n×m) |
| O(n log n) | Bucket sort / counting sort (limited range) | O(n) |
| O(n) space | In-place manipulation | O(1) space |

**The conversation flow in an interview**:
1. "My brute force approach is O(n²) using nested loops."
2. "I notice we're doing repeated lookups — a hash map can reduce the inner search to O(1)."
3. "The optimised solution is O(n) time and O(n) space."
4. "The trade-off is we're using extra memory, but the time improvement is significant."

## Key Takeaways

- **Know the hierarchy**: O(1) < O(log n) < O(n) < O(n log n) < O(n²) < O(2ⁿ) < O(n!). You'll reference this in every interview.
- **Multiply nested loops, add sequential blocks**: the two fundamental rules for loop analysis.
- **Draw recursion trees**: they make recursive complexity visual and much easier to analyse correctly.
- **Watch for hidden costs**: string concatenation, list membership testing, and sorting inside loops are common traps.
- **State both time and space**: a complete complexity analysis always covers both dimensions.
- **Connect to optimisation**: knowing the brute-force complexity tells you how much room there is to improve.
