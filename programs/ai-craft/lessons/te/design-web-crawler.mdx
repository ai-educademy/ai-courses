# Web Crawler డిజైన్ చేయండి - Google ఇంటర్నెట్‌ను ఎలా Index చేస్తుంది

Web crawler క్రమబద్ధంగా ఇంటర్నెట్‌ను browse చేసి web pages ను download మరియు index చేస్తుంది. ఇది ప్రతి search engine కు పునాది, మరియు దీన్ని డిజైన్ చేయడం వల్ల distributed systems, graph traversal, మరియు real web యొక్క complexities ను handle చేయడం నేర్చుకుంటారు.

## Step 1 - Requirements సేకరణ

### Functional Requirements
- Seed URLs సెట్ నుండి web pages crawl చేయడం
- Hyperlinks extract చేసి కొత్త pages discover చేయడం
- Download చేసిన content ను indexing మరియు processing కోసం store చేయడం
- robots.txt మరియు crawl politeness rules పాటించడం

### Non-Functional Requirements
- **Scale** - నెలకు 1 billion pages crawl చేయడం
- **Politeness** - ఏ ఒక్క web server ను overload చేయకపోవడం
- **Freshness** - ముఖ్యమైన pages ను క్రమం తప్పకుండా re-crawl చేయడం
- **Robustness** - malformed HTML, infinite loops, మరియు spider traps ను handle చేయడం

## Step 2 - High-Level Architecture

```
Seed URLs → URL Frontier (priority queue)
                  ↓
            URL Scheduler (politeness + priority)
                  ↓
         Fetcher (HTTP client pool)
                  ↓
         HTML Parser → Extract URLs → URL Filter
              ↓                           ↓
       Content Store              URL Dedup (Bloom Filter)
       (S3 / HDFS)                      ↓
                                 URL Frontier (loop)
```

## Step 3 - Crawl Loop

### BFS vs DFS Crawling
- **BFS (Breadth-First)** - web crawling కోసం ఇష్టపడతారు. త్వరగా అనేక domains discover చేస్తుంది మరియు ఒకే site లో లోతుగా చిక్కుకోకుండా నిరోధిస్తుంది
- **DFS (Depth-First)** - ఇతర domains discover చేయకముందే dynamically generated pages లో infinitely deep గా వెళ్ళే risk ఉంది

### URL Frontier Design
Frontier ఒక simple queue కాదు - ఇది రెండు sub-components తో ఉన్న sophisticated priority system:

1. **Priority Queue (Front)** - URLs ను importance ఆధారంగా rank చేస్తుంది. PageRank, update frequency, మరియు domain authority వంటి factors ఉంటాయి
2. **Politeness Queue (Back)** - ప్రతి domain కు దాని స్వంత FIFO queue ఉంటుంది. ఒక scheduler వివిధ domain queues నుండి pick చేస్తుంది, తద్వారా ఏ ఒక్క server పై అధిక load పడదు

```
Front (Priority):       Back (Politeness):
┌──────────────┐       ┌─── bbc.co.uk queue ───┐
│ High Priority │ ───→  │ url1 → url4 → url7    │
│ Med Priority  │ ───→  ├─── github.com queue ──┤
│ Low Priority  │ ───→  │ url2 → url5 → url8    │
└──────────────┘       ├─── wikipedia.org queue ┤
                        │ url3 → url6 → url9    │
                        └───────────────────────┘
```

## Step 4 - Politeness మరియు Robots.txt

### Robots.txt పాటించడం
ఏదైనా domain ను crawl చేయడానికి ముందు, దాని `robots.txt` file ను fetch మరియు cache చేయండి:

```
# Example robots.txt
User-agent: *
Disallow: /admin/
Disallow: /private/
Crawl-delay: 2
```

ప్రతి domain యొక్క robots.txt ను TTL (24 గంటలు) తో cache చేయండి. `Disallow` rules మరియు `Crawl-delay` directives ను ఎల్లప్పుడూ పాటించండి.

### Crawl Rate Limiting
Explicit `Crawl-delay` లేకపోయినా politeness కొనసాగించండి:
- ప్రతి domain కు second కు maximum 1 request (configurable)
- ప్రతి domain కు separate download queues ఉపయోగించండి
- Response times track చేయండి - server slow అయితే, automatically back off అవ్వండి

<FunFact>Google యొక్క web crawler (Googlebot) వందల billions pages crawl చేస్తుంది. Polite గా ఉండటానికి, ఇది server response times monitor చేస్తుంది మరియు ఒక site slow గా respond చేయడం మొదలుపెడితే crawl rate తగ్గిస్తుంది - effectively website ను crawler కు వ్యతిరేకంగా "push back" చేయనిస్తుంది.</FunFact>

## Step 5 - Deduplication

### URL Deduplication (Bloom Filter)
Billions URLs తో, ఒక URL ను ఇంతకు ముందు చూశామా అని check చేయడానికి efficient data structure అవసరం. Bloom filter O(1) lookups ను చిన్న memory footprint తో అందిస్తుంది:

- **Space:** 1 billion URLs కు ~1 GB, 1% false positive rate తో
- **Trade-off:** అప్పుడప్పుడు page re-crawl అవ్వవచ్చు (false positive) కానీ కొత్త URL ఎప్పటికీ miss కాదు (no false negatives)

Check చేయడానికి ముందు URLs normalize చేయండి: fragments (#) తొలగించండి, query parameters sort చేయండి, hostname lowercase చేయండి, మరియు relative paths resolve చేయండి.

### Content Deduplication (SimHash)
వేర్వేరు URLs identical content serve చేయవచ్చు (mirrors, copied articles). SimHash page content యొక్క fingerprint generate చేస్తుంది - similar fingerprints (చిన్న Hamming distance) ఉన్న pages likely duplicates:

```
Page A content → SimHash → 10110010...
Page B content → SimHash → 10110011...
Hamming distance = 1 → likely duplicate, skip storing
```

## Step 6 - Distributed Crawling

### Horizontally Scale చేయడం
ఒక్క machine నెలకు billion pages crawl చేయలేదు. పనిని distribute చేయండి:

- Domain names పై **consistent hashing** ప్రతి domain ను specific crawler node కు assign చేస్తుంది. ఇది politeness ensure చేస్తుంది - ఒక given domain ను ఒకే node crawl చేస్తుంది
- **URL frontier** hash ring ఆధారంగా nodes మధ్య partition అవుతుంది
- Node assignments manage చేసి, nodes join లేదా leave అయినప్పుడు rebalance చేసే lightweight service ద్వారా **coordination**

### DNS Resolution Caching
DNS lookups ప్రతి request కు 10-100ms add చేస్తాయి. Scale వద్ద ఇది major bottleneck:
- ప్రతి crawler node కు local DNS cache ఉంచండి
- Queue లో ఉన్న URLs కోసం batch DNS pre-resolution చేయండి
- Public DNS servers overload కాకుండా dedicated DNS resolver ఉపయోగించండి

### Spider Trap Detection
Malicious లేదా poorly built sites infinite URL spaces create చేయవచ్చు:
- `/page/1/page/1/page/1/...` - repeating path patterns
- ప్రతి future date కు URLs generate చేసే calendar pages
- Identical content కు unique URLs create చేసే URLs లో Session IDs

**రక్షణలు:** maximum URL length, ప్రతి domain కు maximum crawl depth, repetitive URL structures కోసం pattern detection, మరియు domain-level page count limits.

<ThinkAboutIt>Search index ను fresh గా ఉంచాలంటే, ఏ pages ను ముందుగా re-crawl చేయాలో ఎలా decide చేస్తారు? ప్రతి గంటకు update అయ్యే news site vs నెలకోసారి మారే Wikipedia article గురించి ఆలోచించండి. Re-crawl scheduling prioritise చేయడానికి ఏ signals ఉపయోగిస్తారు?</ThinkAboutIt>

## Step 7 - Bottlenecks మరియు Optimisations

| Bottleneck | Solution |
|-----------|----------|
| DNS resolution latency | Local DNS cache, batch pre-resolution |
| Slow/unresponsive servers | Timeouts (30s), async I/O, retries తర్వాత skip |
| Crawled content storage | Pages compress చేయడం, changed content మాత్రమే store చేయడం |
| URL frontier memory | Disk-backed priority queue (RocksDB) |
| Duplicate content | SimHash fingerprinting, canonical URL detection |

<ThinkAboutIt>Search engines crawl freshness (pages ను తరచుగా re-crawl చేయడం) మరియు discovery (కొత్త pages కనుగొనడం) మధ్య balance చేయాలి. మీ crawler కు రోజుకు 1 million pages అనే fixed budget ఉంటే, ఆ budget ను existing pages re-crawl చేయడం మరియు కొత్త pages discover చేయడం మధ్య ఎలా విభజిస్తారు?</ThinkAboutIt>
