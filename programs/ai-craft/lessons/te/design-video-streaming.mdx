# Video Streaming Platform Design చేయండి - YouTube/Netflix Architecture

Video streaming మొత్తం global internet traffic లో 65% కంటే ఎక్కువ. ఈ system design చేయడంలో file processing pipelines, CDN architecture, మరియు adaptive streaming వంటి concepts ఉంటాయి - ఇవి video కి మాత్రమే పరిమితం కాదు.

## Step 1 - Requirements సేకరణ

### Functional Requirements
- Videos upload చేయడం (ప్రతి file కి 1 GB వరకు)
- Network conditions ఆధారంగా adaptive quality తో videos stream చేయడం
- Title, description, మరియు tags ద్వారా videos search చేయడం
- View counts మరియు engagement metrics track చేయడం

### Non-Functional Requirements
- **Availability** - streaming కోసం 99.99% uptime
- **Low latency** - video playback 2 seconds లోపు start అవ్వాలి
- **Global reach** - అన్ని continents లోని users కి efficiently serve చేయాలి
- **Cost-efficient** - storage మరియు bandwidth optimize చేయాలి (రెండు అతిపెద్ద costs)

## Step 2 - High-Level Architecture

```
Upload Flow:
Client → Upload Service → Object Storage (raw)
                              ↓
                    Transcoding Pipeline (async)
                         ↓          ↓
                   Multiple      Thumbnail
                   Qualities     Generator
                         ↓          ↓
                    CDN Origin Storage → CDN Edge Servers

Streaming Flow:
Client → CDN Edge → (cache miss) → CDN Origin → Object Storage
```

## Step 3 - Video Upload Pipeline

### దశల వారీ Processing
1. **Upload** - client raw video ని object storage (ఉదా. S3) కి pre-signed URLs ద్వారా upload చేస్తుంది, పెద్ద files కోసం resumable uploads తో
2. **Transcoding** - source video ని multiple resolutions (1080p, 720p, 480p, 360p) మరియు codecs (H.264, VP9, AV1) లోకి convert చేయడం
3. **Adaptive Bitrate Packaging** - ప్రతి quality level ని చిన్న segments (2-10 seconds) గా split చేయడం HLS/DASH streaming కోసం
4. **Thumbnail Generation** - preview thumbnails కోసం regular intervals లో frames extract చేయడం
5. **Metadata Update** - అన్ని processing complete అయిన తర్వాత database లో video ని "ready" గా mark చేయడం

### Transcoding at Scale
ఒక 10-minute 4K video ని ఒక machine లో transcode చేయడానికి 30+ minutes పట్టవచ్చు. వేలాది concurrent uploads handle చేయడానికి:
- Videos ని chunks గా split చేసి DAG-based workflow (AWS Step Functions లాంటిది) తో parallel లో transcode చేయండి
- Spot/preemptible instances వాడి compute costs ని 60-80% తగ్గించండి

## Step 4 - Streaming Protocols మరియు CDN

### Adaptive Bitrate Streaming
Player viewer యొక్క network speed ఆధారంగా automatically quality switch చేస్తుంది:

| Protocol | వాడేవారు | Segment Format | Manifest |
|----------|---------|----------------|----------|
| HLS | Apple, చాలా browsers | .ts లేదా .fmp4 | .m3u8 playlist |
| DASH | YouTube, Netflix | .mp4 segments | .mpd manifest |

Client ఒక manifest file download చేస్తుంది, అందులో అన్ని available quality levels ఉంటాయి, తర్వాత ఒక్కొక్క segment request చేస్తుంది, bandwidth మారినప్పుడు seamlessly quality switch చేస్తుంది.

### CDN Architecture
Streaming performance కోసం CDN అత్యంత critical component:

```
User in London → London Edge PoP (cache hit: 95%)
                     ↓ (cache miss)
              European Regional Cache
                     ↓ (cache miss)
              Origin Storage (S3)
```

**Key insight:** popular videos users కి అత్యంత దగ్గరగా ఉన్న edge locations లో cache అవుతాయి. Long-tail content regional nodes లో మాత్రమే cache అవ్వవచ్చు, rare content కోసం origin fetch జరుగుతుంది.

<FunFact>Netflix ప్రతి వారం 700 million+ hours video serve చేస్తుంది. వారి Open Connect CDN ISP data centres లోపల custom hardware appliances ఉంచుతుంది, popular content ని last mile లో cache చేసి internet backbone traffic ని 90% కంటే ఎక్కువ తగ్గిస్తుంది.</FunFact>

## Step 5 - Supporting Services

### Video Metadata Service
Title, description, tags, upload date, view counts, మరియు processing status store చేస్తుంది. Structured queries మరియు search indexing కోసం relational database (PostgreSQL) వాడండి.

### View Counting at Scale
Naive approach (ప్రతి view కి increment) write hotspot సృష్టిస్తుంది. బదులుగా:
- View events ని stream (Kafka) లోకి batch చేయండి
- Counts ని periodically aggregate చేయండి (ప్రతి 30 seconds)
- Real-time display కోసం approximate counting, analytics కోసం exact counts వాడండి

### Live Streaming Architecture
Live streaming on-demand కంటే కీలకమైన విధాలుగా భిన్నంగా ఉంటుంది:
- **Pre-transcoding లేదు** - video ని real-time లో transcode చేయాలి
- **Ultra-low latency** - viewers 5-second కంటే తక్కువ delay expect చేస్తారు
- **Ingestion protocol** - streamers RTMP ద్వారా push చేస్తారు; viewers HLS/DASH low-latency extensions ద్వారా receive చేస్తారు

## Step 6 - Cost Breakdown మరియు Optimisations

Video platforms నడపడం **చాలా ఖర్చుతో కూడుకున్నది**. మూడు అతిపెద్ద costs:

| Cost Centre | అంచనా (10M daily users) | Optimisation |
|------------|------------------------|--------------|
| Storage | ~$200K/నెల | Tiered storage (hot/warm/cold), failed uploads delete చేయండి |
| CDN Bandwidth | ~$500K/నెల | Edge caching, modern codecs (AV1) తో compress చేయండి |
| Transcoding Compute | ~$100K/నెల | Spot instances, అరుదుగా చూసే quality levels skip చేయండి |

<ThinkAboutIt>Netflix content delivery కోసం ఏటా billions ఖర్చు చేస్తుంది. మీరు వారి system ని మొదటి నుండి design చేస్తుంటే, మీ సొంత CDN build చేస్తారా (Netflix Open Connect లాగా) లేదా commercial CDN (CloudFront లాంటిది) వాడతారా? ఏ scale లో సొంత CDN build చేయడం cost-effective అవుతుంది?</ThinkAboutIt>

### Content Protection (DRM)
Premium content కి Digital Rights Management అవసరం:
- **Widevine** (Google) - Chrome, Android
- **FairPlay** (Apple) - Safari, iOS
- **PlayReady** (Microsoft) - Edge, Windows

ప్రతి DRM system కి separate encryption మరియు licence server integration అవసరం, ఇది గణనీయమైన complexity add చేస్తుంది.

<ThinkAboutIt>YouTube 800 million+ videos store చేస్తుంది. ప్రతి video 5 quality levels లో transcode చేయబడి ప్రతి level సగటున 500 MB ఉంటే, మొత్తం ఎంత storage అవసరమో అంచనా వేయండి. Cost-effective storage strategy ఎలా design చేస్తారు?</ThinkAboutIt>
