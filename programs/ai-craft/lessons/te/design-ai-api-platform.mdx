## Interview Prompt

*"Multiple machine learning models ను developers కి REST API ద్వారా serve చేసే AI API platform డిజైన్ చేయండి - OpenAI లేదా Hugging Face scale లో models serve చేసినట్లుగా."*

ఈ ప్రశ్న interviews లో చాలా popular అవుతోంది ఎందుకంటే ఇది **API platform design** ను AI workloads కు unique అయిన **GPU infrastructure challenges** తో combine చేస్తుంది.

## Step 1: Requirements Gathering

**Functional requirements:**
- Multiple AI models (text generation, image classification, embeddings) ను REST API ద్వారా serve చేయడం
- API key authentication with per-key usage quotas
- ప్రతి API call కి usage tracking మరియు billing
- Multiple model versions కు A/B testing తో support
- Long-running generation tasks కోసం streaming responses

**Non-functional requirements:**
- Latency: embedding models కి p50 500ms లోపు, generative models కి p50 2s లోపు
- Throughput: అన్ని models లో 10,000 requests per second
- Availability: graceful degradation తో 99.9% uptime
- Cost efficiency: GPUs ఖరీదైనవి - utilisation maximize చేయాలి

<Callout>
AI platforms కు చాలా variable latency ఉంటుంది. ఒక text completion request output length ని బట్టి 200ms లేదా 30 seconds పట్టవచ్చు. మీ design ఈ variance ను ఎలా handle చేస్తుందో ఎప్పుడూ discuss చేయండి.
</Callout>

## Step 2: API Design

```
POST /v1/completions
Headers: Authorization: Bearer sk-abc123
Body: { "model": "gpt-4", "prompt": "Explain caching", "max_tokens": 200 }

POST /v1/embeddings
Headers: Authorization: Bearer sk-abc123
Body: { "model": "text-embedding-3", "input": "Hello world" }

GET  /v1/models                    - available models list
GET  /v1/usage?start=2024-01-01    - usage మరియు billing data

SSE: POST /v1/completions (stream: true)
     → returns Server-Sent Events with incremental tokens
```

Generative models కోసం **Server-Sent Events (SSE)** ద్వారా streaming చాలా critical - users పూర్తి response కోసం wait చేయకుండా tokens progressively కనిపిస్తాయి.

## Step 3: API Gateway మరియు Authentication

API gateway platform యొక్క front door:

- **Authentication** - API keys ను key store తో validate చేయండి (speed కోసం Redis, persistence కోసం database)
- **Rate limiting** - per-key limits enforce చేయండి (ఉదా., free tier కోసం 60 requests/minute)
- **Request routing** - requests ను సరైన model-serving cluster కి direct చేయండి
- **Request validation** - payload schema, model availability, token limits check చేయండి

**Rate limiting implementation:**
Redis లో **sliding window counter** వాడండి:
```
Key:   rate:{api_key}:{window_minute}
Value: request count
TTL:   120 seconds
```

ప్రతి request counter ను increment చేస్తుంది. Limit exceed అయితే HTTP 429 return చేయండి.

## Step 4: Model Serving మరియు GPU Allocation

ఇక్కడ AI platforms traditional web services నుండి fundamentally భిన్నంగా ఉంటాయి. GPUs scarce, ఖరీదైనవి మరియు specialised orchestration అవసరం.

**Model serving layer:**
- **Triton Inference Server** లేదా **vLLM** లాంటి frameworks తో models ను GPU memory లోకి load చేస్తారు
- ప్రతి model version dedicated GPU instances pool లో run అవుతుంది
- Large language models (LLMs) కు multiple GPUs లో **model parallelism** అవసరం కావచ్చు

**Request batching:**
GPUs లో individual inference requests inefficient. **Dynamic batching** multiple requests ను group చేసి కలిసి process చేస్తుంది:
- 10ms వరకు లేదా batch 32 requests చేరే వరకు collect చేయండి
- మొత్తం batch ను ఒక GPU forward pass లో process చేయండి
- ప్రతి caller కు individual responses return చేయండి

Individually process చేయడంతో పోలిస్తే ఇది throughput ను 5-10x improve చేయగలదు.

<FunFact>
ఒక NVIDIA H100 GPU ధర సుమారు £25,000 మరియు 700 watts power consume చేస్తుంది. Model serving కోసం 1,000 GPUs cluster నడపడానికి electricity మరియు hardware లో మాత్రమే నెలకు £2 million కంటే ఎక్కువ ఖర్చు అవుతుంది - అందుకే efficient batching మరియు utilisation చాలా ముఖ్యం.
</FunFact>

## Step 5: Model Versions మధ్య A/B Testing

కొత్త model version deploy చేసేటప్పుడు controlled rollout అవసరం:

- **Traffic splitting** - 5% requests కొత్త model కు, 95% current model కు route చేయండి
- **Shadow mode** - production traffic లో కొత్త model run చేయండి కానీ results users కు return చేయకండి
- **Metrics comparison** - latency, output quality scores, error rates, cost per request
- **Gradual rollout** - confidence పెరిగే కొద్దీ కొత్త version కు traffic పెంచండి

Routing logic API gateway లో ఉంటుంది:
```
model: "gpt-4"  →  95% → gpt-4-v2.1 (current)
                →   5% → gpt-4-v2.2 (candidate)
```

<ThinkAboutIt>
కొత్త LLM version "better" అని evaluate చేయడం surprisingly కష్టం - classification models లా simple accuracy metric ఉండదు. Generative AI models కోసం automated quality evaluation system మీరు ఎలా design చేస్తారు?
</ThinkAboutIt>

## Step 6: Usage Tracking మరియు Billing

AI APIs సాధారణంగా per token bill చేస్తాయి (LLMs కు input + output tokens, ఇతర models కు per request).

**Event-driven billing pipeline:**
1. ప్రతి API response లో usage header ఉంటుంది: `x-usage-tokens: 847`
2. Kafka కు billing event publish అవుతుంది: `{ api_key, model, input_tokens, output_tokens, timestamp }`
3. Billing aggregation service events consume చేసి running totals update చేస్తుంది
4. Usage dashboards time-series database నుండి aggregated data query చేస్తాయి

**Kafka ఎందుకు?** Billing reliable గా ఉండాలి కానీ synchronous గా ఉండాల్సిన అవసరం లేదు. Billing service temporarily down అయితే, events Kafka లో queue అవుతాయి మరియు recover అయినప్పుడు process అవుతాయి - usage data ఏదీ lost కాదు.

## Step 7: Monitoring మరియు Observability

AI platforms కు standard web services కంటే ఎక్కువ monitoring అవసరం:

| Metric | ఎందుకు ముఖ్యం |
|--------|---------------|
| Inference latency (p50, p95, p99) | User experience మరియు SLA compliance |
| GPU utilisation | Cost efficiency - idle GPUs డబ్బు వృథా చేస్తాయి |
| Tokens per second | Throughput capacity మరియు scaling signals |
| Error rate by model | Model-specific failures detect చేయడం |
| Queue depth | Scaling trigger - deep queues అంటే demand capacity కంటే ఎక్కువ |
| Output quality scores | Model degradation detect చేయడం (automated eval ద్వారా) |

**Alerting thresholds:** GPU utilisation 40% కంటే తగ్గితే scale-down trigger; queue depth 100 కంటే ఎక్కువైతే scale-up trigger.

## Step 8: Auto-Scaling

AI workloads కు చాలా variable demand ఉంటుంది. Auto-scaling responsiveness మరియు cost ను balance చేయాలి:

- **Horizontal scaling** - queue depth లేదా latency thresholds exceed అయినప్పుడు మరిన్ని GPU instances add చేయండి
- **Predictive scaling** - expected peaks ముందు capacity pre-provision చేయడానికి historical traffic patterns వాడండి
- **Scale-to-zero** - rarely used models కోసం అన్ని instances shut down చేసి demand మీద cold-start చేయండి
- **Warm pools** - cold-start latency avoid చేయడానికి కొన్ని instances models pre-loaded గా ఉంచండి

Scale-to-zero strategy వందల models host చేసే platforms కు ప్రత్యేకంగా ముఖ్యం - ప్రతి model ను ఎల్లప్పుడూ GPU memory లో loaded గా ఉంచలేరు.

<ThinkAboutIt>
OpenAI platform fast embedding models నుండి slow reasoning models వరకు serve చేస్తుంది. ప్రతి దాని latency characteristics, GPU requirements మరియు pricing వేరు. ఇంత different profiles ఉన్న models మధ్య finite GPU cluster ను efficiently share చేయడానికి routing layer ను మీరు ఎలా design చేస్తారు?
</ThinkAboutIt>

</FunFact>

## ముఖ్యమైన Takeaways

- AI API platforms traditional API design ను GPU-specific infrastructure challenges తో combine చేస్తాయి
- Request batching GPU throughput మరియు cost efficiency ని dramatically improve చేస్తుంది
- Rate limiting platform మరియు customers ను runaway costs నుండి protect చేస్తుంది
- Kafka ద్వారా asynchronous billing requests block చేయకుండా reliable usage tracking ensure చేస్తుంది
- Auto-scaling responsiveness మరియు cost ను balance చేయాలి - low-traffic models కోసం scale-to-zero
- GPU utilisation మరియు queue depth monitor చేయడం intelligent scaling decisions drive చేస్తుంది
