# Rate Limiter డిజైన్ చేయండి — APIs ను Scale లో ప్రొటెక్ట్ చేయడం

Rate limiting అనేది అత్యంత practical system design topics లో ఒకటి. ప్రతి production API — GitHub నుండి Stripe వరకు — abuse నివారించడానికి, fair usage నిర్ధారించడానికి మరియు backend services overload కాకుండా రక్షించడానికి దీనిపై ఆధారపడుతుంది.

## Step 1 — Requirements సేకరణ

### Functional Requirements
- Configurable rules ఆధారంగా requests limit చేయండి (per user, IP లేదా API key)
- Client throttle అయినప్పుడు clear feedback ఇవ్వండి (HTTP 429)
- ఏకకాలంలో multiple rate limiting rules support చేయండి

### Non-Functional Requirements
- **Low latency** — limiter significant overhead add చేయకూడదు
- **Distributed** — multiple servers లో పనిచేయాలి
- **Fault-tolerant** — limiter fail అయితే, traffic flow అవుతూనే ఉండాలి (fail-open)

<ThinkAboutIt>Rate limiter crash అయినప్పుడు fail-open (traffic allow చేయడం) అవ్వాలా లేదా fail-closed (traffic block చేయడం) అవ్వాలా? E-commerce site vs banking API కి trade-offs ఏమిటి?</ThinkAboutIt>

## Step 2 — Rate Limiting Algorithms

### Fixed Window Counter
Time ను fixed windows గా divide చేయండి (ఉదా., 60-second intervals). ప్రతి request కి counter increment చేయండి. Window boundaries వద్ద reset చేయండి. **సమస్య:** window edges వద్ద burst traffic 2× limit ని allow చేయవచ్చు.

### Sliding Window Log
ప్రతి request యొక్క timestamp store చేయండి. Trailing window లోని entries count చేయండి. Precise కానీ **memory-intensive** — scale లో ప్రతి timestamp store చేయడం expensive.

### Sliding Window Counter
ఒక hybrid approach: current window count ను previous window యొక్క weighted portion తో combine చేయండి. Accuracy మరియు memory efficiency ని balance చేస్తుంది.

### Token Bucket Algorithm (అత్యంత Popular)
ఒక bucket maximum capacity వరకు tokens hold చేస్తుంది. Tokens fixed rate లో add అవుతాయి. ప్రతి request ఒక token consume చేస్తుంది. Bucket empty అయితే, request reject అవుతుంది.

```
Bucket: capacity=10, refill_rate=2/sec

Time 0s: [🪙🪙🪙🪙🪙🪙🪙🪙🪙🪙] → 10 tokens
Burst of 8 requests → [🪙🪙] → 2 tokens remain
Time 1s: refill +2 → [🪙🪙🪙🪙] → 4 tokens
```

### Algorithm పోలిక

| Algorithm | Memory | Accuracy | Burst Handling |
|-----------|--------|----------|----------------|
| Fixed Window | తక్కువ | తక్కువ (edge bursts) | బలహీనం |
| Sliding Log | ఎక్కువ | Exact | మంచిది |
| Sliding Counter | మధ్యస్థం | Approximate | మంచిది |
| Token Bucket | తక్కువ | మంచిది | అద్భుతం |

## Step 3 — High-Level Architecture

```
Client → API Gateway (Rate Limiter Middleware) → Backend Services
                    ↕
              Redis Cluster
         (counters + token buckets)
```

<Illustration src="/images/lessons/design-rate-limiter.svg" alt="Rate limiter architecture — clients, rate limiting middleware తో API gateway, distributed state కోసం Redis cluster మరియు backend services చూపిస్తోంది" caption="Rate limiter API gateway లో ఉంటుంది, distributed counter storage కోసం Redis ఉపయోగిస్తుంది" />

### Rate Limiter ఎక్కడ ఉంచాలి
- **Client-side** — spoof చేయడం సులభం, నమ్మదగినది కాదు
- **Server-side** — per-instance మాత్రమే, scale కాదు
- **API Gateway / Middleware** — distributed systems కి అత్యుత్తమ approach; అన్ని instances లో centralised enforcement

## Step 4 — Deep Dive: Distributed Rate Limiting

### Redis + Lua Scripts
Distributed setup లో, multiple servers rate limiting state share చేయాలి. Redis standard choice ఎందుకంటే దాని atomic operations మరియు sub-millisecond latency.

Lua script atomicity నిర్ధారిస్తుంది — check-and-decrement ఒకే Redis operation లో జరుగుతుంది, race conditions నివారిస్తుంది:

```
-- Token Bucket in Redis (pseudocode)
local tokens = redis.call('GET', key)
local last_refill = redis.call('GET', key..':time')
-- Calculate tokens to add based on elapsed time
-- If tokens > 0: decrement and ALLOW
-- Else: DENY
```

### Response Headers
బాగా డిజైన్ చేసిన rate limiters clients కి state communicate చేస్తాయి:
- `X-RateLimit-Limit: 100` — per window maximum requests
- `X-RateLimit-Remaining: 42` — మిగిలిన requests
- `X-RateLimit-Reset: 1672531200` — window reset అయ్యే UTC epoch
- HTTP `429 Too Many Requests` తో `Retry-After` header

## Step 5 — Edge Cases Handle చేయడం

### Race Conditions
Atomic operations లేకుండా, concurrent requests limits bypass చేయవచ్చు. Solutions: Redis Lua scripts లేదా Redis `MULTI/EXEC` transactions.

### Rate Limiting Dimensions
ఏకకాలంలో multiple levels లో limits apply చేయండి:
- **Per user** — authenticated users కి 100 requests/minute
- **Per IP** — anonymous traffic కి 50 requests/minute
- **Per API key** — tiered plans (free: 10/min, pro: 1000/min)

### Distributed Clock Skew
Servers clocks కొంచెం different గా ఉండవచ్చు. Application server timestamps పై ఆధారపడకుండా Redis server time ను single source of truth గా వాడండి.

<FunFact>GitHub REST API authenticated users కి గంటకు 5,000 requests allow చేస్తుంది కానీ unauthenticated requests కి గంటకు 60 మాత్రమే. Stripe sliding window approach వాడుతుంది, live మరియు test mode API keys కి separate limits తో.</FunFact>

## Step 6 — Bottlenecks మరియు Optimisations

| Bottleneck | Solution |
|-----------|----------|
| Redis single point of failure | Replication తో Redis Cluster |
| Redis కి network latency | Periodic sync తో local in-memory cache |
| Hot keys (popular API keys) | Multiple Redis nodes లో shard చేయండి |
| Rule evaluation overhead | Rules pre-compile చేయండి, configurations cache చేయండి |
