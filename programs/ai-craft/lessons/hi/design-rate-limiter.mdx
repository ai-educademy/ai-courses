# Rate Limiter डिज़ाइन करें — APIs को Scale पर प्रोटेक्ट करना

Rate limiting सबसे practical system design topics में से एक है। हर production API — GitHub से लेकर Stripe तक — इस पर depend करती है ताकि abuse रोका जा सके, fair usage ensure हो, और backend services को overwhelm होने से बचाया जा सके।

## Step 1 — Requirements इकट्ठा करना

### Functional Requirements
- Configurable rules के basis पर requests limit करें (per user, IP, या API key)
- Client को throttle होने पर clear feedback दें (HTTP 429)
- एक साथ multiple rate limiting rules support करें

### Non-Functional Requirements
- **Low latency** — limiter को significant overhead नहीं add करना चाहिए
- **Distributed** — multiple servers पर काम करे
- **Fault-tolerant** — अगर limiter fail हो जाए, तो traffic flow होता रहे (fail-open)

<ThinkAboutIt>क्या rate limiter crash होने पर fail-open (traffic allow करे) होना चाहिए या fail-closed (traffic block करे)? एक e-commerce site vs banking API के लिए trade-offs क्या होंगे?</ThinkAboutIt>

## Step 2 — Rate Limiting Algorithms

### Fixed Window Counter
Time को fixed windows में divide करें (जैसे 60-second intervals)। हर request पर counter increment करें। Window boundary पर reset करें। **Problem:** window edges पर burst traffic 2× limit allow कर सकता है।

### Sliding Window Log
हर request का timestamp store करें। Trailing window के अंदर entries count करें। Precise लेकिन **memory-intensive** — scale पर हर timestamp store करना expensive है।

### Sliding Window Counter
एक hybrid approach: current window count को previous window के weighted portion के साथ combine करें। Accuracy और memory efficiency का balance बनाए रखता है।

### Token Bucket Algorithm (सबसे Popular)
एक bucket maximum capacity तक tokens hold करता है। Tokens एक fixed rate पर add होते हैं। हर request एक token consume करती है। अगर bucket empty है, तो request reject हो जाती है।

```
Bucket: capacity=10, refill_rate=2/sec

Time 0s: [🪙🪙🪙🪙🪙🪙🪙🪙🪙🪙] → 10 tokens
Burst of 8 requests → [🪙🪙] → 2 tokens remain
Time 1s: refill +2 → [🪙🪙🪙🪙] → 4 tokens
```

### Algorithm Comparison

| Algorithm | Memory | Accuracy | Burst Handling |
|-----------|--------|----------|----------------|
| Fixed Window | कम | कम (edge bursts) | खराब |
| Sliding Log | ज़्यादा | Exact | अच्छी |
| Sliding Counter | मध्यम | Approximate | अच्छी |
| Token Bucket | कम | अच्छी | बेहतरीन |

## Step 3 — High-Level Architecture

```
Client → API Gateway (Rate Limiter Middleware) → Backend Services
                    ↕
              Redis Cluster
         (counters + token buckets)
```

### Rate Limiter कहाँ रखें
- **Client-side** — spoof करना आसान, unreliable
- **Server-side** — सिर्फ per-instance, scale नहीं करता
- **API Gateway / Middleware** — distributed systems के लिए सबसे अच्छा approach; सभी instances पर centralised enforcement

## Step 4 — Deep Dive: Distributed Rate Limiting

### Redis + Lua Scripts
Distributed setup में, multiple servers को rate limiting state share करनी होती है। Redis standard choice है क्योंकि इसके atomic operations और sub-millisecond latency होती है।

Lua script atomicity ensure करता है — check-and-decrement एक single Redis operation में होता है, जिससे race conditions prevent होती हैं:

```
-- Token Bucket in Redis (pseudocode)
local tokens = redis.call('GET', key)
local last_refill = redis.call('GET', key..':time')
-- Calculate tokens to add based on elapsed time
-- If tokens > 0: decrement and ALLOW
-- Else: DENY
```

### Response Headers
अच्छे rate limiters clients को state communicate करते हैं:
- `X-RateLimit-Limit: 100` — per window maximum requests
- `X-RateLimit-Remaining: 42` — बची हुई requests
- `X-RateLimit-Reset: 1672531200` — UTC epoch जब window reset होगी
- HTTP `429 Too Many Requests` with a `Retry-After` header

<Quiz question="Rate limiting में Redis के साथ Lua scripts क्यों use होते हैं, अलग-अलग GET और SET commands की जगह?" options="Lua, Redis commands से faster है|Lua scripts Redis पर atomically execute होती हैं, race conditions prevent करती हैं|Redis natively GET/SET support नहीं करता|Lua network bandwidth कम करता है" answer="1" explanation="Lua scripts Redis server पर atomically run होती हैं। Atomicity के बिना, दो concurrent requests एक ही token count read कर सकती हैं, दोनों tokens available देखें, और दोनों decrement करें — जिससे limit से ज़्यादा requests allow हो जाएँ।" />

## Step 5 — Edge Cases Handle करना

### Race Conditions
Atomic operations के बिना, concurrent requests limits bypass कर सकती हैं। Solutions: Redis Lua scripts या Redis `MULTI/EXEC` transactions।

### Rate Limiting Dimensions
एक साथ multiple levels पर limits apply करें:
- **Per user** — authenticated users के लिए 100 requests/minute
- **Per IP** — anonymous traffic के लिए 50 requests/minute
- **Per API key** — tiered plans (free: 10/min, pro: 1000/min)

### Distributed Clock Skew
Servers के clocks थोड़े different हो सकते हैं। Application server timestamps पर depend करने की बजाय Redis server time को single source of truth के रूप में use करें।

<FunFact>GitHub की REST API authenticated users के लिए 5,000 requests per hour allow करती है लेकिन unauthenticated requests के लिए सिर्फ 60 per hour। Stripe sliding window approach use करता है जिसमें live और test mode API keys के लिए अलग-अलग limits होती हैं।</FunFact>

## Step 6 — Bottlenecks और Optimisations

| Bottleneck | Solution |
|-----------|----------|
| Redis single point of failure | Replication के साथ Redis Cluster |
| Redis तक network latency | Periodic sync के साथ local in-memory cache |
| Hot keys (popular API keys) | Multiple Redis nodes पर shard करें |
| Rule evaluation overhead | Rules pre-compile करें, configurations cache करें |

<ThinkAboutIt>अगर आप London, New York, और Tokyo में data centres वाली global API के लिए rate limiting डिज़ाइन कर रहे हैं — तो क्या आप single centralised Redis use करेंगे या regional Redis clusters? आप कौन से consistency trade-offs accept करेंगे?</ThinkAboutIt>
