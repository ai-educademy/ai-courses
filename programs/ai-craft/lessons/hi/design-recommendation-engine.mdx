## Interview Prompt

*"एक e-commerce platform के लिए recommendation engine डिज़ाइन करें जो 100 million users को personalised product suggestions serve करे।"*

यह question आपकी **machine learning pipelines** और **scalable system design** को combine करने की ability test करता है - senior-level interviews में यह combination काफ़ी common हो गया है।

## Step 1: Requirements Gathering

**Functional requirements:**
- Homepage पर personalised "Recommended for You" (top-20 products)
- Product pages पर "Customers who bought this also bought"
- Regional activity के basis पर "Trending in your area"
- Real-time updates: recommendations recent browsing को minutes में reflect करें

**Non-functional requirements:**
- 100ms (p99) से कम में recommendations serve करें
- Peak पर 50,000 requests per second handle करें
- 100 million users और 10 million products support करें
- Graceful degradation: अगर personalisation fail हो तो popular items दिखाएँ

<Callout>
हमेशा एक fallback strategy define करें। Interviewers को यह सुनना अच्छा लगता है - "अगर ML model fail हो जाए, तो हम gracefully popularity-based recommendations पर degrade करेंगे" - इससे production thinking दिखती है।
</Callout>

## Step 2: Recommendation Approaches

तीन core techniques ज़्यादातर recommendation systems को power करती हैं:

**Collaborative filtering** - "आपसे मिलते-जुलते users को ये items पसंद आए"
- User-item interaction matrices (views, purchases, ratings) analyse करें
- Matrix factorisation से similar users या similar items खोजें
- Sufficient interaction data होने पर अच्छे से काम करता है

**Content-based filtering** - "क्योंकि आपको इन features वाले items पसंद आए"
- Product attributes (category, brand, price range, description) use करें
- Transformer models से product descriptions के embeddings generate करें
- Embedding space में items के बीच similarity compute करें

**Hybrid approach** - stronger recommendations के लिए दोनों signals combine करें
- ज़्यादातर production systems hybrid use करते हैं: personalisation के लिए collaborative signals, diversity और cold-start coverage के लिए content signals

## Step 3: Recommendation Pipeline

Production recommendation systems आमतौर पर **two-stage architecture** use करते हैं:

**Stage 1 - Candidate Generation (offline/near-real-time)**
- 10 million products को ~1,000 candidates per user तक narrow करें
- Lightweight models use करें: collaborative filtering, embedding similarity
- Batch jobs (hourly/daily) या streaming से near-real-time चलता है

**Stage 2 - Ranking (real-time)**
- ~1,000 candidates को एक richer model से score करें
- Context consider करें: time of day, device, recent activity, cart contents
- Top-20 ranked items return करें
- 100ms latency budget में execute होना चाहिए

## Step 4: Data Model और Feature Store

Recommendations multiple data sources से compute किए गए rich features पर depend करती हैं:

**User features:** purchase history, browsing patterns, demographic segment, favourite categories
**Item features:** category, price, ratings, description embeddings, sales velocity
**Interaction features:** click-through rates, time spent viewing, add-to-cart rates

एक **feature store** (जैसे Feast, Tecton) provide करता है:
- **Offline features** - model training के लिए precomputed batch features
- **Online features** - real-time ranking के लिए low-latency feature retrieval
- **Consistency** - training और serving में same feature definitions

```
Feature Store Schema:
  user_features:    { user_id → [purchase_vector, category_prefs, avg_price, ...] }
  item_features:    { item_id → [embedding, category, price, popularity_score, ...] }
  interaction_features: { (user_id, item_id) → [clicks, views, last_interaction, ...] }
```

<FunFact>
Netflix का अनुमान है कि उसका recommendation system subscriber churn कम करके कंपनी को सालाना $1 billion बचाता है। जिन users को relevant suggestions मिलते हैं, वे अपनी subscription cancel करने की काफ़ी कम संभावना रखते हैं।
</FunFact>

## Step 5: Real-Time vs Batch Recommendations

| पहलू | Batch | Real-Time |
|-------|-------|-----------|
| Freshness | घंटों पुराना | सेकंडों पुराना |
| Compute cost | कम (scheduled jobs) | ज़्यादा (per-request inference) |
| Complexity | सरल pipeline | Streaming infrastructure ज़रूरी |
| Use case | Homepage "for you" | "Because you just viewed X" |

**Practical hybrid approach:**
- हर कुछ घंटों में batch में candidate lists pre-compute करें
- Latest session context से real time में re-rank करें
- Streaming (Kafka) use करके events आने पर user features near-real-time update करें

<ThinkAboutIt>
जब आप Amazon पर कोई product browse करते हैं, तो recommendations लगभग तुरंत update हो जाती हैं। लेकिन collaborative filtering models को train करने में घंटों लगते हैं। आपके हिसाब से Amazon batch model training और real-time personalisation के बीच इस gap को कैसे bridge करता है?
</ThinkAboutIt>

## Step 6: A/B Testing Framework

Recommendation systems को rigorous experimentation चाहिए:

- **Traffic splitting** - users का एक percentage नए model पर route करें
- **Metrics** - click-through rate (CTR), conversion rate, revenue per session, diversity
- **Statistical significance** - reliable conclusions तक पहुँचने के लिए tests काफ़ी समय तक चलाएँ
- **Guardrail metrics** - ensure करें कि नए models overall revenue या engagement degrade न करें

एक typical setup **feature flag service** use करता है जो users को experiment buckets में consistently assign करे (same user हमेशा test के दौरान same variant देखे)।

## Step 7: Cold-Start Solutions

Cold-start problem नए users और नए items दोनों के लिए आता है:

**नए users:**
- शुरू में popularity-based recommendations दिखाएँ
- Onboarding questions पूछें ("अपनी पसंदीदा categories select करें")
- Initial features के लिए demographic या geographic signals use करें
- 5-10 interactions के बाद personalised recommendations पर transition करें

**नए items:**
- Content-based features (description, category, price) से items को embedding space में place करें
- एक dedicated "New arrivals" section में नए items को boost करें
- Exploration/exploitation strategies चलाएँ (interaction data gather करने के लिए नए items को users के subset को दिखाएँ)

## Step 8: System को Scale करना

**50K requests per second के लिए architecture:**
- **API gateway** - rate limiting, authentication, request routing
- **Recommendation service** - stateless, load balancer के पीछे horizontally scaled
- **Feature store** - low-latency feature retrieval के लिए Redis cluster
- **Model serving** - memory में pre-loaded models (TensorFlow Serving, Triton)
- **Pre-computed cache** - हर user के top recommendations का Redis cache, batch jobs से refresh
- **Fallback** - cache miss और model timeout होने पर popularity-based defaults return करें

</FunFact>

## मुख्य बातें

- Two-stage architecture use करें: candidate generation (broad) → ranking (precise)
- Hybrid approaches (collaborative + content-based) single-method systems से बेहतर perform करते हैं
- Feature stores model training और real-time serving के बीच consistency ensure करते हैं
- A/B testing ज़रूरी है - सिर्फ़ ML metrics नहीं, business metrics के लिए optimise करें
- Cold-start strategies नए users और items के लिए खराब experience रोकती हैं
- Peak traffic handle करने के लिए aggressively cache करें और gracefully degrade करें
