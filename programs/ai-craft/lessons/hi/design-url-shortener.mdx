## Interview का Prompt

*"bit.ly जैसी एक URL shortening service डिज़ाइन करें जो लंबे URLs लेकर छोटे, unique codes बनाती है।"*

यह system design का सबसे common पहला सवाल है क्योंकि यह देखने में simple लगता है लेकिन core concepts कवर करता है: API design, encoding strategies, database choices, caching, और scaling। चलिए इसे Lesson 1 के structured framework से solve करते हैं।

## Step 1: Requirements इकट्ठा करना (5 मिनट)

**Functional requirements:**
- एक लंबे URL से एक छोटा unique URL generate करें
- Short URL से redirect करने पर original long URL मिले
- Users optional custom short codes सेट कर सकें
- Links एक configurable period के बाद expire हों (default: 5 साल)
- Basic analytics: click count, referrer, geography

**Non-functional requirements:**
- हर महीने 100 million URLs generate (~40 per second)
- Read-heavy: 100:1 read-to-write ratio → 4,000 redirects per second
- URL redirection latency 50ms से कम
- Short URLs जितने छोटे हो सकें उतने छोटे हों
- High availability — एक टूटा हुआ link terrible user experience है

<Callout>
हर system design interview की शुरुआत clarifying questions पूछकर करें। Explicit numbers लिखना (40 writes/sec, 4,000 reads/sec) दिखाता है कि आप scale के बारे में quantitatively सोच सकते हैं।
</Callout>

## Step 2: API Design (5 मिनट)

```
POST /api/shorten
Body: { "url": "https://example.com/very/long/path", "custom_code": "mylink", "expires_in": "30d" }
Response: { "short_url": "https://sho.rt/aB3kx9", "expires_at": "2025-02-15T00:00:00Z" }

GET /:code
Response: HTTP 301 Redirect → Location: https://example.com/very/long/path

GET /api/analytics/:code
Response: { "clicks": 14523, "created_at": "...", "top_referrers": [...] }
```

**HTTP 301 vs 302 क्यों?**
- **301 (Permanent Redirect)** — browsers इसे cache करते हैं, जिससे servers पर load कम होता है
- **302 (Temporary Redirect)** — browsers हमेशा हमारे servers को hit करते हैं, जिससे accurate analytics मिलती है

Click tracking वाली service के लिए **302 ज़्यादा बेहतर choice है** क्योंकि हर redirect हमारे servers से होकर गुज़रता है।

## Step 3: Encoding Strategy

मुख्य challenge: छोटे, unique codes कैसे generate करें?

**Base62 encoding** `[a-zA-Z0-9]` use करती है — 62 characters। 7-character code से हमें 62⁷ = **3.5 trillion** unique combinations मिलते हैं, हमारे scale के लिए काफ़ी से ज़्यादा।

**Option A — Counter-based:**
- Auto-incrementing counter (1, 2, 3, ...) को base62 में convert करें
- Counter 1000 → `g8` | Counter 1000000 → `4c92`
- **फ़ायदे:** guaranteed unique, simple, कोई collisions नहीं
- **नुकसान:** predictable (users अगला URL guess कर सकते हैं), centralised counter चाहिए

**Option B — Hash-based:**
- Long URL को MD5/SHA-256 से hash करें, base62 के पहले 7 characters लें
- **फ़ायदे:** stateless, कोई central coordination नहीं चाहिए
- **नुकसान:** collisions possible — database check करके retry करना होगा

**Option C — Pre-generated keys (recommended):**
- एक background service पहले से millions of unique base62 codes generate करके key pool में store करती है
- नया URL आने पर pool से अगला available code ले लें
- **फ़ायदे:** fast, कोई collision risk नहीं, predictable भी नहीं
- **नुकसान:** key generation service manage करनी होगी

<FunFact>
Original TinyURL service 2002 में launch हुई थी और आज भी चल रही है। एक समय पर यह हर दिन 10 लाख से ज़्यादा short URLs generate कर रही थी। Domain tinyurl.com इतना iconic बन गया कि "tinyurl" शब्द URL shortening के लिए generic term बन गया।
</FunFact>

## Step 4: Database Choice और Schema

हर महीने 100 million नए URLs और 5 साल retention के साथ, हम लगभग **6 billion records** store करेंगे। यह read-heavy workload है जिसमें simple key-value lookups होते हैं।

**Database choice: DynamoDB या Cassandra**
- Lookups हमेशा short code (partition key) से होते हैं — complex joins नहीं चाहिए
- दोनों massive read throughput handle करते हैं horizontal scaling के साथ
- दोनों partition key से sub-10ms lookups देते हैं

**Schema:**
```
Table: urls
  short_code (partition key): "aB3kx9"
  long_url: "https://example.com/very/long/path"
  user_id: "user_123"
  created_at: 1706000000
  expires_at: 1863000000
  click_count: 14523
```

Analytics के लिए, एक अलग time-series table individual click events store करती है:
```
Table: clicks
  short_code (partition key): "aB3kx9"
  timestamp (sort key): 1706000001
  referrer: "twitter.com"
  country: "GB"
  device: "mobile"
```

## Step 5: Caching Strategy

4,000 reads per second और heavy-tailed distribution (कुछ URLs को millions of clicks मिलते हैं, ज़्यादातर को बहुत कम) के साथ caching ज़रूरी है।

**Redis cache** सबसे frequently accessed mappings store करता है:
```
Key:   url:aB3kx9
Value: "https://example.com/very/long/path"
TTL:   24 hours
```

**Cache hit flow (fast path):**
1. Request आती है: `GET /aB3kx9`
2. Redis check → cache hit → 302 redirect return
3. Asynchronously click event Kafka में log

**Cache miss flow:**
1. Redis check → miss
2. Database query → found
3. Redis cache populate करें
4. 302 redirect return

80% cache hit rate के साथ, 4,000 में से सिर्फ़ 800 requests per second ही database तक पहुँचती हैं।

<Quiz question="आपका URL shortener 4,000 redirects per second serve करता है। Cache hit rate 80% है। Database को कितने reads per second handle करने होंगे?" options="4,000|3,200|800|0 — cache सब handle करता है" answer="2" explanation="80% cache hit rate के साथ, 20% requests cache miss करती हैं: 4,000 × 0.20 = 800 database reads per second।" />

## Step 6: Analytics और Click Tracking

Click tracking से redirects slow नहीं होने चाहिए। Redirect response तुरंत return होना चाहिए जबकि analytics asynchronously process हों।

**Architecture:**
1. Redirect handler click event को **Kafka** में publish करता है
2. एक click-processing service events consume करती है और:
   - URL record में `click_count` increment करती है
   - Clicks table में detailed click data लिखती है
   - Streaming aggregation से real-time dashboards update करती है

यह decoupling ensure करता है कि slow analytics pipeline कभी redirect latency को impact न करे।

<ThinkAboutIt>
URL shorteners का इस्तेमाल कभी-कभी malicious links छुपाने के लिए होता है। अगर आप bit.ly डिज़ाइन कर रहे होते, तो destination URLs को phishing या malware के लिए scan करने का safety check कैसे implement करते — बिना shortening process में significant latency जोड़े?
</ThinkAboutIt>

## Step 7: System को Scale करना

**Horizontal scaling approach:**
- **API servers** stateless हैं — load balancer के पीछे instances add करके scale करें
- **Key generation service** अलग microservice के रूप में चलती है अपने pre-generated codes के database के साथ
- **Redis cluster** consistent hashing से nodes में cache distribute करता है
- **Database sharding** short code के पहले character से data evenly distribute करती है

**Peak loads handle करना:**
- Viral events में एक URL को per minute millions of hits मिल सकते हैं
- Redis cache naturally इसे absorb करता है — एक cache entry unlimited reads serve करती है
- Kafka click events buffer करता है, analytics database पर write load smooth करता है

<ThinkAboutIt>
bit.ly जैसी companies हर महीने billions of redirects handle करती हैं। अगर कोई short URL billboard या TV ad पर share हो, तो traffic सेकंडों में zero से millions तक spike कर सकता है। Caching के अलावा, इन अचानक, unpredictable traffic bursts को absorb करने के लिए आप और कौन सी strategies use करेंगे?
</ThinkAboutIt>

<FunFact>
2009 में bit.ly हर दिन 80 million से ज़्यादा clicks process कर रहा था। जब Libya — जिसका country code .ly है — ने political instability के कारण .ly domains temporarily बंद किए, तो पूरे internet में panic मच गई क्योंकि millions of shortened links अचानक काम करना बंद कर दिए।
</FunFact>

## मुख्य बातें

- Framework follow करें: requirements → API → encoding → data model → architecture → deep dives
- 7 characters की Base62 से 3.5 trillion unique codes मिलते हैं — काफ़ी से ज़्यादा
- Pre-generated key pools collision issues और central counter bottlenecks से बचाते हैं
- Accurate click analytics चाहिए तो 302 redirects use करें (301 नहीं)
- Redis caching read-heavy workloads के लिए database load dramatically कम करती है
- Kafka से async processing करके analytics को redirect path से decouple करें
