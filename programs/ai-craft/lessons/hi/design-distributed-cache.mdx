# Distributed Cache डिज़ाइन करें — Redis Architecture शुरू से

Caching system performance बेहतर करने की सबसे effective technique है। एक अच्छी तरह डिज़ाइन किया गया distributed cache database load को 90%+ तक कम कर सकता है और response times को सैकड़ों milliseconds से sub-millisecond तक ला सकता है।

## Step 1 — Requirements इकट्ठा करना

### Functional Requirements
- Key-value pairs store करना optional TTL (time-to-live) के साथ
- Common operations support करना: GET, SET, DELETE, और bulk operations
- Horizontal scaling के लिए data को multiple nodes में distribute करना

### Non-Functional Requirements
- **Low latency** — cache hits के लिए sub-millisecond reads
- **High throughput** — प्रति node 100K+ operations per second
- **High availability** — cache failures से system down नहीं होना चाहिए
- **Scalability** — बिना downtime के nodes add करना

### कब Cache करें
- **Read-heavy workloads** — जब reads, writes से बहुत ज़्यादा हों
- **Expensive computations** — ML model inference results, complex aggregations
- **Frequently accessed data** — user sessions, product catalogues, configuration

## Step 2 — Caching Strategies

### Cache-Aside (Lazy Loading)
सबसे common pattern। Application पहले cache check करता है; miss होने पर database से read करता है, फिर cache populate करता है:

```
GET user:123 → Cache HIT → return cached data
GET user:456 → Cache MISS → query DB → SET cache → return data
```

**Pros:** सिर्फ़ वही cache होता है जो actually request किया जाता है। **Cons:** cache misses पर latency बढ़ती है; data stale हो सकता है।

### Read-Through
Cache खुद miss होने पर database reads handle करता है। Application सिर्फ़ cache layer से बात करती है, जो data source को abstract करता है।

### Write-Through
हर write cache और database दोनों में synchronously जाता है। Cache हमेशा consistent रहता है लेकिन write latency बढ़ती है।

### Write-Behind (Write-Back)
Writes तुरंत cache में जाते हैं; cache asynchronously batches में database को flush करता है। Write latency कम होती है लेकिन अगर cache node flush से पहले crash हो जाए तो data loss का risk है।

| Strategy | Consistency | Read Latency | Write Latency | Use Case |
|----------|-------------|-------------|---------------|----------|
| Cache-Aside | Eventual | Miss penalty | N/A | General purpose |
| Read-Through | Eventual | Miss penalty | N/A | Simplified reads |
| Write-Through | Strong | Fast | Slower | Consistency-critical |
| Write-Behind | Eventual | Fast | Fast | Write-heavy workloads |

## Step 3 — Eviction Policies

जब cache भर जाए, तो कौन सी entries हटाएँ?

- **LRU (Least Recently Used)** — सबसे लंबे समय से access न की गई entry को evict करता है। सबसे अच्छी general-purpose policy
- **LFU (Least Frequently Used)** — सबसे कम बार access की गई entry को evict करता है। Skewed access patterns के लिए बेहतर
- **TTL-based** — entries एक set time बाद expire होती हैं चाहे access हों या नहीं। Data freshness के लिए ज़रूरी
- **Random** — surprisingly effective और implement करने में बहुत simple

<FunFact>Redis एक approximated LRU algorithm use करता है जो कुछ keys sample करता है (default: 5) और उनमें से least recently used को evict करता है। यह true LRU जितना ही accurate है लेकिन access order track करने के लिए zero additional memory overhead लगती है।</FunFact>

## Step 4 — Distribution के लिए Consistent Hashing

### Simple Hashing की समस्या
`hash(key) % N` nodes के साथ, कोई node add या remove करने पर लगभग सारी keys remap हो जाती हैं, जिससे massive cache miss storm आता है।

### Virtual Nodes के साथ Consistent Hashing
Consistent hashing keys और nodes दोनों को एक ring पर map करता है। हर key ring पर अगले clockwise node को assign होती है:

```
Hash Ring (0 to 2^32):
     Node A (3 virtual nodes: A1, A2, A3)
     Node B (3 virtual nodes: B1, B2, B3)
     Node C (3 virtual nodes: C1, C2, C3)

Key "user:123" → hash → lands between B2 and A1 → assigned to Node A
```

**Virtual nodes** uneven distribution problem solve करते हैं। हर physical node को ring पर multiple positions मिलती हैं, जिससे कम physical nodes होने पर भी balanced load ensure होता है।

जब कोई node add या remove होता है, तो सिर्फ़ ring पर उसके adjacent keys remap होती हैं — लगभग `K/N` keys (जहाँ K total keys और N node count है), सारी नहीं।

## Step 5 — Cache Stampede Prevention

**Cache stampede** तब होता है जब कोई popular cached entry expire होती है और सैकड़ों concurrent requests एक साथ cache miss करती हैं और database को hit करती हैं।

### Solutions

**Mutex/Locking:** Cache miss होने पर एक distributed lock acquire करें। सिर्फ़ एक request database से fetch करती है; बाकी cache repopulate होने तक wait करती हैं।

**Probabilistic Early Expiry:** हर request को TTL expire होने से पहले cache refresh करने का एक small random chance होता है। इससे refresh load समय में spread होता है बजाय expiry पर thundering herd बनने के।

**Background Refresh:** एक background process popular cache entries को expire होने से पहले refresh करता है। Hot keys के लिए cache कभी empty नहीं होता।

<ThinkAboutIt>एक flash sale सोचिए जहाँ एक product page को 100,000 requests per second मिल रही हैं। Product details cache entry expire हो जाती है। Stampede prevention के बिना, सभी 100K requests एक साथ database hit करेंगी। आप कौन सी prevention strategy चुनेंगे, और क्यों?</ThinkAboutIt>

## Step 6 — Redis Cluster Architecture

Redis Cluster सबसे widely deployed distributed cache है:

- **16,384 hash slots** master nodes में distribute होते हैं
- हर key एक slot पर map होती है `CRC16(key) % 16384` से
- **Replication:** हर master के पास failover के लिए एक या ज़्यादा replicas होते हैं
- **Automatic failover:** अगर master fail होता है, तो उसका replica seconds में promote हो जाता है

```
Slots 0-5460:     Master A ←→ Replica A'
Slots 5461-10922: Master B ←→ Replica B'
Slots 10923-16383: Master C ←→ Replica C'
```

### Key Metrics Monitor करना
- **Hit ratio** — target 95%+; 80% से नीचे caching problem indicate करता है
- **Memory usage** — evictions से बचने के लिए 80% से नीचे रखें
- **Eviction rate** — ज़्यादा evictions मतलब cache undersized है
- **Latency percentiles** — cache hits के लिए p99 1ms से कम होना चाहिए

## Step 7 — Real-World Caching Patterns

| Pattern | Implementation | TTL |
|---------|---------------|-----|
| Session store | `SET session:{id} {data}` | 30 minutes |
| API response cache | `SET api:/users/123 {json}` | 5 minutes |
| Database query cache | `SET query:{hash} {result}` | 1 minute |
| Rate limiter counter | `INCR ratelimit:{user}:{window}` | Window duration |
| Feature flags | `HSET features {flag} {value}` | 10 seconds |

### Cache Invalidation
Computer science की सबसे कठिन समस्या (naming things के साथ):
- **TTL-based** — simple, TTL duration तक staleness tolerate करता है
- **Event-driven** — data बदलने पर invalidation events publish करें (Kafka या Redis Pub/Sub से)
- **Version-based** — cache key में version include करें; writes पर version increment करें ताकि पुरानी entries effectively invalidate हों
