# Web Crawler डिज़ाइन करें - Google इंटरनेट को कैसे Index करता है

एक web crawler व्यवस्थित तरीके से इंटरनेट ब्राउज़ करता है ताकि web pages को download और index किया जा सके। यह हर search engine की नींव है, और इसे डिज़ाइन करने से आप distributed systems, graph traversal, और real web की complexities को handle करना सीखते हैं।

## Step 1 - Requirements इकट्ठा करना

### Functional Requirements
- Seed URLs के एक set से शुरू करके web pages crawl करना
- Hyperlinks extract करके नए pages discover करना
- Downloaded content को indexing और processing के लिए store करना
- robots.txt और crawl politeness rules का पालन करना

### Non-Functional Requirements
- **Scale** - हर महीने 1 billion pages crawl करना
- **Politeness** - किसी भी single web server पर overload नहीं डालना
- **Freshness** - important pages को नियमित रूप से re-crawl करना
- **Robustness** - malformed HTML, infinite loops, और spider traps को handle करना

## Step 2 - High-Level Architecture

```
Seed URLs → URL Frontier (priority queue)
                  ↓
            URL Scheduler (politeness + priority)
                  ↓
         Fetcher (HTTP client pool)
                  ↓
         HTML Parser → Extract URLs → URL Filter
              ↓                           ↓
       Content Store              URL Dedup (Bloom Filter)
       (S3 / HDFS)                      ↓
                                 URL Frontier (loop)
```

## Step 3 - Crawl Loop

### BFS vs DFS Crawling
- **BFS (Breadth-First)** - web crawling के लिए preferred। जल्दी से कई domains discover करता है और किसी single site में deep फँसने से बचाता है
- **DFS (Depth-First)** - dynamically generated pages में infinitely deep जाने का risk है, बिना दूसरे domains discover किए

### URL Frontier Design
Frontier एक simple queue नहीं है - यह दो sub-components वाला एक sophisticated priority system है:

1. **Priority Queue (Front)** - URLs को importance के आधार पर rank करता है। इसमें PageRank, update frequency, और domain authority जैसे factors शामिल हैं
2. **Politeness Queue (Back)** - हर domain की अपनी FIFO queue होती है। एक scheduler अलग-अलग domain queues से pick करता है ताकि किसी single server पर ज़्यादा load न पड़े

```
Front (Priority):       Back (Politeness):
┌──────────────┐       ┌─── bbc.co.uk queue ───┐
│ High Priority │ ───→  │ url1 → url4 → url7    │
│ Med Priority  │ ───→  ├─── github.com queue ──┤
│ Low Priority  │ ───→  │ url2 → url5 → url8    │
└──────────────┘       ├─── wikipedia.org queue ┤
                        │ url3 → url6 → url9    │
                        └───────────────────────┘
```

## Step 4 - Politeness और Robots.txt

### Robots.txt का पालन करना
किसी भी domain को crawl करने से पहले, उसकी `robots.txt` file fetch और cache करें:

```
# Example robots.txt
User-agent: *
Disallow: /admin/
Disallow: /private/
Crawl-delay: 2
```

हर domain की robots.txt को TTL (24 घंटे) के साथ cache करें। `Disallow` rules और `Crawl-delay` directives का हमेशा पालन करें।

### Crawl Rate Limiting
Explicit `Crawl-delay` न होने पर भी politeness बनाए रखें:
- हर domain पर maximum 1 request per second (configurable)
- हर domain के लिए separate download queues use करें
- Response times track करें - अगर server slow हो जाए तो automatically back off करें

<FunFact>Google का web crawler (Googlebot) सैकड़ों billions pages crawl करता है। Polite रहने के लिए, यह server response times monitor करता है और अगर कोई site slow respond करने लगे तो crawl rate कम कर देता है - effectively website को crawler के खिलाफ "push back" करने देता है।</FunFact>

## Step 5 - Deduplication

### URL Deduplication (Bloom Filter)
Billions URLs के साथ, किसी URL को पहले देखा गया है या नहीं - यह check करने के लिए एक efficient data structure चाहिए। Bloom filter O(1) lookups देता है कम memory footprint के साथ:

- **Space:** 1 billion URLs के लिए ~1 GB, 1% false positive rate पर
- **Trade-off:** कभी-कभी page re-crawl हो सकता है (false positive) लेकिन कोई नया URL कभी miss नहीं होगा (no false negatives)

Check करने से पहले URLs normalise करें: fragments (#) हटाएँ, query parameters sort करें, hostname lowercase करें, और relative paths resolve करें।

### Content Deduplication (SimHash)
अलग-अलग URLs identical content serve कर सकते हैं (mirrors, copied articles)। SimHash page content का fingerprint generate करता है - similar fingerprints (छोटी Hamming distance) वाले pages likely duplicates हैं:

```
Page A content → SimHash → 10110010...
Page B content → SimHash → 10110011...
Hamming distance = 1 → likely duplicate, skip storing
```

## Step 6 - Distributed Crawling

### Horizontally Scale करना
एक single machine monthly billion pages crawl नहीं कर सकती। काम distribute करें:

- **Consistent hashing** domain names पर - हर domain को एक specific crawler node assign करता है। इससे politeness ensure होती है - एक given domain को सिर्फ़ एक node crawl करता है
- **URL frontier** hash ring के basis पर nodes में partition होता है
- **Coordination** एक lightweight service से जो node assignments manage करती है और nodes join/leave होने पर rebalance करती है

### DNS Resolution Caching
DNS lookups हर request में 10-100ms add करते हैं। Scale पर यह एक major bottleneck है:
- हर crawler node पर local DNS cache रखें
- Queued URLs के लिए batch DNS pre-resolution करें
- Public DNS servers पर overload से बचने के लिए dedicated DNS resolver use करें

### Spider Trap Detection
Malicious या poorly built sites infinite URL spaces create कर सकती हैं:
- `/page/1/page/1/page/1/...` - repeating path patterns
- Calendar pages जो हर future date के लिए URLs generate करते हैं
- URLs में Session IDs जो identical content के लिए unique URLs create करते हैं

**बचाव:** maximum URL length, हर domain के लिए maximum crawl depth, repetitive URL structures के लिए pattern detection, और domain-level page count limits।

<ThinkAboutIt>अगर आपको search index fresh रखना है, तो कौन से pages पहले re-crawl करेंगे? एक news site जो hourly update होती है vs एक Wikipedia article जो monthly बदलता है - इसके बारे में सोचें। Re-crawl scheduling prioritise करने के लिए आप कौन से signals use करेंगे?</ThinkAboutIt>

## Step 7 - Bottlenecks और Optimisations

| Bottleneck | Solution |
|-----------|----------|
| DNS resolution latency | Local DNS cache, batch pre-resolution |
| Slow/unresponsive servers | Timeouts (30s), async I/O, retries के बाद skip करें |
| Crawled content का storage | Pages compress करें, सिर्फ़ changed content store करें |
| URL frontier memory | Disk-backed priority queue (RocksDB) |
| Duplicate content | SimHash fingerprinting, canonical URL detection |

<ThinkAboutIt>Search engines को crawl freshness (pages बार-बार re-crawl करना) और discovery (नए pages ढूँढना) के बीच balance बनाना होता है। अगर आपके crawler का fixed budget per day 1 million pages है, तो आप उसे existing pages re-crawl करने और नए pages discover करने में कैसे बाँटेंगे?</ThinkAboutIt>
