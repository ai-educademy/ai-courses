## Interview Prompt

*"WhatsApp या Slack जैसी chat application डिज़ाइन करें जो one-to-one messaging, group chats, और online presence indicators support करे।"*

यह system design interviews में एक पसंदीदा सवाल है क्योंकि यह एक ही problem में real-time communication, connection management, storage at scale, और distributed system trade-offs - सब कुछ test करता है।

## Step 1: Requirements Gathering

**Functional requirements:**
- One-to-one messaging with delivery और read receipts
- Group chat (500 members तक)
- Online/offline presence indicators
- Message history और search
- Offline users के लिए push notifications

**Non-functional requirements:**
- Low latency: messages 200ms के अंदर deliver हों
- High availability: 99.99% uptime
- एक conversation के अंदर message ordering guaranteed
- 50 million concurrent connections support करे

<Callout>
Non-functional requirements को हमेशा specific numbers से quantify करें। "Fast delivery" vague है - "sub-200ms delivery" दिखाता है कि आपको domain की समझ है।
</Callout>

## Step 2: API Design

REST-only systems के विपरीत, chat को **bidirectional communication** चाहिए। हमें दो protocols की ज़रूरत है:

**REST API** (non-real-time operations के लिए):
```
POST /api/users/login
GET  /api/conversations
GET  /api/conversations/:id/messages?before=timestamp&limit=50
POST /api/groups
```

**WebSocket** (real-time messaging के लिए):
```
CONNECT  ws://chat.example.com/ws?token=jwt
SEND     { type: "message", to: "user_456", content: "Hello!" }
RECEIVE  { type: "message", from: "user_123", content: "Hello!", ts: 1706000000 }
SEND     { type: "ack", messageId: "msg_789", status: "read" }
```

## Step 3: WebSocket Connection Management

हर connected user एक persistent WebSocket connection maintain करता है। 50 million concurrent users पर यह एक बड़ी engineering challenge है।

**Connection servers** (जिन्हें gateway servers भी कहते हैं) WebSocket connections hold करते हैं। हर server लगभग 50,000-100,000 concurrent connections handle कर सकता है, यानी हमें 500-1,000 connection servers चाहिए।

एक **connection registry** (Redis cluster) हर user को उनके connection server से map करती है:
```
user_123 → ws-server-042
user_456 → ws-server-187
```

जब user A, user B को message भेजता है, flow यह होता है:
1. Message, A के connection server पर WebSocket से आता है
2. Server, Redis में B का connection server look up करता है
3. Message internal messaging से B के connection server को route होता है
4. B का server, B के WebSocket पर message push करता है

## Step 4: Message Storage

Chat applications बहुत बड़ी volume में data generate करती हैं। WhatsApp रोज़ 100 billion से ज़्यादा messages process करता है।

**Storage strategy:**
- **Recent messages** - fast datastore (Cassandra या HBase) में, conversation ID से partitioned
- **Message index** - Elasticsearch से search enable करता है
- **Media files** - object storage (S3) में CDN delivery के साथ

**Schema design (Cassandra):**
```
Table: messages
  partition_key: conversation_id
  clustering_key: message_timestamp (descending)
  columns: sender_id, content, type, status
```

यह design ensure करती है कि एक conversation के सारे messages same partition पर co-located हों, जिससे "पुराने messages load करो" queries बहुत efficient हो जाती हैं - बस एक partition में sequential read।

## Step 5: Presence System

कौन online है दिखाना simple लगता है लेकिन scale पर surprisingly tricky है। आप अपनी contact list के हर user के लिए हर कुछ seconds में database poll नहीं कर सकते।

**Heartbeat approach:**
1. Connected clients हर 30 seconds में WebSocket से heartbeat भेजते हैं
2. Server एक Redis key update करता है जिसमें 60 seconds का TTL होता है: `presence:user_123 → online (TTL 60s)`
3. अगर heartbeat नहीं आता, key expire हो जाती है और user offline दिखता है

**Status updates के लिए fan-out:**
जब कोई user online आता है, उसका status सभी online contacts को push होना चाहिए। 500 contacts वाले user के लिए, यह check करना पड़ता है कि कौन-से contacts online हैं और सिर्फ उन्हें presence updates भेजने हैं - **lazy fan-out** offline users को broadcast करने से बचाता है।

<FunFact>
WhatsApp के "last seen" feature ने दुनिया भर में unexpected social drama पैदा किया। कुछ देशों में relationship disputes बढ़ गए क्योंकि partners एक-दूसरे के exact last-active timestamps देख सकते थे। WhatsApp ने आखिरकार यह information छुपाने के लिए privacy controls add किए।
</FunFact>

## Step 6: Push Notifications

जब user offline हो (कोई active WebSocket नहीं), तो messages push notifications से पहुँचने चाहिए।

Message flow बदल जाता है:
1. Connection registry दिखाती है कि user B offline है
2. Message database में "undelivered" status के साथ store होता है
3. Notification service APNs (iOS) या FCM (Android) से push भेजती है
4. जब user B reconnect करता है, undelivered messages fetch करके WebSocket से भेजे जाते हैं
5. Receipt confirm करने के लिए delivery receipts भेजी जाती हैं

<ThinkAboutIt>
500 members वाले group chats में fan-out challenge होती है। अगर एक message ऐसे group में भेजा जाए जहाँ 400 members offline हैं, तो 400 push notifications भेजनी पड़ेंगी। Peak hours में notification services को overwhelm होने से बचाने के लिए आप इन्हें कैसे batch और rate-limit करेंगे?
</ThinkAboutIt>

## Step 7: E2E Encryption Basics

Interview discussions में E2E encryption का ज़िक्र security awareness दिखाता है:

- हर user अपने device पर **public/private key pair** generate करता है
- Messages recipient की public key से encrypt होते हैं sender का device छोड़ने से पहले
- Server को सिर्फ ciphertext दिखता है - वह message content नहीं पढ़ सकता
- Group chats एक **shared group key** use करते हैं जो members join या leave करने पर rotate होती है

इसका मतलब server के message storage में सिर्फ encrypted blobs होते हैं, जो data compliance के नज़रिये से actually simpler है।

## Step 8: Millions तक Scale करना

**Key scaling strategies:**
- **Connection servers** horizontally scale होते हैं - users बढ़ने पर servers add करें
- **Message routing** inter-server communication के लिए Kafka या Redis Pub/Sub use करती है
- **Database sharding** conversation ID से storage load distribute करती है
- **Geographic distribution** - latency कम करने के लिए multiple regions में connection servers deploy करें
- **Message queue buffering** - peak usage hours में spikes absorb करे

## Interview Tips

1. **WebSocket decision से शुरू करें** - explain करें कि इस scale पर HTTP polling क्यों काम नहीं करेगा
2. **Connection capacity quantify करें** - दिखाएं कि आपको server limits समझ हैं (~50-100K connections per server)
3. **Offline delivery address करें** - बहुत से candidates भूल जाते हैं कि users हमेशा connected नहीं होते
4. **Message ordering mention करें** - timestamp-based ordering with conflict resolution

<ThinkAboutIt>
Slack और WhatsApp message history के लिए बहुत अलग approaches लेते हैं। WhatsApp messages device पर store करता है और new devices पर full history sync नहीं करता। Slack सब कुछ server-side store करता है और full search देता है। दोनों approaches के privacy, cost, और user experience trade-offs क्या हैं?
</ThinkAboutIt>

## Key Takeaways

- Chat systems को real-time bidirectional communication के लिए WebSockets चाहिए
- Scale पर connection management के लिए users-to-servers mapping registry ज़रूरी है
- Cassandra की write throughput और partition-based queries chat message storage के लिए उपयुक्त हैं
- Presence systems Redis में TTL-based expiry के साथ heartbeats use करते हैं
- Push notifications offline message delivery handle करती हैं
- Geographic distribution और horizontal scaling global latency requirements address करते हैं
