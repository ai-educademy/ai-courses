# Design a Rate Limiter — Protecting APIs at Scale

Rate limiting is one of the most practical system design topics. Every production API — from GitHub to Stripe — relies on it to prevent abuse, ensure fair usage, and protect backend services from being overwhelmed.

## Step 1 — Requirements Gathering

### Functional Requirements
- Limit requests based on configurable rules (per user, IP, or API key)
- Return clear feedback when a client is throttled (HTTP 429)
- Support multiple rate limiting rules simultaneously

### Non-Functional Requirements
- **Low latency** — the limiter must not add significant overhead
- **Distributed** — works across multiple servers
- **Fault-tolerant** — if the limiter fails, traffic should still flow (fail-open)

## Step 2 — Rate Limiting Algorithms

### Fixed Window Counter
Divide time into fixed windows (e.g., 60-second intervals). Increment a counter per request. Reset at window boundaries. **Problem:** burst traffic at window edges can allow 2× the limit.

### Sliding Window Log
Store the timestamp of every request. Count entries within the trailing window. Precise but **memory-intensive** — storing every timestamp is expensive at scale.

### Sliding Window Counter
A hybrid: combine the current window count with a weighted portion of the previous window. Balances accuracy and memory efficiency.

### Token Bucket Algorithm (Most Popular)
A bucket holds tokens up to a maximum capacity. Tokens are added at a fixed rate. Each request consumes one token. If the bucket is empty, the request is rejected.

```
Bucket: capacity=10, refill_rate=2/sec

Time 0s: [🪙🪙🪙🪙🪙🪙🪙🪙🪙🪙] → 10 tokens
Burst of 8 requests → [🪙🪙] → 2 tokens remain
Time 1s: refill +2 → [🪙🪙🪙🪙] → 4 tokens
```

### Algorithm Comparison

| Algorithm | Memory | Accuracy | Burst Handling |
|-----------|--------|----------|----------------|
| Fixed Window | Low | Low (edge bursts) | Poor |
| Sliding Log | High | Exact | Good |
| Sliding Counter | Medium | Approximate | Good |
| Token Bucket | Low | Good | Excellent |

This makes it ideal for APIs that need to handle bursty traffic patterns." />

## Step 3 — High-Level Architecture

```
Client → API Gateway (Rate Limiter Middleware) → Backend Services
                    ↕
              Redis Cluster
         (counters + token buckets)
```

<Illustration src="/images/lessons/design-rate-limiter.svg" alt="Rate limiter architecture showing clients, API gateway with rate limiting middleware, Redis cluster for distributed state, and backend services" caption="Rate limiter sits in the API gateway, using Redis for distributed counter storage" />

### Where to Place the Rate Limiter
- **Client-side** — easy to spoof, unreliable
- **Server-side** — per-instance only, doesn't scale
- **API Gateway / Middleware** — the best approach for distributed systems; centralised enforcement across all instances

## Step 4 — Deep Dive: Distributed Rate Limiting

### Redis + Lua Scripts
In a distributed setup, multiple servers must share rate limiting state. Redis is the standard choice because of its atomic operations and sub-millisecond latency.

A Lua script ensures atomicity — the check-and-decrement happens in a single Redis operation, preventing race conditions:

```
-- Token Bucket in Redis (pseudocode)
local tokens = redis.call('GET', key)
local last_refill = redis.call('GET', key..':time')
-- Calculate tokens to add based on elapsed time
-- If tokens > 0: decrement and ALLOW
-- Else: DENY
```

### Response Headers
Well-designed rate limiters communicate state back to clients:
- `X-RateLimit-Limit: 100` — maximum requests per window
- `X-RateLimit-Remaining: 42` — requests left
- `X-RateLimit-Reset: 1672531200` — UTC epoch when the window resets
- HTTP `429 Too Many Requests` with a `Retry-After` header

<Quiz question="Why are Lua scripts used with Redis for rate limiting instead of separate GET and SET commands?" options="Lua is faster than Redis commands|Lua scripts execute atomically on Redis, preventing race conditions|Redis doesn't support GET/SET natively|Lua reduces network bandwidth" answer="1" explanation="Lua scripts run atomically on the Redis server. Without atomicity, two concurrent requests could both read the same token count, both see tokens available, and both decrement — allowing more requests than the limit permits." />

## Step 5 — Handling Edge Cases

### Race Conditions
Without atomic operations, concurrent requests can bypass limits. Solutions: Redis Lua scripts or Redis `MULTI/EXEC` transactions.

### Rate Limiting Dimensions
Apply limits at multiple levels simultaneously:
- **Per user** — 100 requests/minute for authenticated users
- **Per IP** — 50 requests/minute for anonymous traffic
- **Per API key** — tiered plans (free: 10/min, pro: 1000/min)

### Distributed Clock Skew
Servers may have slightly different clocks. Use Redis server time as the single source of truth rather than relying on application server timestamps.

<FunFact>GitHub's REST API allows 5,000 requests per hour for authenticated users but only 60 per hour for unauthenticated requests. Stripe uses a sliding window approach with separate limits for live and test mode API keys.</FunFact>

## Step 6 — Bottlenecks and Optimisations

| Bottleneck | Solution |
|-----------|----------|
| Redis single point of failure | Redis Cluster with replication |
| Network latency to Redis | Local in-memory cache with periodic sync |
| Hot keys (popular API keys) | Shard across multiple Redis nodes |
| Rule evaluation overhead | Pre-compile rules, cache configurations |
