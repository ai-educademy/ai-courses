## The Interview Prompt

*"Design a recommendation engine for an e-commerce platform that serves personalised product suggestions to 100 million users."*

This question tests your ability to blend **machine learning pipelines** with **scalable system design** - a combination increasingly common in senior-level interviews.

## Step 1: Requirements Gathering

**Functional requirements:**
- Personalised "Recommended for You" on the homepage (top-20 products)
- "Customers who bought this also bought" on product pages
- "Trending in your area" based on regional activity
- Real-time updates: recommendations reflect recent browsing within minutes

**Non-functional requirements:**
- Serve recommendations in under 100ms (p99)
- Handle 50,000 requests per second at peak
- Support 100 million users and 10 million products
- Graceful degradation: show popular items if personalisation fails

<Callout>
Always define a fallback strategy. Interviewers love hearing "if the ML model fails, we degrade gracefully to popularity-based recommendations" - it shows production thinking.
</Callout>

## Step 2: Recommendation Approaches

Three core techniques power most recommendation systems:

**Collaborative filtering** - "users similar to you liked these items"
- Analyse user-item interaction matrices (views, purchases, ratings)
- Find similar users or similar items using matrix factorisation
- Works well with sufficient interaction data

**Content-based filtering** - "because you liked items with these features"
- Use product attributes (category, brand, price range, description)
- Generate embeddings from product descriptions using transformer models
- Compute similarity between items in embedding space

**Hybrid approach** - combine both signals for stronger recommendations
- Most production systems use a hybrid: collaborative signals for personalisation, content signals for diversity and cold-start coverage

## Step 3: The Recommendation Pipeline

Production recommendation systems typically use a **two-stage architecture**:

**Stage 1 - Candidate Generation (offline/near-real-time)**
- Narrow 10 million products down to ~1,000 candidates per user
- Uses lightweight models: collaborative filtering, embedding similarity
- Runs as batch jobs (hourly/daily) or near-real-time with streaming

**Stage 2 - Ranking (real-time)**
- Score the ~1,000 candidates using a richer model
- Considers context: time of day, device, recent activity, cart contents
- Returns the top-20 ranked items
- Must execute within the 100ms latency budget

## Step 4: Data Model and Feature Store

Recommendations depend on rich features computed from multiple data sources:

**User features:** purchase history, browsing patterns, demographic segment, favourite categories
**Item features:** category, price, ratings, description embeddings, sales velocity
**Interaction features:** click-through rates, time spent viewing, add-to-cart rates

A **feature store** (e.g., Feast, Tecton) provides:
- **Offline features** - precomputed batch features for model training
- **Online features** - low-latency feature retrieval for real-time ranking
- **Consistency** - same feature definitions used in training and serving

```
Feature Store Schema:
  user_features:    { user_id → [purchase_vector, category_prefs, avg_price, ...] }
  item_features:    { item_id → [embedding, category, price, popularity_score, ...] }
  interaction_features: { (user_id, item_id) → [clicks, views, last_interaction, ...] }
```

Netflix estimates its recommendation system saves the company $1 billion annually by reducing subscriber churn. Users who receive relevant suggestions are significantly less likely to cancel their subscriptions.
</FunFact>

## Step 5: Real-Time vs Batch Recommendations

| Aspect | Batch | Real-Time |
|--------|-------|-----------|
| Freshness | Hours old | Seconds old |
| Compute cost | Lower (scheduled jobs) | Higher (per-request inference) |
| Complexity | Simpler pipeline | Requires streaming infrastructure |
| Use case | Homepage "for you" | "Because you just viewed X" |

**Practical hybrid approach:**
- Pre-compute candidate lists in batch every few hours
- Re-rank in real time using the latest session context
- Use streaming (Kafka) to update user features near-real-time as events occur

## Step 6: A/B Testing Framework

Recommendation systems need rigorous experimentation:

- **Traffic splitting** - route a percentage of users to the new model
- **Metrics** - click-through rate (CTR), conversion rate, revenue per session, diversity
- **Statistical significance** - run tests long enough to reach reliable conclusions
- **Guardrail metrics** - ensure new models don't degrade overall revenue or engagement

A typical setup uses a **feature flag service** to assign users to experiment buckets consistently (same user always sees the same variant during a test).

## Step 7: Cold-Start Solutions

The cold-start problem appears for both new users and new items:

**New users:**
- Show popularity-based recommendations initially
- Ask onboarding questions ("Select categories you're interested in")
- Use demographic or geographic signals as initial features
- Transition to personalised recommendations after 5-10 interactions

**New items:**
- Use content-based features (description, category, price) to place items in embedding space
- Boost new items in a dedicated "New arrivals" section
- Run exploration/exploitation strategies (show new items to a subset of users to gather interaction data)

## Step 8: Scaling the System

**Architecture for 50K requests per second:**
- **API gateway** - rate limiting, authentication, request routing
- **Recommendation service** - stateless, horizontally scaled behind a load balancer
- **Feature store** - Redis cluster for low-latency feature retrieval
- **Model serving** - pre-loaded models in memory (TensorFlow Serving, Triton)
- **Pre-computed cache** - Redis cache of top recommendations per user, refreshed by batch jobs
- **Fallback** - if cache miss and model timeout, return popularity-based defaults

<Quiz question="Your recommendation service averages 40ms latency but spikes to 500ms under peak load. ## Key Takeaways

- Use a two-stage architecture: candidate generation (broad) → ranking (precise)
- Hybrid approaches (collaborative + content-based) outperform single-method systems
- Feature stores ensure consistency between model training and real-time serving
- A/B testing is essential - optimise for business metrics, not just ML metrics
- Cold-start strategies prevent poor experiences for new users and items
- Cache aggressively and degrade gracefully to handle peak traffic
