## The Interview Prompt

*"Design an AI API platform that serves multiple machine learning models to developers via a REST API - similar to how OpenAI or Hugging Face serve models at scale."*

This question is increasingly popular in interviews because it combines **API platform design** with **GPU infrastructure challenges** unique to AI workloads.

## Step 1: Requirements Gathering

**Functional requirements:**
- Serve multiple AI models (text generation, image classification, embeddings) via REST API
- API key authentication with per-key usage quotas
- Usage tracking and billing per API call
- Support multiple model versions with A/B testing
- Streaming responses for long-running generation tasks

**Non-functional requirements:**
- Latency: p50 under 500ms for embedding models, p50 under 2s for generative models
- Throughput: 10,000 requests per second across all models
- Availability: 99.9% uptime with graceful degradation
- Cost efficiency: GPUs are expensive - maximise utilisation

<Callout>
AI platforms have uniquely variable latency. A text completion request might take 200ms or 30 seconds depending on output length. Always discuss how your design handles this variance.
</Callout>

## Step 2: API Design

```
POST /v1/completions
Headers: Authorization: Bearer sk-abc123
Body: { "model": "gpt-4", "prompt": "Explain caching", "max_tokens": 200 }

POST /v1/embeddings
Headers: Authorization: Bearer sk-abc123
Body: { "model": "text-embedding-3", "input": "Hello world" }

GET  /v1/models                    - list available models
GET  /v1/usage?start=2024-01-01    - usage and billing data

SSE: POST /v1/completions (stream: true)
     → returns Server-Sent Events with incremental tokens
```

Streaming via **Server-Sent Events (SSE)** is critical for generative models - users see tokens appear progressively rather than waiting for the full response.

## Step 3: API Gateway and Authentication

The API gateway is the front door to the platform:

- **Authentication** - validate API keys against a key store (Redis for speed, database for persistence)
- **Rate limiting** - enforce per-key limits (e.g., 60 requests/minute for free tier)
- **Request routing** - direct requests to the correct model-serving cluster
- **Request validation** - check payload schema, model availability, token limits

**Rate limiting implementation:**
Use a **sliding window counter** in Redis:
```
Key:   rate:{api_key}:{window_minute}
Value: request count
TTL:   120 seconds
```

Each request increments the counter. If it exceeds the limit, return HTTP 429.

## Step 4: Model Serving and GPU Allocation

This is where AI platforms differ fundamentally from traditional web services. GPUs are scarce, expensive, and require specialised orchestration.

**Model serving layer:**
- Models are loaded into GPU memory using frameworks like **Triton Inference Server** or **vLLM**
- Each model version runs on a dedicated pool of GPU instances
- Large language models (LLMs) may require **model parallelism** across multiple GPUs

**Request batching:**
Individual inference requests are inefficient on GPUs. **Dynamic batching** groups multiple requests and processes them together:
- Collect requests for up to 10ms or until the batch reaches 32 requests
- Process the entire batch in one GPU forward pass
- Return individual responses to each caller

This can improve throughput by 5-10x compared to processing requests individually.

<FunFact>
A single NVIDIA H100 GPU costs roughly £25,000 and consumes 700 watts of power. Running a cluster of 1,000 GPUs for model serving costs over £2 million per month in electricity and hardware alone - which is why efficient batching and utilisation matter enormously.
</FunFact>

## Step 5: A/B Testing Between Model Versions

When deploying a new model version, you need controlled rollout:

- **Traffic splitting** - route 5% of requests to the new model, 95% to the current one
- **Shadow mode** - run the new model on production traffic without returning its results to users
- **Metrics comparison** - latency, output quality scores, error rates, cost per request
- **Gradual rollout** - increase traffic to the new version as confidence grows

The routing logic lives in the API gateway:
```
model: "gpt-4"  →  95% → gpt-4-v2.1 (current)
                →   5% → gpt-4-v2.2 (candidate)
```

Evaluating whether a new LLM version is "better" is surprisingly difficult - unlike classification models, there's no simple accuracy metric. How would you design an automated quality evaluation system for generative AI models?
</ThinkAboutIt>

## Step 6: Usage Tracking and Billing

AI APIs typically bill per token (input + output tokens for LLMs, per request for other models).

**Event-driven billing pipeline:**
1. Each API response includes a usage header: `x-usage-tokens: 847`
2. A billing event is published to Kafka: `{ api_key, model, input_tokens, output_tokens, timestamp }`
3. A billing aggregation service consumes events and updates running totals
4. Usage dashboards query aggregated data from a time-series database

**Why Kafka?** Billing must be reliable but doesn't need to be synchronous. If the billing service is temporarily down, events queue in Kafka and are processed when it recovers - no usage data is lost.

<Quiz question="Your billing pipeline processes usage events asynchronously via Kafka. A customer checks their usage dashboard and sees a number lower than expected. What's the most likely cause?" options="Events were lost due to a Kafka partition failure|The aggregation service has processing lag - events haven't been consumed yet|The API gateway failed to emit usage events|The customer's API key was rate-limited" answer="1" explanation="Asynchronous pipelines inherently have lag. Events published to Kafka may take seconds or minutes to be consumed and aggregated, so the dashboard shows a slightly delayed view of usage." />

## Step 7: Monitoring and Observability

AI platforms require monitoring beyond standard web services:

| Metric | Why It Matters |
|--------|----------------|
| Inference latency (p50, p95, p99) | User experience and SLA compliance |
| GPU utilisation | Cost efficiency - idle GPUs waste money |
| Tokens per second | Throughput capacity and scaling signals |
| Error rate by model | Detect model-specific failures |
| Queue depth | Scaling trigger - deep queues mean demand exceeds capacity |
| Output quality scores | Detect model degradation (via automated eval) |

**Alerting thresholds:** GPU utilisation below 40% triggers scale-down; queue depth above 100 triggers scale-up.

## Step 8: Auto-Scaling

AI workloads have highly variable demand. Auto-scaling must balance responsiveness with cost:

- **Horizontal scaling** - add more GPU instances when queue depth or latency exceeds thresholds
- **Predictive scaling** - use historical traffic patterns to pre-provision capacity before expected peaks
- **Scale-to-zero** - for rarely used models, spin down all instances and cold-start on demand
- **Warm pools** - keep a small number of instances with models pre-loaded to avoid cold-start latency

The scale-to-zero strategy is particularly important for platforms hosting hundreds of models - you can't keep every model loaded in GPU memory at all times.

<Quiz question="A model that receives 2 requests per hour currently runs on one always-on GPU instance costing £3/hour. Cold-starting from zero takes 45 seconds. What's the best scaling strategy?" options="Keep the instance always on - 45 seconds is too long to wait|Scale to zero and accept 45-second cold starts for the first request|Use a warm pool of one CPU instance that offloads to GPU on demand|Batch the 2 requests into a single scheduled job" answer="1" explanation="At 2 requests per hour, the GPU sits idle 99.9% of the time. Scaling to zero and accepting a 45-second cold start saves nearly £2,100/month per model - well worth the occasional delay for a low-traffic model." />

## Key Takeaways

- AI API platforms combine traditional API design with GPU-specific infrastructure challenges
- Request batching dramatically improves GPU throughput and cost efficiency
- Rate limiting protects both the platform and customers from runaway costs
- Asynchronous billing via Kafka ensures reliable usage tracking without blocking requests
- Auto-scaling must balance responsiveness with cost - scale-to-zero for low-traffic models
- Monitoring GPU utilisation and queue depth drives intelligent scaling decisions
