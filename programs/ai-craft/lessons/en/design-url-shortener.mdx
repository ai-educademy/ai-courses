## The Interview Prompt

*"Design a URL shortening service like bit.ly that takes long URLs and produces short, unique codes."*

This is the most common first system design question because it's deceptively simple yet covers core concepts: API design, encoding strategies, database choices, caching, and scaling. Let's work through it using the structured framework from Lesson 1.

## Step 1: Requirements Gathering (5 Minutes)

**Functional requirements:**
- Given a long URL, generate a short unique URL
- Redirecting via the short URL returns the original long URL
- Users can optionally set custom short codes
- Links expire after a configurable period (default: 5 years)
- Basic analytics: click count, referrer, geography

**Non-functional requirements:**
- 100 million URLs generated per month (~40 per second)
- Read-heavy: 100:1 read-to-write ratio → 4,000 redirects per second
- URL redirection latency under 50ms
- Short URLs should be as short as possible
- High availability — a broken link is a terrible user experience

<Callout>
Start every system design interview by asking clarifying questions. Writing down explicit numbers (40 writes/sec, 4,000 reads/sec) shows you can reason quantitatively about scale.
</Callout>

## Step 2: API Design (5 Minutes)

```
POST /api/shorten
Body: { "url": "https://example.com/very/long/path", "custom_code": "mylink", "expires_in": "30d" }
Response: { "short_url": "https://sho.rt/aB3kx9", "expires_at": "2025-02-15T00:00:00Z" }

GET /:code
Response: HTTP 301 Redirect → Location: https://example.com/very/long/path

GET /api/analytics/:code
Response: { "clicks": 14523, "created_at": "...", "top_referrers": [...] }
```

**Why HTTP 301 vs 302?**
- **301 (Permanent Redirect)** — browsers cache it, reducing load on our servers
- **302 (Temporary Redirect)** — browsers always hit our servers, enabling accurate analytics

For a service that tracks clicks, **302 is usually the better choice** since every redirect passes through our servers.

## Step 3: Encoding Strategy

The core challenge: how do we generate short, unique codes?

**Base62 encoding** uses `[a-zA-Z0-9]` — 62 characters. A 7-character code gives us 62⁷ = **3.5 trillion** unique combinations, more than enough for our scale.

**Option A — Counter-based:**
- Use an auto-incrementing counter (1, 2, 3, ...) and convert to base62
- Counter 1000 → `g8` | Counter 1000000 → `4c92`
- **Pros:** guaranteed unique, simple, no collisions
- **Cons:** predictable (users can guess next URL), requires a centralised counter

**Option B — Hash-based:**
- Hash the long URL with MD5/SHA-256, take the first 7 characters of base62
- **Pros:** stateless, no central coordination needed
- **Cons:** possible collisions — must check database and retry

**Option C — Pre-generated keys (recommended):**
- A background service pre-generates millions of unique base62 codes and stores them in a key pool
- When a new URL arrives, grab the next available code from the pool
- **Pros:** fast, no collision risk, no predictability
- **Cons:** requires managing the key generation service

<FunFact>
The original TinyURL service launched in 2002 and is still running today. At one point, it was generating over 1 million short URLs per day. The domain itself — tinyurl.com — has become so iconic that "tinyurl" is used as a generic term for URL shortening.
</FunFact>

## Step 4: Database Choice and Schema

With 100 million new URLs per month and a 5-year retention, we'll store roughly **6 billion records**. This is a read-heavy workload with simple key-value lookups.

**Database choice: DynamoDB or Cassandra**
- Lookups are always by short code (partition key) — no complex joins needed
- Both handle massive read throughput with horizontal scaling
- Both offer sub-10ms lookups by partition key

**Schema:**
```
Table: urls
  short_code (partition key): "aB3kx9"
  long_url: "https://example.com/very/long/path"
  user_id: "user_123"
  created_at: 1706000000
  expires_at: 1863000000
  click_count: 14523
```

For analytics, a separate time-series table stores individual click events:
```
Table: clicks
  short_code (partition key): "aB3kx9"
  timestamp (sort key): 1706000001
  referrer: "twitter.com"
  country: "GB"
  device: "mobile"
```

## Step 5: Caching Strategy

At 4,000 reads per second and a heavy-tailed distribution (some URLs get millions of clicks, most get very few), caching is essential.

**Redis cache** stores the most frequently accessed mappings:
```
Key:   url:aB3kx9
Value: "https://example.com/very/long/path"
TTL:   24 hours
```

**Cache hit flow (fast path):**
1. Request arrives: `GET /aB3kx9`
2. Check Redis → cache hit → return 302 redirect
3. Asynchronously log the click event to Kafka

**Cache miss flow:**
1. Check Redis → miss
2. Query database → found
3. Populate Redis cache
4. Return 302 redirect

With an 80% cache hit rate, only 800 of the 4,000 requests per second actually reach the database.

<Quiz question="Your URL shortener serves 4,000 redirects per second. The cache hit rate is 80%. How many database reads per second does the database need to handle?" options="4,000|3,200|800|0 — the cache handles everything" answer="2" explanation="With an 80% cache hit rate, 20% of requests miss the cache: 4,000 × 0.20 = 800 database reads per second." />

## Step 6: Analytics and Click Tracking

Click tracking must not slow down redirects. The redirect response should return immediately while analytics are processed asynchronously.

**Architecture:**
1. Redirect handler publishes a click event to **Kafka**
2. A click-processing service consumes events and:
   - Increments the `click_count` in the URL record
   - Writes detailed click data to the clicks table
   - Updates real-time dashboards via a streaming aggregation

This decoupling ensures a slow analytics pipeline never impacts redirect latency.

<ThinkAboutIt>
URL shorteners are sometimes used to disguise malicious links. If you were designing bit.ly, how would you implement a safety check that scans destination URLs for phishing or malware without adding significant latency to the shortening process?
</ThinkAboutIt>

## Step 7: Scaling the System

**Horizontal scaling approach:**
- **API servers** are stateless — scale by adding instances behind a load balancer
- **Key generation service** runs as a separate microservice with its own database of pre-generated codes
- **Redis cluster** handles cache distribution across nodes using consistent hashing
- **Database sharding** by the first character of the short code distributes data evenly

**Handling peak loads:**
- During viral events, a single URL might receive millions of hits per minute
- The Redis cache absorbs this naturally — one cache entry serves unlimited reads
- Kafka buffers click events, smoothing the write load on the analytics database

<Quiz question="A URL goes viral and receives 500,000 clicks per minute. Which component is most likely to remain unaffected?" options="The Redis cache (single key serves all requests)|The Kafka click-event consumers|The analytics database write path|The API server CPU" answer="0" explanation="Redis handles millions of reads per second for a single cached key effortlessly. The Kafka consumers and analytics database face 500K writes/minute, and API servers handle 500K requests/minute — all under significant load." />

<ThinkAboutIt>
Companies like bit.ly handle billions of redirects monthly. If a short URL is shared on a billboard or TV advert, traffic can spike from zero to millions in seconds. Beyond caching, what other strategies would you use to absorb these sudden, unpredictable traffic bursts?
</ThinkAboutIt>

<FunFact>

## Key Takeaways

- Follow the framework: requirements → API → encoding → data model → architecture → deep dives
- Base62 with 7 characters yields 3.5 trillion unique codes — more than sufficient
- Pre-generated key pools avoid collision issues and central counter bottlenecks
- Use 302 redirects (not 301) when you need accurate click analytics
- Redis caching dramatically reduces database load for read-heavy workloads
- Decouple analytics from the redirect path using Kafka for async processing
