## The Interview Prompt

*"Design a chat application like WhatsApp or Slack that supports one-to-one messaging, group chats, and online presence indicators."*

This is a favourite in system design interviews because it tests real-time communication, connection management, storage at scale, and distributed system trade-offs — all in one problem.

## Step 1: Requirements Gathering

**Functional requirements:**
- One-to-one messaging with delivery and read receipts
- Group chat (up to 500 members)
- Online/offline presence indicators
- Message history and search
- Push notifications for offline users

**Non-functional requirements:**
- Low latency: messages delivered within 200ms
- High availability: 99.99% uptime
- Message ordering guaranteed within a conversation
- Support 50 million concurrent connections

<Callout>
Always quantify your non-functional requirements with specific numbers. "Fast delivery" is vague — "sub-200ms delivery" shows you understand the domain.
</Callout>

## Step 2: API Design

Unlike REST-only systems, chat requires **bidirectional communication**. We need two protocols:

**REST API** (for non-real-time operations):
```
POST /api/users/login
GET  /api/conversations
GET  /api/conversations/:id/messages?before=timestamp&limit=50
POST /api/groups
```

**WebSocket** (for real-time messaging):
```
CONNECT  ws://chat.example.com/ws?token=jwt
SEND     { type: "message", to: "user_456", content: "Hello!" }
RECEIVE  { type: "message", from: "user_123", content: "Hello!", ts: 1706000000 }
SEND     { type: "ack", messageId: "msg_789", status: "read" }
```

## Step 3: WebSocket Connection Management

Each connected user maintains a persistent WebSocket connection. At 50 million concurrent users, this is a significant engineering challenge.

**Connection servers** (sometimes called gateway servers) hold WebSocket connections. Each server can handle roughly 50,000-100,000 concurrent connections, meaning we need 500-1,000 connection servers.

A **connection registry** (Redis cluster) maps each user to their connection server:
```
user_123 → ws-server-042
user_456 → ws-server-187
```

When user A sends a message to user B, the flow is:
1. Message arrives at A's connection server via WebSocket
2. Server looks up B's connection server in Redis
3. Message is routed to B's connection server via internal messaging
4. B's server pushes the message down B's WebSocket

## Step 4: Message Storage

Chat applications generate enormous volumes of data. WhatsApp processes over 100 billion messages daily.

**Storage strategy:**
- **Recent messages** — kept in a fast datastore (Cassandra or HBase) partitioned by conversation ID
- **Message index** — enables search via Elasticsearch
- **Media files** — stored in object storage (S3) with CDN delivery

**Schema design (Cassandra):**
```
Table: messages
  partition_key: conversation_id
  clustering_key: message_timestamp (descending)
  columns: sender_id, content, type, status
```

This design ensures all messages in a conversation are co-located on the same partition, making "load older messages" queries extremely efficient — just a sequential read within one partition.

## Step 5: Presence System

Showing who's online seems simple but is surprisingly tricky at scale. You can't poll the database for every user in your contact list every few seconds.

**Heartbeat approach:**
1. Connected clients send a heartbeat every 30 seconds via WebSocket
2. The server updates a Redis key with a TTL of 60 seconds: `presence:user_123 → online (TTL 60s)`
3. If no heartbeat arrives, the key expires and the user appears offline

**Fan-out for status updates:**
When a user comes online, their status must be pushed to all contacts who are also online. For a user with 500 contacts, this means checking which contacts are online and sending presence updates only to those — **lazy fan-out** avoids broadcasting to offline users.

<FunFact>
WhatsApp's "last seen" feature caused unexpected social drama globally. Some countries saw relationship disputes spike because partners could see each other's exact last-active timestamps. WhatsApp eventually added privacy controls to hide this information.
</FunFact>

## Step 6: Push Notifications

When a user is offline (no active WebSocket), messages must still reach them via push notifications.

The message flow changes:
1. Connection registry shows user B is offline
2. Message is stored in the database with status "undelivered"
3. A notification service sends a push via APNs (iOS) or FCM (Android)
4. When user B reconnects, undelivered messages are fetched and sent via WebSocket
5. Delivery receipts are sent back to confirm receipt

<ThinkAboutIt>
Group chats with 500 members present a fan-out challenge. If a message is sent to a group where 400 members are offline, that's 400 push notifications. How would you batch and rate-limit these to avoid overwhelming notification services during peak hours?
</ThinkAboutIt>

## Step 7: End-to-End Encryption Basics

In interview discussions, briefly mentioning E2E encryption shows security awareness:

- Each user generates a **public/private key pair** on their device
- Messages are encrypted with the recipient's public key before leaving the sender's device
- The server only sees ciphertext — it cannot read message content
- Group chats use a **shared group key** that rotates when members join or leave

This means the server's message storage contains only encrypted blobs, which is actually simpler from a data compliance perspective.

## Step 8: Scaling to Millions

**Key scaling strategies:**
- **Connection servers** scale horizontally — add more servers as users grow
- **Message routing** uses Kafka or Redis Pub/Sub for inter-server communication
- **Database sharding** by conversation ID distributes storage load
- **Geographic distribution** — deploy connection servers in multiple regions to reduce latency
- **Message queue buffering** — absorb spikes during peak usage hours

## Interview Tips for Chat System Design

1. **Start with the WebSocket decision** — explain why HTTP polling won't work at this scale
2. **Quantify connection capacity** — show you understand server limits (~50-100K connections per server)
3. **Address offline delivery** — many candidates forget that users aren't always connected
4. **Mention message ordering** — timestamp-based ordering with conflict resolution

<ThinkAboutIt>
Slack and WhatsApp take very different approaches to message history. WhatsApp stores messages on-device and doesn't sync full history to new devices. Slack stores everything server-side and provides full search. What are the privacy, cost, and user experience trade-offs of each approach?
</ThinkAboutIt>

## Key Takeaways

- Chat systems require WebSockets for real-time bidirectional communication
- Connection management at scale needs a registry mapping users to servers
- Cassandra's write throughput and partition-based queries suit chat message storage
- Presence systems use heartbeats with TTL-based expiry in Redis
- Push notifications handle offline message delivery
- Geographic distribution and horizontal scaling address global latency requirements
