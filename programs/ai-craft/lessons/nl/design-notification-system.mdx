# Ontwerp een notificatiesysteem - Push, Email en SMS op miljardenschaal

Moderne applicaties versturen dagelijks miljarden notificaties via meerdere kanalen. Het ontwerpen van dit systeem leert je over message queues, integraties met externe diensten en het elegant afhandelen van fouten op grote schaal.

## Stap 1 - Requirements verzamelen

### Functionele requirements
- Ondersteuning van meerdere kanalen: push notifications, email, SMS en in-app
- Gebruikers kunnen notificatievoorkeuren instellen en zich per kanaal afmelden
- Berichten met templates en personalisatie (bijv. "Hoi {name}, je bestelling is verzonden")
- Ondersteuning voor geplande en directe notificaties

### Niet-functionele requirements
- **Schaalbaarheid** - meer dan 10 miljoen notificaties per dag verwerken
- **Betrouwbaarheid** - geen verloren notificaties; at-least-once delivery
- **Lage latency** - urgente notificaties binnen seconden bezorgd
- **Uitbreidbaar** - eenvoudig nieuwe kanalen toevoegen (WhatsApp, Slack)

<ThinkAboutIt>Moet de bezorging van notificaties exactly-once of at-least-once zijn? Denk na over de gebruikerservaring: een dubbele SMS ontvangen versus een kritieke beveiligingsmelding missen.</ThinkAboutIt>

## Stap 2 - High-level architectuur

```
Caller Services → Notification Service → Validation + Preference Check
                                              ↓
                                    Message Queue (Kafka)
                                    ↓       ↓       ↓       ↓
                                 Push    Email    SMS    In-App
                                Worker   Worker  Worker  Worker
                                  ↓        ↓       ↓       ↓
                                 FCM   SendGrid  Twilio  WebSocket
```

<Illustration src="/images/lessons/design-notification-system.svg" alt="Architectuur van het notificatiesysteem met caller services, notification service, Kafka message queues, kanaalspecifieke workers en externe providers" caption="Message queues ontkoppelen de notification service van delivery workers, waardoor elk kanaal onafhankelijk kan schalen" />

### Kerncomponenten
1. **Notification Service** - valideert verzoeken, controleert gebruikersvoorkeuren, rendert templates en plaatst berichten in de wachtrij
2. **Message Queues** - aparte wachtrijen per kanaal voor onafhankelijke schaalbaarheid en isolatie
3. **Delivery Workers** - kanaalspecifieke workers die externe API-aanroepen afhandelen
4. **Template Service** - slaat notificatietemplates op en rendert ze met variabele substitutie

## Stap 3 - Deep dive: kerncomponenten

### Gebruikersvoorkeuren en afmelden
Sla voorkeuren op per gebruiker en per notificatietype:

```
user_preferences:
  user_id: "u123"
  channels:
    email: { enabled: true, quiet_hours: "22:00-08:00" }
    push: { enabled: true }
    sms: { enabled: false }
  categories:
    marketing: { enabled: false }
    security: { enabled: true, override_quiet_hours: true }
```

De notification service controleert voorkeuren **vóór** het enqueuen - verstuur nooit wat de gebruiker niet wil ontvangen.

### Priority Queues
Niet alle notificaties zijn gelijk. Gebruik prioriteitsniveaus:
- **Kritiek** - beveiligingsmeldingen, OTP-codes → onmiddellijk verwerkt
- **Hoog** - orderbevestigingen, betaalbewijzen → seconden
- **Gemiddeld** - sociale interacties, reacties → minuten acceptabel
- **Laag** - marketing, aanbevelingen → kan gebundeld worden

## Stap 4 - Betrouwbaarheid en foutafhandeling

### Deduplication
Voorkom dubbele notificaties met een idempotency key. Controleer vóór verwerking of het notification-ID bestaat in een deduplication store (Redis met TTL):

```
Key: "dedup:{notification_id}" → TTL: 24 hours
If key exists → skip (already sent)
If not → process and set key
```

### Retry met Exponential Backoff
Externe diensten falen. Implementeer retries met toenemende wachttijden:

```
Attempt 1: immediate
Attempt 2: wait 1 second
Attempt 3: wait 4 seconds
Attempt 4: wait 16 seconds
Attempt 5: move to dead-letter queue
```

### Rate Limiting per gebruiker
Bescherm gebruikers tegen notificatiemoeheid:
- Maximaal 3 push notifications per uur per gebruiker
- Maximaal 1 marketing-email per dag
- Beveiligingsnotificaties omzeilen rate limits

<FunFact>Facebook verstuurt dagelijks meer dan 10 miljard push notifications. Om dit te verwerken bouwden ze een eigen notificatiepipeline die miljoenen berichten per seconde kan verwerken, met aparte bezorgpaden voor iOS (APNs) en Android (FCM).</FunFact>

## Stap 5 - Integratie met externe diensten

| Kanaal | Provider | Protocol | Uitdagingen |
|--------|----------|----------|-------------|
| Push (iOS) | APNs | HTTP/2 | Tokenbeheer, limieten op silent push |
| Push (Android) | FCM | HTTP | Topic-gebaseerde broadcasting |
| Email | SendGrid/SES | SMTP/API | Afleverbaarheid, spamfilters |
| SMS | Twilio | REST API | Kosten per bericht, regionale compliance |
| In-App | WebSocket | WS | Verbindingsbeheer, offline buffering |

### Analytics pipeline
Volg de levenscyclus van elke notificatie:
- **Created** → **Queued** → **Sent** → **Delivered** → **Opened** → **Clicked**

Sla gebeurtenissen op in een time-series database of event stream voor dashboardrapportages: bezorgpercentages, openingspercentages en doorklikpercentages per kanaal en notificatietype.

## Stap 6 - Knelpunten en optimalisaties

| Knelpunt | Oplossing |
|----------|-----------|
| Rate limits van externe diensten | Wachtrij met gecontroleerde doorvoer per provider |
| Template rendering op schaal | Veelgebruikte templates pre-renderen en cachen |
| Voorkeurencontrole per bericht | Gebruikersvoorkeuren cachen in Redis |
| Verkeerspieken | Workers automatisch schalen op basis van wachtrijdiepte |
| Cross-regio bezorging | Workers dicht bij provider-endpoints deployen |
