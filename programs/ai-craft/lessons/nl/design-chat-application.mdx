## De interviewvraag

*"Ontwerp een chatapplicatie zoals WhatsApp of Slack die one-to-one messaging, groepschats en online presence indicators ondersteunt."*

Dit is een favoriet bij system design interviews omdat het real-time communicatie, connection management, storage op schaal en distributed system trade-offs test - allemaal in één probleem.

## Stap 1: Requirements verzamelen

**Functionele requirements:**
- One-to-one messaging met delivery en read receipts
- Groepschat (tot 500 leden)
- Online/offline presence indicators
- Berichtgeschiedenis en zoeken
- Push notifications voor offline gebruikers

**Niet-functionele requirements:**
- Lage latency: berichten binnen 200ms bezorgd
- Hoge beschikbaarheid: 99,99% uptime
- Berichtvolgorde gegarandeerd binnen een conversatie
- Ondersteuning voor 50 miljoen gelijktijdige verbindingen

<Callout>
Kwantificeer je niet-functionele requirements altijd met specifieke getallen. "Snelle bezorging" is vaag - "sub-200ms delivery" toont dat je het domein begrijpt.
</Callout>

## Stap 2: API-ontwerp

In tegenstelling tot REST-only systemen vereist chat **bidirectionele communicatie**. We hebben twee protocollen nodig:

**REST API** (voor niet-real-time operaties):
```
POST /api/users/login
GET  /api/conversations
GET  /api/conversations/:id/messages?before=timestamp&limit=50
POST /api/groups
```

**WebSocket** (voor real-time messaging):
```
CONNECT  ws://chat.example.com/ws?token=jwt
SEND     { type: "message", to: "user_456", content: "Hello!" }
RECEIVE  { type: "message", from: "user_123", content: "Hello!", ts: 1706000000 }
SEND     { type: "ack", messageId: "msg_789", status: "read" }
```

## Stap 3: WebSocket Connection Management

Elke verbonden gebruiker onderhoudt een persistente WebSocket-verbinding. Bij 50 miljoen gelijktijdige gebruikers is dit een aanzienlijke technische uitdaging.

**Connection servers** (ook wel gateway servers genoemd) houden WebSocket-verbindingen vast. Elke server kan ongeveer 50.000-100.000 gelijktijdige verbindingen aan, wat betekent dat we 500-1.000 connection servers nodig hebben.

Een **connection registry** (Redis cluster) koppelt elke gebruiker aan hun connection server:
```
user_123 → ws-server-042
user_456 → ws-server-187
```

Wanneer gebruiker A een bericht stuurt naar gebruiker B, is de flow:
1. Het bericht komt aan op A's connection server via WebSocket
2. De server zoekt B's connection server op in Redis
3. Het bericht wordt gerouteerd naar B's connection server via interne messaging
4. B's server pusht het bericht via B's WebSocket

## Stap 4: Berichtopslag

Chatapplicaties genereren enorme hoeveelheden data. WhatsApp verwerkt dagelijks meer dan 100 miljard berichten.

**Opslagstrategie:**
- **Recente berichten** - opgeslagen in een snelle datastore (Cassandra of HBase), gepartitioneerd op conversation ID
- **Berichtindex** - maakt zoeken mogelijk via Elasticsearch
- **Mediabestanden** - opgeslagen in object storage (S3) met CDN-levering

**Schema design (Cassandra):**
```
Table: messages
  partition_key: conversation_id
  clustering_key: message_timestamp (descending)
  columns: sender_id, content, type, status
```

Dit ontwerp zorgt ervoor dat alle berichten in een conversatie op dezelfde partition staan, waardoor "laad oudere berichten"-queries extreem efficiënt zijn - slechts een sequentiële lezing binnen één partition.

## Stap 5: Presence-systeem

Tonen wie online is lijkt eenvoudig, maar is verrassend lastig op schaal. Je kunt niet elke paar seconden de database raadplegen voor elke gebruiker in je contactenlijst.

**Heartbeat-aanpak:**
1. Verbonden clients sturen elke 30 seconden een heartbeat via WebSocket
2. De server werkt een Redis-sleutel bij met een TTL van 60 seconden: `presence:user_123 → online (TTL 60s)`
3. Als er geen heartbeat komt, verloopt de sleutel en verschijnt de gebruiker als offline

**Fan-out voor statusupdates:**
Wanneer een gebruiker online komt, moet hun status worden gepusht naar alle contacten die ook online zijn. Voor een gebruiker met 500 contacten betekent dit controleren welke contacten online zijn en presence updates alleen naar hen sturen - **lazy fan-out** vermijdt broadcasting naar offline gebruikers.

<FunFact>
WhatsApp's "last seen"-functie veroorzaakte wereldwijd onverwachte sociale drama's. In sommige landen namen relatieconflicten toe omdat partners elkaars exacte laatst-actieve timestamps konden zien. WhatsApp voegde uiteindelijk privacyinstellingen toe om deze informatie te verbergen.
</FunFact>

## Stap 6: Push Notifications

Wanneer een gebruiker offline is (geen actieve WebSocket), moeten berichten hen nog steeds bereiken via push notifications.

De berichtstroom verandert:
1. De connection registry toont dat gebruiker B offline is
2. Het bericht wordt opgeslagen in de database met status "undelivered"
3. Een notification service stuurt een push via APNs (iOS) of FCM (Android)
4. Wanneer gebruiker B opnieuw verbinding maakt, worden onbezorgde berichten opgehaald en via WebSocket verzonden
5. Delivery receipts worden teruggestuurd ter bevestiging

<ThinkAboutIt>
Groepschats met 500 leden vormen een fan-out uitdaging. Als een bericht wordt gestuurd naar een groep waar 400 leden offline zijn, betekent dat 400 push notifications. Hoe zou je deze batchen en rate-limiten om de notification services tijdens piekuren niet te overbelasten?
</ThinkAboutIt>

## Stap 7: Basisprincipes van E2E Encryption

Het kort noemen van E2E encryption in interviewgesprekken toont beveiligingsbewustzijn:

- Elke gebruiker genereert een **public/private key pair** op hun apparaat
- Berichten worden versleuteld met de public key van de ontvanger vóór verzending
- De server ziet alleen ciphertext - het kan de berichtinhoud niet lezen
- Groepschats gebruiken een **shared group key** die roteert wanneer leden toetreden of vertrekken

Dit betekent dat de server alleen versleutelde blobs opslaat, wat vanuit data compliance-perspectief eigenlijk eenvoudiger is.

## Stap 8: Schalen naar miljoenen

**Belangrijke scaling-strategieën:**
- **Connection servers** schalen horizontaal - voeg servers toe naarmate gebruikers groeien
- **Message routing** gebruikt Kafka of Redis Pub/Sub voor inter-server communicatie
- **Database sharding** op conversation ID verdeelt de opslagbelasting
- **Geografische spreiding** - deploy connection servers in meerdere regio's om latency te verlagen
- **Message queue buffering** - vangt pieken op tijdens drukke uren

## Interviewtips

1. **Begin met de WebSocket-beslissing** - leg uit waarom HTTP polling op deze schaal niet werkt
2. **Kwantificeer verbindingscapaciteit** - laat zien dat je serverlimieten begrijpt (~50-100K verbindingen per server)
3. **Behandel offline bezorging** - veel kandidaten vergeten dat gebruikers niet altijd verbonden zijn
4. **Noem berichtvolgorde** - timestamp-gebaseerde ordening met conflictoplossing

<ThinkAboutIt>
Slack en WhatsApp hanteren zeer verschillende benaderingen voor berichtgeschiedenis. WhatsApp slaat berichten op het apparaat op en synchroniseert geen volledige geschiedenis naar nieuwe apparaten. Slack slaat alles server-side op en biedt volledige zoekfunctionaliteit. Wat zijn de privacy-, kosten- en gebruikerservaring trade-offs van elke aanpak?
</ThinkAboutIt>

## Belangrijkste punten

- Chatsystemen vereisen WebSockets voor real-time bidirectionele communicatie
- Connection management op schaal heeft een registry nodig die gebruikers aan servers koppelt
- Cassandra's schrijfdoorvoer en partition-based queries passen bij chatberichtopslag
- Presence-systemen gebruiken heartbeats met TTL-gebaseerde expiry in Redis
- Push notifications verzorgen offline berichtbezorging
- Geografische spreiding en horizontaal schalen adresseren wereldwijde latency-eisen
