# Ontwerp een Rate Limiter - APIs beschermen op schaal

Rate limiting is een van de meest praktische system design onderwerpen. Elke productie-API - van GitHub tot Stripe - vertrouwt erop om misbruik te voorkomen, eerlijk gebruik te waarborgen en backend services te beschermen tegen overbelasting.

## Stap 1 - Requirements verzamelen

### Functionele Requirements
- Beperk requests op basis van configureerbare regels (per gebruiker, IP of API key)
- Geef duidelijke feedback wanneer een client wordt beperkt (HTTP 429)
- Ondersteun meerdere rate limiting regels tegelijkertijd

### Niet-functionele Requirements
- **Lage latency** - de limiter mag geen significante overhead toevoegen
- **Gedistribueerd** - werkt over meerdere servers
- **Fouttolerant** - als de limiter faalt, moet het verkeer doorstromen (fail-open)

<ThinkAboutIt>Moet een rate limiter fail-open (verkeer toestaan) of fail-closed (verkeer blokkeren) wanneer hij crasht? Wat zijn de afwegingen voor een e-commerce site versus een bank-API?</ThinkAboutIt>

## Stap 2 - Rate Limiting Algorithms

### Fixed Window Counter
Verdeel tijd in vaste vensters (bijv. intervallen van 60 seconden). Verhoog een teller per request. Reset bij venstergrenzen. **Probleem:** burstverkeer op de randen van vensters kan 2× de limiet toestaan.

### Sliding Window Log
Sla de timestamp van elke request op. Tel de entries binnen het verschuivende venster. Precies maar **geheugenintensief** - het opslaan van elke timestamp is duur op grote schaal.

### Sliding Window Counter
Een hybride: combineer de teller van het huidige venster met een gewogen deel van het vorige venster. Balanceert nauwkeurigheid en geheugenefficiëntie.

### Token Bucket Algorithm (meest populair)
Een bucket bevat tokens tot een maximale capaciteit. Tokens worden met een vast tempo toegevoegd. Elke request verbruikt één token. Als de bucket leeg is, wordt de request afgewezen.

```
Bucket: capacity=10, refill_rate=2/sec

Time 0s: [🪙🪙🪙🪙🪙🪙🪙🪙🪙🪙] → 10 tokens
Burst of 8 requests → [🪙🪙] → 2 tokens remain
Time 1s: refill +2 → [🪙🪙🪙🪙] → 4 tokens
```

### Algoritmevergelijking

| Algoritme | Geheugen | Nauwkeurigheid | Burstafhandeling |
|-----------|----------|----------------|------------------|
| Fixed Window | Laag | Laag (edge bursts) | Slecht |
| Sliding Log | Hoog | Exact | Goed |
| Sliding Counter | Gemiddeld | Benaderend | Goed |
| Token Bucket | Laag | Goed | Uitstekend |

## Stap 3 - Architectuur op hoog niveau

```
Client → API Gateway (Rate Limiter Middleware) → Backend Services
                    ↕
              Redis Cluster
         (counters + token buckets)
```

<Illustration src="/images/lessons/design-rate-limiter.svg" alt="Rate limiter architectuur met clients, API gateway met rate limiting middleware, Redis cluster voor gedistribueerde state en backend services" caption="De rate limiter bevindt zich in de API gateway en gebruikt Redis voor gedistribueerde telleropslag" />

### Waar de Rate Limiter plaatsen
- **Client-side** - eenvoudig te omzeilen, onbetrouwbaar
- **Server-side** - alleen per instantie, schaalt niet
- **API Gateway / Middleware** - de beste aanpak voor gedistribueerde systemen; gecentraliseerde handhaving over alle instanties

## Stap 4 - Deep Dive: Gedistribueerde Rate Limiting

### Redis + Lua Scripts
In een gedistribueerde setup moeten meerdere servers de rate limiting state delen. Redis is de standaardkeuze vanwege zijn atomic operations en sub-milliseconde latency.

Een Lua script garandeert atomiciteit - de controle en decrement gebeuren in één enkele Redis-operatie, waardoor race conditions worden voorkomen:

```
-- Token Bucket in Redis (pseudocode)
local tokens = redis.call('GET', key)
local last_refill = redis.call('GET', key..':time')
-- Calculate tokens to add based on elapsed time
-- If tokens > 0: decrement and ALLOW
-- Else: DENY
```

### Response Headers
Goed ontworpen rate limiters communiceren hun status terug naar clients:
- `X-RateLimit-Limit: 100` - maximaal aantal requests per venster
- `X-RateLimit-Remaining: 42` - resterende requests
- `X-RateLimit-Reset: 1672531200` - UTC epoch wanneer het venster reset
- HTTP `429 Too Many Requests` met een `Retry-After` header

## Stap 5 - Omgaan met randgevallen

### Race Conditions
Zonder atomic operations kunnen gelijktijdige requests limieten omzeilen. Oplossingen: Redis Lua scripts of Redis `MULTI/EXEC` transactions.

### Rate Limiting Dimensies
Pas limieten op meerdere niveaus tegelijkertijd toe:
- **Per gebruiker** - 100 requests/minuut voor geauthenticeerde gebruikers
- **Per IP** - 50 requests/minuut voor anoniem verkeer
- **Per API key** - gelaagde plannen (free: 10/min, pro: 1000/min)

### Distributed Clock Skew
Servers kunnen licht verschillende klokken hebben. Gebruik de Redis-servertijd als enige bron van waarheid in plaats van te vertrouwen op timestamps van applicatieservers.

<FunFact>GitHub's REST API staat 5.000 requests per uur toe voor geauthenticeerde gebruikers, maar slechts 60 per uur voor niet-geauthenticeerde requests. Stripe gebruikt een sliding window aanpak met aparte limieten voor live en test mode API keys.</FunFact>

## Stap 6 - Knelpunten en optimalisaties

| Knelpunt | Oplossing |
|----------|-----------|
| Redis single point of failure | Redis Cluster met replicatie |
| Netwerklatency naar Redis | Lokale in-memory cache met periodieke synchronisatie |
| Hot keys (populaire API keys) | Verdeel over meerdere Redis-nodes |
| Overhead bij regelevaluatie | Pre-compileer regels, cache configuraties |
