# Ontwerp een Web Crawler — Hoe Google het internet indexeert

Een web crawler doorzoekt systematisch het internet om webpagina's te downloaden en te indexeren. Het is de basis van elke zoekmachine, en het ontwerpen ervan leert je over distributed systems, graph traversal en het omgaan met de rommeligheid van het echte web.

## Stap 1 — Requirements verzamelen

### Functionele requirements
- Webpagina's crawlen vanaf een set seed URLs
- Hyperlinks extraheren en volgen om nieuwe pagina's te ontdekken
- Gedownloade content opslaan voor indexering en verwerking
- robots.txt en crawl politeness-regels respecteren

### Niet-functionele requirements
- **Scale** — 1 miljard pagina's per maand crawlen
- **Politeness** — nooit een enkele webserver overbelasten
- **Versheid** — belangrijke pagina's regelmatig opnieuw crawlen
- **Robuustheid** — omgaan met malformed HTML, oneindige lussen en spider traps

## Stap 2 — High-level architectuur

```
Seed URLs → URL Frontier (priority queue)
                  ↓
            URL Scheduler (politeness + priority)
                  ↓
         Fetcher (HTTP client pool)
                  ↓
         HTML Parser → Extract URLs → URL Filter
              ↓                           ↓
       Content Store              URL Dedup (Bloom Filter)
       (S3 / HDFS)                      ↓
                                 URL Frontier (loop)
```

## Stap 3 — De crawl-loop

### BFS vs DFS Crawling
- **BFS (Breadth-First)** — de voorkeur voor web crawling. Ontdekt snel een breed scala aan domeinen en voorkomt dat je vast komt te zitten diep in één site
- **DFS (Depth-First)** — risico om oneindig diep te gaan in dynamisch gegenereerde pagina's voordat andere domeinen ontdekt worden

### URL Frontier-ontwerp
De frontier is geen simpele wachtrij — het is een geavanceerd prioriteitssysteem met twee subcomponenten:

1. **Priority Queue (Front)** — rangschikt URLs op belangrijkheid. Factoren zijn onder andere PageRank, updatefrequentie en domeinautoriteit
2. **Politeness Queue (Back)** — zorgt ervoor dat elk domein een eigen FIFO-wachtrij heeft. Een scheduler kiest uit verschillende domeinwachtrijen om te voorkomen dat één server overbelast wordt

```
Front (Priority):       Back (Politeness):
┌──────────────┐       ┌─── bbc.co.uk queue ───┐
│ High Priority │ ───→  │ url1 → url4 → url7    │
│ Med Priority  │ ───→  ├─── github.com queue ──┤
│ Low Priority  │ ───→  │ url2 → url5 → url8    │
└──────────────┘       ├─── wikipedia.org queue ┤
                        │ url3 → url6 → url9    │
                        └───────────────────────┘
```

## Stap 4 — Politeness en Robots.txt

### Robots.txt respecteren
Haal vóór het crawlen van een domein het `robots.txt`-bestand op en cache het:

```
# Example robots.txt
User-agent: *
Disallow: /admin/
Disallow: /private/
Crawl-delay: 2
```

Cache robots.txt per domein met een TTL (24 uur). Respecteer altijd `Disallow`-regels en `Crawl-delay`-richtlijnen.

### Crawl rate limiting
Zelfs zonder expliciete `Crawl-delay`, handhaaf politeness:
- Maximaal 1 request per seconde per domein (configureerbaar)
- Gebruik aparte download-wachtrijen per domein
- Monitor responstijden — als een server vertraagt, verminder dan automatisch het tempo

<FunFact>Google's web crawler (Googlebot) crawlt honderden miljarden pagina's. Om beleefd te blijven, monitort het de responstijden van servers en verlaagt het crawltempo als een site langzamer begint te reageren — waardoor de website effectief kan "terugduwen" tegen de crawler.</FunFact>

## Stap 5 — Deduplicatie

### URL-deduplicatie (Bloom Filter)
Met miljarden URLs vereist het controleren of een URL eerder gezien is een efficiënte datastructuur. Een Bloom filter biedt O(1) lookups met een kleine geheugenvoetafdruk:

- **Ruimte:** ~1 GB voor 1 miljard URLs bij 1% false positive rate
- **Afweging:** kan af en toe een pagina opnieuw crawlen (false positive) maar mist nooit een nieuwe URL (geen false negatives)

Normaliseer URLs vóór controle: verwijder fragments (#), sorteer queryparameters, maak de hostname lowercase en los relatieve paden op.

### Content-deduplicatie (SimHash)
Verschillende URLs kunnen identieke content serveren (mirrors, gekopieerde artikelen). SimHash genereert een fingerprint van de pagina-inhoud — pagina's met vergelijkbare fingerprints (kleine Hamming-afstand) zijn waarschijnlijk duplicaten:

```
Page A content → SimHash → 10110010...
Page B content → SimHash → 10110011...
Hamming distance = 1 → likely duplicate, skip storing
```

## Stap 6 — Distributed crawling

### Horizontaal schalen
Eén machine kan geen miljard pagina's per maand crawlen. Verdeel het werk:

- **Consistent hashing** op domeinnamen wijst elk domein toe aan een specifiek crawler-node. Dit garandeert politeness — slechts één node crawlt een bepaald domein
- De **URL frontier** wordt over nodes gepartitioneerd op basis van de hash ring
- **Coördinatie** via een lichtgewicht service die node-toewijzingen beheert en herbalanceert wanneer nodes toetreden of vertrekken

### DNS Resolution Caching
DNS lookups voegen 10-100ms toe per request. Op grote schaal is dit een groot knelpunt:
- Onderhoud een lokale DNS-cache per crawler-node
- Voer batch DNS pre-resolution uit voor URLs in de wachtrij
- Gebruik een dedicated DNS resolver om publieke DNS-servers niet te overbelasten

### Spider Trap-detectie
Kwaadaardige of slecht gebouwde sites kunnen oneindige URL-ruimtes creëren:
- `/page/1/page/1/page/1/...` — herhalende padpatronen
- Kalenderpagina's die URLs genereren voor elke toekomstige datum
- Session IDs in URLs die unieke URLs creëren voor identieke content

**Verdedigingen:** maximale URL-lengte, maximale crawldiepte per domein, patroondetectie voor repetitieve URL-structuren en domein-niveau paginalimits.

<ThinkAboutIt>Als je een zoekindex vers moet houden, hoe zou je dan beslissen welke pagina's je het eerst opnieuw crawlt? Denk aan een nieuwssite die elk uur bijwerkt versus een Wikipedia-artikel dat maandelijks verandert. Welke signalen zou je gebruiken om de re-crawl planning te prioriteren?</ThinkAboutIt>

## Stap 7 — Knelpunten en optimalisaties

| Knelpunt | Oplossing |
|-----------|----------|
| DNS-resolutielatentie | Lokale DNS-cache, batch pre-resolution |
| Langzame/niet-reagerende servers | Timeouts (30s), async I/O, overslaan na retries |
| Opslag voor gecrawlde content | Pagina's comprimeren, alleen gewijzigde content opslaan |
| URL frontier-geheugen | Disk-backed priority queue (RocksDB) |
| Dubbele content | SimHash fingerprinting, canonical URL-detectie |

<ThinkAboutIt>Zoekmachines moeten een balans vinden tussen crawlversheid (pagina's vaak opnieuw crawlen) en ontdekking (nieuwe pagina's vinden). Als je crawler een vast budget heeft van 1 miljoen pagina's per dag, hoe zou je dat budget verdelen tussen het opnieuw crawlen van bestaande pagina's en het ontdekken van nieuwe?</ThinkAboutIt>
