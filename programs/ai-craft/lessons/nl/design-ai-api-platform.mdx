## De interviewvraag

*"Ontwerp een AI API platform dat meerdere machine learning modellen serveert aan developers via een REST API - vergelijkbaar met hoe OpenAI of Hugging Face modellen op schaal aanbieden."*

Deze vraag wordt steeds populairder in interviews omdat het **API platform design** combineert met **GPU infrastructure-uitdagingen** die uniek zijn voor AI workloads.

## Stap 1: Requirements verzamelen

**Functionele requirements:**
- Meerdere AI-modellen (tekstgeneratie, beeldclassificatie, embeddings) serveren via REST API
- API key authentication met gebruiksquota per key
- Gebruikstracking en facturering per API-aanroep
- Ondersteuning voor meerdere modelversies met A/B testing
- Streaming responses voor langlopende generatietaken

**Niet-functionele requirements:**
- Latency: p50 onder 500ms voor embedding-modellen, p50 onder 2s voor generatieve modellen
- Throughput: 10.000 requests per seconde over alle modellen
- Beschikbaarheid: 99,9% uptime met graceful degradation
- Kostenefficiëntie: GPUs zijn duur - maximaliseer de benutting

<Callout>
AI-platforms hebben een uniek variabele latency. Een text completion request kan 200ms of 30 seconden duren, afhankelijk van de output-lengte. Bespreek altijd hoe jouw ontwerp met deze variatie omgaat.
</Callout>

## Stap 2: API-ontwerp

```
POST /v1/completions
Headers: Authorization: Bearer sk-abc123
Body: { "model": "gpt-4", "prompt": "Explain caching", "max_tokens": 200 }

POST /v1/embeddings
Headers: Authorization: Bearer sk-abc123
Body: { "model": "text-embedding-3", "input": "Hello world" }

GET  /v1/models                    - beschikbare modellen ophalen
GET  /v1/usage?start=2024-01-01    - gebruiks- en factureringsgegevens

SSE: POST /v1/completions (stream: true)
     → returns Server-Sent Events with incremental tokens
```

Streaming via **Server-Sent Events (SSE)** is cruciaal voor generatieve modellen - gebruikers zien tokens geleidelijk verschijnen in plaats van te wachten op het volledige antwoord.

## Stap 3: API Gateway en authentication

De API gateway is de voordeur van het platform:

- **Authentication** - API keys valideren tegen een key store (Redis voor snelheid, database voor persistentie)
- **Rate limiting** - limieten per key afdwingen (bijv. 60 requests/minuut voor de gratis tier)
- **Request routing** - requests doorsturen naar het juiste model-serving cluster
- **Request validation** - payload-schema, modelbeschikbaarheid en tokenlimieten controleren

**Rate limiting implementatie:**
Gebruik een **sliding window counter** in Redis:
```
Key:   rate:{api_key}:{window_minute}
Value: request count
TTL:   120 seconds
```

Elke request verhoogt de teller. Als de limiet wordt overschreden, retourneer HTTP 429.

## Stap 4: Model serving en GPU-toewijzing

Hier verschillen AI-platforms fundamenteel van traditionele webservices. GPUs zijn schaars, duur en vereisen gespecialiseerde orchestratie.

**Model serving laag:**
- Modellen worden in GPU-geheugen geladen met frameworks zoals **Triton Inference Server** of **vLLM**
- Elke modelversie draait op een dedicated pool van GPU-instances
- Large language models (LLMs) kunnen **model parallelism** over meerdere GPUs vereisen

**Request batching:**
Individuele inference requests zijn inefficiënt op GPUs. **Dynamic batching** groepeert meerdere requests en verwerkt ze samen:
- Verzamel requests gedurende maximaal 10ms of tot de batch 32 requests bereikt
- Verwerk de gehele batch in één GPU forward pass
- Retourneer individuele responses aan elke aanroeper

Dit kan de throughput met 5-10x verbeteren vergeleken met individuele verwerking.

<FunFact>
Een enkele NVIDIA H100 GPU kost ongeveer £25.000 en verbruikt 700 watt. Een cluster van 1.000 GPUs draaien voor model serving kost meer dan £2 miljoen per maand alleen al aan elektriciteit en hardware - daarom zijn efficiënt batching en benutting enorm belangrijk.
</FunFact>

## Stap 5: A/B Testing tussen modelversies

Bij het uitrollen van een nieuwe modelversie is een gecontroleerde rollout noodzakelijk:

- **Traffic splitting** - routeer 5% van de requests naar het nieuwe model, 95% naar het huidige
- **Shadow mode** - draai het nieuwe model op productieverkeer zonder de resultaten aan gebruikers te tonen
- **Metriekvergelijking** - latency, kwaliteitsscores, foutpercentages, kosten per request
- **Geleidelijke rollout** - verhoog het verkeer naar de nieuwe versie naarmate het vertrouwen groeit

De routinglogica zit in de API gateway:
```
model: "gpt-4"  →  95% → gpt-4-v2.1 (current)
                →   5% → gpt-4-v2.2 (candidate)
```

<ThinkAboutIt>
Evalueren of een nieuwe LLM-versie "beter" is, is verrassend moeilijk - in tegenstelling tot classificatiemodellen is er geen eenvoudige accuracy-metriek. Hoe zou je een geautomatiseerd kwaliteitsevaluatiesysteem ontwerpen voor generatieve AI-modellen?
</ThinkAboutIt>

## Stap 6: Gebruikstracking en facturering

AI APIs factureren doorgaans per token (input + output tokens voor LLMs, per request voor andere modellen).

**Event-driven factureringspipeline:**
1. Elke API-response bevat een gebruiksheader: `x-usage-tokens: 847`
2. Een factureringsevent wordt gepubliceerd naar Kafka: `{ api_key, model, input_tokens, output_tokens, timestamp }`
3. Een factureringsaggregatieservice consumeert events en werkt lopende totalen bij
4. Gebruiksdashboards bevragen geaggregeerde data uit een time-series database

**Waarom Kafka?** Facturering moet betrouwbaar zijn, maar hoeft niet synchroon te zijn. Als de factureringsservice tijdelijk niet beschikbaar is, worden events in Kafka gebufferd en verwerkt bij herstel - er gaat geen gebruiksdata verloren.

## Stap 7: Monitoring en observability

AI-platforms vereisen monitoring die verder gaat dan standaard webservices:

| Metriek | Waarom het belangrijk is |
|---------|--------------------------|
| Inference latency (p50, p95, p99) | Gebruikerservaring en SLA-naleving |
| GPU-benutting | Kostenefficiëntie - inactieve GPUs verspillen geld |
| Tokens per seconde | Throughput-capaciteit en scaling-signalen |
| Foutpercentage per model | Modelspecifieke storingen detecteren |
| Queue depth | Scaling-trigger - diepe queues betekenen dat vraag de capaciteit overschrijdt |
| Kwaliteitsscores output | Modeldegradatie detecteren (via geautomatiseerde evaluatie) |

**Alertdrempels:** GPU-benutting onder 40% triggert scale-down; queue depth boven 100 triggert scale-up.

## Stap 8: Auto-Scaling

AI workloads hebben een sterk variabele vraag. Auto-scaling moet responsiviteit en kosten in balans houden:

- **Horizontal scaling** - meer GPU-instances toevoegen wanneer queue depth of latency drempels overschrijdt
- **Predictive scaling** - historische verkeerspatronen gebruiken om capaciteit vooraf te provisioneren vóór verwachte pieken
- **Scale-to-zero** - voor zelden gebruikte modellen alle instances uitschakelen en on-demand cold-starten
- **Warm pools** - een klein aantal instances met voorgeladen modellen behouden om cold-start latency te vermijden

De scale-to-zero strategie is bijzonder belangrijk voor platforms die honderden modellen hosten - je kunt niet elk model altijd in GPU-geheugen geladen houden.

<Quiz question="Een model dat 2 requests per uur ontvangt, draait momenteel op één always-on GPU-instance die £3/uur kost. Cold-starten vanuit nul duurt 45 seconden. <ThinkAboutIt>
Het platform van OpenAI serveert modellen variërend van snelle embedding-modellen tot trage redeneermodellen. Elk heeft andere latency-kenmerken, GPU-vereisten en prijzen. Hoe zou je de routinglaag ontwerpen om een eindig GPU-cluster efficiënt te delen tussen modellen met zulke verschillende profielen?
</ThinkAboutIt>

<FunFact>
OpenAI verwerkt naar verluidt meer dan 100 miljoen API-requests per dag. Bij een geschatte gemiddelde van 500 tokens per request zijn dat 50 miljard tokens per dag - waarvoor duizenden GPUs continu moeten draaien.
</FunFact>

## Belangrijkste inzichten

- AI API platforms combineren traditioneel API-ontwerp met GPU-specifieke infrastructuuruitdagingen
- Request batching verbetert GPU-throughput en kostenefficiëntie drastisch
- Rate limiting beschermt zowel het platform als klanten tegen ongecontroleerde kosten
- Asynchrone facturering via Kafka zorgt voor betrouwbare gebruikstracking zonder requests te blokkeren
- Auto-scaling moet responsiviteit en kosten in balans houden - scale-to-zero voor modellen met weinig verkeer
