# Ontwerp een Distributed Cache - Redis Architecture vanaf nul

Caching is de meest effectieve techniek om systeemprestaties te verbeteren. Een goed ontworpen distributed cache kan de databasebelasting met meer dan 90% verlagen en responstijden terugbrengen van honderden milliseconden naar minder dan een milliseconde.

## Stap 1 - Requirements verzamelen

### Functionele requirements
- Key-value paren opslaan met optionele TTL (time-to-live)
- Veelgebruikte operaties ondersteunen: GET, SET, DELETE en bulkoperaties
- Data verdelen over meerdere nodes voor horizontale schaalbaarheid

### Niet-functionele requirements
- **Lage latency** - sub-milliseconde reads voor cache hits
- **Hoge throughput** - 100K+ operaties per seconde per node
- **Hoge beschikbaarheid** - cache-storingen mogen het systeem niet platleggen
- **Schaalbaarheid** - nodes toevoegen zonder downtime

### Wanneer cachen
- **Read-heavy workloads** - wanneer reads de writes ver overtreffen
- **Dure berekeningen** - ML model inference resultaten, complexe aggregaties
- **Veelgebruikte data** - user sessions, productcatalogi, configuratie

## Stap 2 - Caching-strategieën

### Cache-Aside (Lazy Loading)
Het meest gebruikte patroon. De applicatie controleert eerst de cache; bij een miss leest het de database en vult vervolgens de cache:

```
GET user:123 → Cache HIT → return cached data
GET user:456 → Cache MISS → query DB → SET cache → return data
```

**Voordelen:** alleen wat daadwerkelijk gevraagd wordt, wordt gecachet. **Nadelen:** cache misses veroorzaken extra latency; data kan verouderd raken.

### Read-Through
De cache handelt zelf database-reads af bij een miss. De applicatie communiceert alleen met de cache-laag, die de databron abstraheert.

### Write-Through
Elke write gaat synchroon naar zowel de cache als de database. De cache is altijd consistent, maar dit verhoogt de write latency.

### Write-Behind (Write-Back)
Writes gaan direct naar de cache; de cache flusht asynchroon in batches naar de database. Lagere write latency, maar risico op dataverlies als de cache node crasht vóór het flushen.

| Strategie | Consistentie | Read latency | Write latency | Toepassing |
|-----------|-------------|-------------|---------------|------------|
| Cache-Aside | Eventual | Miss penalty | N/A | Algemeen gebruik |
| Read-Through | Eventual | Miss penalty | N/A | Vereenvoudigde reads |
| Write-Through | Strong | Snel | Langzamer | Consistentie-kritisch |
| Write-Behind | Eventual | Snel | Snel | Write-heavy workloads |

## Stap 3 - Eviction Policies

Wanneer de cache vol is, welke entries verwijderen we?

- **LRU (Least Recently Used)** - verwijdert de entry die het langst niet is benaderd. Beste algemene policy
- **LFU (Least Frequently Used)** - verwijdert de entry die het minst vaak is benaderd. Beter voor scheve toegangspatronen
- **TTL-based** - entries verlopen na een ingestelde tijd, ongeacht toegang. Essentieel voor dataversheid
- **Random** - verrassend effectief en zeer eenvoudig te implementeren

<FunFact>Redis gebruikt een benaderd LRU-algoritme dat een klein aantal keys bemonstert (standaard: 5) en de minst recent gebruikte daarvan verwijdert. Dit is bijna even nauwkeurig als echte LRU, maar vereist nul extra geheugenoverhead voor het bijhouden van de toegangsvolgorde.</FunFact>

## Stap 4 - Consistent Hashing voor distributie

### Het probleem met eenvoudige hashing
Met `hash(key) % N` nodes zorgt het toevoegen of verwijderen van een node ervoor dat bijna alle keys opnieuw worden toegewezen, wat een massale cache miss storm veroorzaakt.

### Consistent Hashing met virtual nodes
Consistent hashing plaatst zowel keys als nodes op een ring. Elke key wordt toegewezen aan de eerstvolgende node met de klok mee op de ring:

```
Hash Ring (0 to 2^32):
     Node A (3 virtual nodes: A1, A2, A3)
     Node B (3 virtual nodes: B1, B2, B3)
     Node C (3 virtual nodes: C1, C2, C3)

Key "user:123" → hash → lands between B2 and A1 → assigned to Node A
```

**Virtual nodes** lossen het probleem van ongelijke verdeling op. Elke fysieke node krijgt meerdere posities op de ring, waardoor een evenwichtige belasting wordt gegarandeerd, zelfs met weinig fysieke nodes.

Wanneer een node wordt toegevoegd of verwijderd, worden alleen de aangrenzende keys op de ring opnieuw toegewezen - ongeveer `K/N` keys (waarbij K het totaal aantal keys is en N het aantal nodes), niet allemaal.

## Stap 5 - Cache Stampede preventie

Een **cache stampede** treedt op wanneer een populaire cached entry verloopt en honderden gelijktijdige requests tegelijk de cache missen en de database raken.

### Oplossingen

**Mutex/Locking:** Bij een cache miss wordt een distributed lock verkregen. Slechts één request haalt data op uit de database; de rest wacht tot de cache opnieuw is gevuld.

**Probabilistic Early Expiry:** Elke request heeft een kleine willekeurige kans om de cache te verversen vóórdat de TTL daadwerkelijk verloopt. Dit spreidt de verversingsbelasting over de tijd in plaats van een thundering herd te creëren bij het verlopen.

**Background Refresh:** Een achtergrondproces ververst populaire cache-entries voordat ze verlopen. De cache is nooit leeg voor hot keys.

<ThinkAboutIt>Stel je een flash sale voor waarbij een productpagina 100.000 requests per seconde ontvangt. De cache-entry met productdetails verloopt. Zonder stampede-preventie raken alle 100K requests tegelijk de database. Welke preventiestrategie zou je kiezen, en waarom?</ThinkAboutIt>

## Stap 6 - Redis Cluster Architecture

Redis Cluster is de meest gebruikte distributed cache:

- **16.384 hash slots** verdeeld over master nodes
- Elke key wordt gekoppeld aan een slot via `CRC16(key) % 16384`
- **Replicatie:** elke master heeft een of meer replicas voor failover
- **Automatische failover:** als een master uitvalt, wordt zijn replica binnen seconden gepromoveerd

```
Slots 0-5460:     Master A ←→ Replica A'
Slots 5461-10922: Master B ←→ Replica B'
Slots 10923-16383: Master C ←→ Replica C'
```

### Belangrijke metrics monitoren
- **Hit ratio** - streef naar 95%+; onder 80% wijst op een cachingprobleem
- **Geheugengebruik** - blijf onder 80% om evictions te voorkomen
- **Eviction rate** - hoge evictions betekenen dat de cache te klein is
- **Latency percentielen** - p99 moet onder 1ms liggen voor cache hits

## Stap 7 - Caching-patronen in de praktijk

| Patroon | Implementatie | TTL |
|---------|---------------|-----|
| Session store | `SET session:{id} {data}` | 30 minuten |
| API response cache | `SET api:/users/123 {json}` | 5 minuten |
| Database query cache | `SET query:{hash} {result}` | 1 minuut |
| Rate limiter teller | `INCR ratelimit:{user}:{window}` | Vensterduur |
| Feature flags | `HSET features {flag} {value}` | 10 seconden |

### Cache Invalidation
Het moeilijkste probleem in de informatica (naast het benoemen van dingen):
- **TTL-based** - eenvoudig, tolereert veroudering gedurende de TTL-duur
- **Event-driven** - publiceer invalidation events wanneer data verandert (via Kafka of Redis Pub/Sub)
- **Version-based** - voeg een versie toe aan de cache key; verhoog de versie bij writes om oude entries effectief te invalideren

<Quiz question="Welke cache invalidation aanpak biedt de sterkste consistentiegarantie?" options="TTL-based expiry met een venster van 5 minuten|Event-driven invalidation getriggerd door database writes|Random eviction|Background refresh elke 10 seconden" answer="1" explanation="Event-driven invalidation reageert op daadwerkelijke datawijzigingen en invalideert de cache onmiddellijk wanneer de brondata wordt bijgewerkt. TTL-based aanpakken hebben altijd een verouderingsvenster, terwijl background refresh gaten heeft tussen de verversingsrondes." />
