## Le sujet d'entretien

*"Concevez une application de chat comme WhatsApp ou Slack qui supporte la messagerie individuelle, les discussions de groupe et les indicateurs de présence en ligne."*

C'est un grand classique des entretiens de conception système car il teste la communication en temps réel, la gestion des connexions, le stockage à grande échelle et les compromis des systèmes distribués - le tout en un seul problème.

## Étape 1 : Recueil des exigences

**Exigences fonctionnelles :**
- Messagerie individuelle avec accusés de réception et de lecture
- Chat de groupe (jusqu'à 500 membres)
- Indicateurs de présence online/offline
- Historique des messages et recherche
- Push notifications pour les utilisateurs hors ligne

**Exigences non fonctionnelles :**
- Faible latence : messages livrés en moins de 200 ms
- Haute disponibilité : 99,99 % d'uptime
- Ordre des messages garanti au sein d'une conversation
- Support de 50 millions de connexions simultanées

<Callout>
Quantifiez toujours vos exigences non fonctionnelles avec des chiffres précis. « Livraison rapide » est vague - « livraison en moins de 200 ms » montre que vous maîtrisez le domaine.
</Callout>

## Étape 2 : Conception de l'API

Contrairement aux systèmes exclusivement REST, le chat nécessite une **communication bidirectionnelle**. Deux protocoles sont nécessaires :

**REST API** (pour les opérations non temps réel) :
```
POST /api/users/login
GET  /api/conversations
GET  /api/conversations/:id/messages?before=timestamp&limit=50
POST /api/groups
```

**WebSocket** (pour la messagerie en temps réel) :
```
CONNECT  ws://chat.example.com/ws?token=jwt
SEND     { type: "message", to: "user_456", content: "Hello!" }
RECEIVE  { type: "message", from: "user_123", content: "Hello!", ts: 1706000000 }
SEND     { type: "ack", messageId: "msg_789", status: "read" }
```

## Étape 3 : Gestion des connexions WebSocket

Chaque utilisateur connecté maintient une connexion WebSocket persistante. Avec 50 millions d'utilisateurs simultanés, c'est un défi d'ingénierie considérable.

Les **connection servers** (parfois appelés gateway servers) maintiennent les connexions WebSocket. Chaque serveur peut gérer environ 50 000 à 100 000 connexions simultanées, ce qui nécessite 500 à 1 000 connection servers.

Un **connection registry** (cluster Redis) associe chaque utilisateur à son connection server :
```
user_123 → ws-server-042
user_456 → ws-server-187
```

Quand l'utilisateur A envoie un message à l'utilisateur B, le flux est :
1. Le message arrive sur le connection server de A via WebSocket
2. Le serveur cherche le connection server de B dans Redis
3. Le message est routé vers le connection server de B via le messaging interne
4. Le serveur de B pousse le message sur le WebSocket de B

## Étape 4 : Stockage des messages

Les applications de chat génèrent des volumes de données considérables. WhatsApp traite plus de 100 milliards de messages par jour.

**Stratégie de stockage :**
- **Messages récents** - conservés dans un datastore rapide (Cassandra ou HBase) partitionné par conversation ID
- **Index des messages** - permet la recherche via Elasticsearch
- **Fichiers média** - stockés dans un object storage (S3) avec livraison CDN

**Schema design (Cassandra) :**
```
Table: messages
  partition_key: conversation_id
  clustering_key: message_timestamp (descending)
  columns: sender_id, content, type, status
```

Cette conception garantit que tous les messages d'une conversation sont co-localisés sur la même partition, rendant les requêtes « charger les anciens messages » extrêmement efficaces - une simple lecture séquentielle dans une partition.

## Étape 5 : Système de présence

Afficher qui est en ligne semble simple mais s'avère étonnamment complexe à grande échelle. Impossible d'interroger la base de données pour chaque utilisateur de votre liste de contacts toutes les quelques secondes.

**Approche par heartbeat :**
1. Les clients connectés envoient un heartbeat toutes les 30 secondes via WebSocket
2. Le serveur met à jour une clé Redis avec un TTL de 60 secondes : `presence:user_123 → online (TTL 60s)`
3. Sans heartbeat, la clé expire et l'utilisateur apparaît hors ligne

**Fan-out des mises à jour de statut :**
Quand un utilisateur se connecte, son statut doit être poussé à tous ses contacts également en ligne. Pour un utilisateur avec 500 contacts, il faut vérifier lesquels sont en ligne et envoyer les mises à jour de présence uniquement à ceux-ci - le **lazy fan-out** évite de diffuser vers les utilisateurs hors ligne.

<FunFact>
La fonctionnalité « last seen » de WhatsApp a provoqué des drames sociaux inattendus dans le monde entier. Dans certains pays, les disputes de couple ont augmenté car les partenaires pouvaient voir les horodatages exacts de dernière activité. WhatsApp a finalement ajouté des contrôles de confidentialité pour masquer cette information.
</FunFact>

## Étape 6 : Push Notifications

Quand un utilisateur est hors ligne (pas de WebSocket actif), les messages doivent quand même lui parvenir via push notifications.

Le flux de messages change :
1. Le connection registry indique que l'utilisateur B est hors ligne
2. Le message est stocké en base avec le statut « undelivered »
3. Un service de notification envoie un push via APNs (iOS) ou FCM (Android)
4. Quand l'utilisateur B se reconnecte, les messages non livrés sont récupérés et envoyés via WebSocket
5. Des accusés de réception confirment la bonne livraison

<ThinkAboutIt>
Les chats de groupe avec 500 membres posent un défi de fan-out. Si un message est envoyé dans un groupe où 400 membres sont hors ligne, cela fait 400 push notifications. Comment les regrouper et les limiter en débit pour ne pas surcharger les services de notification aux heures de pointe ?
</ThinkAboutIt>

## Étape 7 : Bases du E2E Encryption

Mentionner brièvement le E2E encryption en entretien démontre votre sensibilité à la sécurité :

- Chaque utilisateur génère une **paire de clés publique/privée** sur son appareil
- Les messages sont chiffrés avec la clé publique du destinataire avant de quitter l'appareil de l'expéditeur
- Le serveur ne voit que du ciphertext - il ne peut pas lire le contenu
- Les chats de groupe utilisent une **clé de groupe partagée** qui est renouvelée quand des membres rejoignent ou quittent le groupe

Le stockage du serveur ne contient donc que des blobs chiffrés, ce qui simplifie en réalité la conformité des données.

## Étape 8 : Passer à l'échelle de millions

**Stratégies de scaling clés :**
- Les **connection servers** scalent horizontalement - ajoutez des serveurs au fur et à mesure
- Le **message routing** utilise Kafka ou Redis Pub/Sub pour la communication inter-serveurs
- Le **database sharding** par conversation ID distribue la charge de stockage
- La **distribution géographique** - déployez des connection servers dans plusieurs régions pour réduire la latence
- Le **message queue buffering** - absorbe les pics lors des heures de pointe

## Conseils pour l'entretien

1. **Commencez par la décision WebSocket** - expliquez pourquoi le HTTP polling ne fonctionne pas à cette échelle
2. **Quantifiez la capacité de connexion** - montrez que vous comprenez les limites serveur (~50-100K connexions par serveur)
3. **Abordez la livraison hors ligne** - beaucoup de candidats oublient que les utilisateurs ne sont pas toujours connectés
4. **Mentionnez l'ordre des messages** - ordonnancement par timestamp avec résolution de conflits

<ThinkAboutIt>
Slack et WhatsApp adoptent des approches très différentes pour l'historique des messages. WhatsApp stocke les messages sur l'appareil et ne synchronise pas l'historique complet sur les nouveaux appareils. Slack stocke tout côté serveur et offre une recherche complète. Quels sont les compromis en termes de confidentialité, de coût et d'expérience utilisateur de chaque approche ?
</ThinkAboutIt>

## Points clés à retenir

- Les systèmes de chat nécessitent des WebSockets pour la communication bidirectionnelle en temps réel
- La gestion des connexions à grande échelle requiert un registry associant utilisateurs et serveurs
- Le débit d'écriture et les requêtes par partition de Cassandra conviennent au stockage de messages
- Les systèmes de présence utilisent des heartbeats avec expiration par TTL dans Redis
- Les push notifications gèrent la livraison des messages hors ligne
- La distribution géographique et le scaling horizontal répondent aux exigences de latence globale
