## Le sujet d'entretien

*"Concevez un service de raccourcissement d'URL comme bit.ly qui prend de longues URLs et produit des codes courts et uniques."*

C'est la question de system design la plus fréquente en premier entretien, car elle paraît simple mais couvre des concepts fondamentaux : API design, stratégies d'encoding, choix de database, caching et scaling. Travaillons dessus avec le framework structuré de la Leçon 1.

## Étape 1 : Recueil des exigences (5 minutes)

**Exigences fonctionnelles :**
- À partir d'une longue URL, générer une URL courte et unique
- La redirection via l'URL courte renvoie l'URL longue originale
- Les utilisateurs peuvent définir des codes courts personnalisés (optionnel)
- Les liens expirent après une période configurable (par défaut : 5 ans)
- Analytics de base : nombre de clics, referrer, géographie

**Exigences non fonctionnelles :**
- 100 millions d'URLs générées par mois (~40 par seconde)
- Lecture intensive : ratio lecture/écriture de 100:1 → 4 000 redirections par seconde
- Latence de redirection inférieure à 50 ms
- Les URLs courtes doivent être aussi courtes que possible
- Haute disponibilité - un lien cassé est une expérience utilisateur désastreuse

<Callout>
Commencez chaque entretien de system design en posant des questions de clarification. Écrire des chiffres explicites (40 writes/sec, 4 000 reads/sec) montre que vous savez raisonner quantitativement sur le scale.
</Callout>

## Étape 2 : API Design (5 minutes)

```
POST /api/shorten
Body: { "url": "https://example.com/very/long/path", "custom_code": "mylink", "expires_in": "30d" }
Response: { "short_url": "https://sho.rt/aB3kx9", "expires_at": "2025-02-15T00:00:00Z" }

GET /:code
Response: HTTP 301 Redirect → Location: https://example.com/very/long/path

GET /api/analytics/:code
Response: { "clicks": 14523, "created_at": "...", "top_referrers": [...] }
```

**Pourquoi HTTP 301 vs 302 ?**
- **301 (Permanent Redirect)** - les navigateurs le mettent en cache, réduisant la charge sur nos servers
- **302 (Temporary Redirect)** - les navigateurs contactent toujours nos servers, permettant des analytics précises

Pour un service qui suit les clics, **le 302 est généralement le meilleur choix** car chaque redirection passe par nos servers.

## Étape 3 : Stratégie d'encoding

Le défi central : comment générer des codes courts et uniques ?

**L'encoding base62** utilise `[a-zA-Z0-9]` - 62 caractères. Un code de 7 caractères donne 62⁷ = **3,5 trillions** de combinaisons uniques, largement suffisant pour notre scale.

**Option A - Basée sur un compteur :**
- Utiliser un compteur auto-incrémenté (1, 2, 3, ...) et le convertir en base62
- Compteur 1000 → `g8` | Compteur 1000000 → `4c92`
- **Avantages :** unicité garantie, simple, aucune collision
- **Inconvénients :** prévisible (les utilisateurs peuvent deviner la prochaine URL), nécessite un compteur centralisé

**Option B - Basée sur le hash :**
- Hasher l'URL longue avec MD5/SHA-256, prendre les 7 premiers caractères en base62
- **Avantages :** stateless, aucune coordination centrale nécessaire
- **Inconvénients :** collisions possibles - vérifier en database et réessayer

**Option C - Clés pré-générées (recommandé) :**
- Un service en arrière-plan pré-génère des millions de codes base62 uniques dans un key pool
- Quand une nouvelle URL arrive, on prend le prochain code disponible du pool
- **Avantages :** rapide, aucun risque de collision, non prévisible
- **Inconvénients :** nécessite la gestion du service de génération de clés

<FunFact>
Le service TinyURL original a été lancé en 2002 et fonctionne encore aujourd'hui. À un moment donné, il générait plus d'un million d'URLs courtes par jour. Le domaine tinyurl.com est devenu si emblématique que « tinyurl » est utilisé comme terme générique pour le raccourcissement d'URL.
</FunFact>

## Étape 4 : Choix de database et schéma

Avec 100 millions de nouvelles URLs par mois et une rétention de 5 ans, nous stockerons environ **6 milliards d'enregistrements**. C'est une charge en lecture intensive avec des lookups clé-valeur simples.

**Choix de database : DynamoDB ou Cassandra**
- Les lookups se font toujours par short code (partition key) - pas de jointures complexes
- Les deux gèrent un throughput en lecture massif avec du horizontal scaling
- Les deux offrent des lookups en moins de 10 ms par partition key

**Schéma :**
```
Table: urls
  short_code (partition key): "aB3kx9"
  long_url: "https://example.com/very/long/path"
  user_id: "user_123"
  created_at: 1706000000
  expires_at: 1863000000
  click_count: 14523
```

Pour les analytics, une table time-series séparée stocke les événements de clic individuels :
```
Table: clicks
  short_code (partition key): "aB3kx9"
  timestamp (sort key): 1706000001
  referrer: "twitter.com"
  country: "GB"
  device: "mobile"
```

## Étape 5 : Stratégie de caching

À 4 000 lectures par seconde et une distribution à longue traîne (certaines URLs reçoivent des millions de clics, la plupart très peu), le caching est essentiel.

Le **cache Redis** stocke les mappings les plus fréquemment accédés :
```
Key:   url:aB3kx9
Value: "https://example.com/very/long/path"
TTL:   24 hours
```

**Flux en cas de cache hit (chemin rapide) :**
1. Requête entrante : `GET /aB3kx9`
2. Vérification Redis → cache hit → retour 302 redirect
3. Enregistrement asynchrone de l'événement de clic dans Kafka

**Flux en cas de cache miss :**
1. Vérification Redis → miss
2. Requête database → trouvé
3. Remplissage du cache Redis
4. Retour 302 redirect

Avec un taux de cache hit de 80 %, seules 800 des 4 000 requêtes par seconde atteignent réellement la database.

## Étape 6 : Analytics et suivi des clics

Le suivi des clics ne doit pas ralentir les redirections. La réponse de redirection doit être renvoyée immédiatement tandis que les analytics sont traitées de manière asynchrone.

**Architecture :**
1. Le handler de redirection publie un événement de clic dans **Kafka**
2. Un service de traitement des clics consomme les événements et :
   - Incrémente le `click_count` dans l'enregistrement URL
   - Écrit les données de clic détaillées dans la table clicks
   - Met à jour les dashboards en temps réel via une agrégation streaming

Ce découplage garantit qu'un pipeline analytics lent n'impacte jamais la latence de redirection.

## Étape 7 : Mise à l'échelle du système

**Approche de horizontal scaling :**
- Les **API servers** sont stateless - on scale en ajoutant des instances derrière un load balancer
- Le **service de génération de clés** tourne comme un microservice séparé avec sa propre database de codes pré-générés
- Le **cluster Redis** gère la distribution du cache entre les nœuds via consistent hashing
- Le **database sharding** par le premier caractère du short code distribue les données uniformément

**Gestion des pics de charge :**
- Lors d'événements viraux, une seule URL peut recevoir des millions de hits par minute
- Le cache Redis absorbe cela naturellement - une seule entrée de cache sert un nombre illimité de lectures
- Kafka tamponne les événements de clic, lissant la charge d'écriture sur la database analytics

<FunFact>

Lorsque la Libye - dont le code pays est .ly - a brièvement désactivé les domaines .ly en raison d'instabilité politique, cela a provoqué la panique sur internet, car des millions de liens raccourcis ont temporairement cessé de fonctionner.
</FunFact>

## Points clés à retenir

- Suivez le framework : requirements → API → encoding → data model → architecture → deep dives
- Base62 avec 7 caractères donne 3,5 trillions de codes uniques - largement suffisant
- Les key pools pré-générés évitent les problèmes de collision et les bottlenecks de compteur central
- Utilisez les redirections 302 (pas 301) quand vous avez besoin d'analytics de clics précises
- Le caching Redis réduit considérablement la charge database pour les workloads en lecture intensive
- Découplez les analytics du chemin de redirection avec Kafka pour un traitement asynchrone
