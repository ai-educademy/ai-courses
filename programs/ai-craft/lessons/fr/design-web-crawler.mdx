# Concevoir un Web Crawler - Comment Google indexe Internet

Un web crawler parcourt systématiquement Internet pour télécharger et indexer des pages web. C'est la base de tout moteur de recherche, et sa conception vous enseigne les systèmes distribués, le parcours de graphes et la gestion du désordre du web réel.

## Étape 1 - Recueil des exigences

### Exigences fonctionnelles
- Crawler des pages web à partir d'un ensemble de seed URLs
- Extraire et suivre les hyperliens pour découvrir de nouvelles pages
- Stocker le contenu téléchargé pour l'indexation et le traitement
- Respecter robots.txt et les règles de politesse de crawl

### Exigences non fonctionnelles
- **Scale** - crawler 1 milliard de pages par mois
- **Politesse** - ne jamais surcharger un seul serveur web
- **Fraîcheur** - re-crawler régulièrement les pages importantes
- **Robustesse** - gérer le HTML malformé, les boucles infinies et les spider traps

## Étape 2 - Architecture de haut niveau

```
Seed URLs → URL Frontier (priority queue)
                  ↓
            URL Scheduler (politeness + priority)
                  ↓
         Fetcher (HTTP client pool)
                  ↓
         HTML Parser → Extract URLs → URL Filter
              ↓                           ↓
       Content Store              URL Dedup (Bloom Filter)
       (S3 / HDFS)                      ↓
                                 URL Frontier (loop)
```

## Étape 3 - La boucle de crawl

### Crawling BFS vs DFS
- **BFS (Breadth-First)** - préféré pour le web crawling. Découvre rapidement un large ensemble de domaines et évite de s'enfoncer trop profondément dans un seul site
- **DFS (Depth-First)** - risque de descendre infiniment dans des pages générées dynamiquement avant de découvrir d'autres domaines

### Conception du URL Frontier
Le frontier n'est pas une simple file d'attente - c'est un système de priorité sophistiqué avec deux sous-composants :

1. **Priority Queue (Front)** - classe les URLs par importance. Les facteurs incluent le PageRank, la fréquence de mise à jour et l'autorité du domaine
2. **Politeness Queue (Back)** - garantit que chaque domaine a sa propre file FIFO. Un scheduler pioche dans différentes files de domaines pour éviter de surcharger un seul serveur

```
Front (Priority):       Back (Politeness):
┌──────────────┐       ┌─── bbc.co.uk queue ───┐
│ High Priority │ ───→  │ url1 → url4 → url7    │
│ Med Priority  │ ───→  ├─── github.com queue ──┤
│ Low Priority  │ ───→  │ url2 → url5 → url8    │
└──────────────┘       ├─── wikipedia.org queue ┤
                        │ url3 → url6 → url9    │
                        └───────────────────────┘
```

## Étape 4 - Politesse et Robots.txt

### Respecter Robots.txt
Avant de crawler un domaine, récupérez et mettez en cache son fichier `robots.txt` :

```
# Example robots.txt
User-agent: *
Disallow: /admin/
Disallow: /private/
Crawl-delay: 2
```

Mettez en cache robots.txt par domaine avec un TTL (24 heures). Respectez toujours les règles `Disallow` et les directives `Crawl-delay`.

### Limitation du débit de crawl
Même sans `Crawl-delay` explicite, maintenez la politesse :
- Maximum 1 requête par seconde par domaine (configurable)
- Utilisez des files de téléchargement séparées par domaine
- Surveillez les temps de réponse - si un serveur ralentit, réduisez automatiquement le rythme

<FunFact>Le web crawler de Google (Googlebot) parcourt des centaines de milliards de pages. Pour rester poli, il surveille les temps de réponse des serveurs et réduit son débit de crawl si un site commence à répondre lentement - laissant ainsi le site « repousser » le crawler.</FunFact>

## Étape 5 - Déduplication

### Déduplication d'URLs (Bloom Filter)
Avec des milliards d'URLs, vérifier si une URL a déjà été vue nécessite une structure de données efficace. Un Bloom filter offre des lookups en O(1) avec une empreinte mémoire réduite :

- **Espace :** ~1 Go pour 1 milliard d'URLs à un taux de faux positifs de 1 %
- **Compromis :** peut occasionnellement re-crawler une page (faux positif) mais ne manque jamais une nouvelle URL (pas de faux négatifs)

Normalisez les URLs avant vérification : supprimez les fragments (#), triez les paramètres de requête, mettez le hostname en minuscules et résolvez les chemins relatifs.

### Déduplication de contenu (SimHash)
Différentes URLs peuvent servir un contenu identique (miroirs, articles copiés). SimHash génère une empreinte du contenu de la page - les pages avec des empreintes similaires (faible distance de Hamming) sont probablement des doublons :

```
Page A content → SimHash → 10110010...
Page B content → SimHash → 10110011...
Hamming distance = 1 → likely duplicate, skip storing
```

## Étape 6 - Crawling distribué

### Mise à l'échelle horizontale
Une seule machine ne peut pas crawler un milliard de pages par mois. Distribuez le travail :

- **Consistent hashing** sur les noms de domaine assigne chaque domaine à un nœud crawler spécifique. Cela garantit la politesse - un seul nœud crawle un domaine donné
- Le **URL frontier** est partitionné entre les nœuds selon le hash ring
- La **coordination** se fait via un service léger qui gère les affectations de nœuds et rééquilibre lorsque des nœuds rejoignent ou quittent le cluster

### Cache de résolution DNS
Les lookups DNS ajoutent 10-100 ms par requête. À grande échelle, c'est un goulot d'étranglement majeur :
- Maintenez un cache DNS local par nœud crawler
- Effectuez la pré-résolution DNS par lots pour les URLs en file d'attente
- Utilisez un résolveur DNS dédié pour éviter de surcharger les serveurs DNS publics

### Détection des Spider Traps
Les sites malveillants ou mal conçus peuvent créer des espaces d'URLs infinis :
- `/page/1/page/1/page/1/...` - motifs de chemins répétitifs
- Pages de calendrier générant des URLs pour chaque date future
- Session IDs dans les URLs créant des URLs uniques pour un contenu identique

**Défenses :** longueur maximale d'URL, profondeur maximale de crawl par domaine, détection de motifs pour les structures d'URLs répétitives et limites de nombre de pages par domaine.

## Étape 7 - Goulots d'étranglement et optimisations

| Goulot d'étranglement | Solution |
|-----------|----------|
| Latence de résolution DNS | Cache DNS local, pré-résolution par lots |
| Serveurs lents ou non réactifs | Timeouts (30s), async I/O, abandon après retries |
| Stockage du contenu crawlé | Compression des pages, stockage du contenu modifié uniquement |
| Mémoire du URL frontier | Priority queue sur disque (RocksDB) |
| Contenu dupliqué | SimHash fingerprinting, détection d'URL canonique |

<ThinkAboutIt>Les moteurs de recherche doivent équilibrer la fraîcheur du crawl (re-crawler les pages fréquemment) et la découverte (trouver de nouvelles pages). Si votre crawler a un budget fixe de 1 million de pages par jour, comment répartiriez-vous ce budget entre le re-crawl des pages existantes et la découverte de nouvelles pages ?</ThinkAboutIt>
