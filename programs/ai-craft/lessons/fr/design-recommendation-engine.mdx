## Le sujet d'entretien

*"Concevez un recommendation engine pour une plateforme e-commerce qui propose des suggestions de produits personnalisées à 100 millions d'utilisateurs."*

Cette question évalue votre capacité à combiner des **pipelines de machine learning** avec un **system design scalable** - une combinaison de plus en plus courante dans les entretiens de niveau senior.

## Étape 1 : Recueil des exigences

**Exigences fonctionnelles :**
- « Recommandé pour vous » personnalisé sur la page d'accueil (top-20 produits)
- « Les clients ayant acheté ceci ont aussi acheté » sur les pages produit
- « Tendances dans votre région » basé sur l'activité régionale
- Mises à jour en temps réel : les recommandations reflètent la navigation récente en quelques minutes

**Exigences non fonctionnelles :**
- Servir les recommandations en moins de 100 ms (p99)
- Gérer 50 000 requêtes par seconde en pic
- Supporter 100 millions d'utilisateurs et 10 millions de produits
- Dégradation gracieuse : afficher les articles populaires si la personnalisation échoue

<Callout>
Définissez toujours une stratégie de fallback. Les recruteurs apprécient d'entendre « si le modèle ML échoue, on dégrade gracieusement vers des recommandations basées sur la popularité » - cela démontre une réflexion orientée production.
</Callout>

## Étape 2 : Approches de recommandation

Trois techniques fondamentales alimentent la plupart des systèmes de recommandation :

**Collaborative filtering** - « des utilisateurs similaires à vous ont aimé ces articles »
- Analyser les matrices d'interactions utilisateur-article (vues, achats, notes)
- Trouver des utilisateurs ou articles similaires via la matrix factorisation
- Fonctionne bien avec suffisamment de données d'interaction

**Content-based filtering** - « parce que vous avez aimé des articles avec ces caractéristiques »
- Utiliser les attributs produit (catégorie, marque, gamme de prix, description)
- Générer des embeddings à partir des descriptions produit avec des modèles transformer
- Calculer la similarité entre articles dans l'espace d'embedding

**Approche hybride** - combiner les deux signaux pour de meilleures recommandations
- La plupart des systèmes en production utilisent une approche hybride : signaux collaboratifs pour la personnalisation, signaux de contenu pour la diversité et la couverture cold-start

## Étape 3 : Le pipeline de recommandation

Les systèmes de recommandation en production utilisent généralement une **architecture en deux étapes** :

**Étape 1 - Candidate Generation (offline/quasi temps réel)**
- Réduire 10 millions de produits à environ 1 000 candidats par utilisateur
- Utiliser des modèles légers : collaborative filtering, similarité d'embedding
- Exécution en batch (horaire/quotidien) ou quasi temps réel avec du streaming

**Étape 2 - Ranking (temps réel)**
- Scorer les ~1 000 candidats avec un modèle plus riche
- Prendre en compte le contexte : heure, appareil, activité récente, contenu du panier
- Retourner les 20 meilleurs articles classés
- Doit s'exécuter dans le budget de latence de 100 ms

## Étape 4 : Modèle de données et Feature Store

Les recommandations dépendent de features riches calculées à partir de multiples sources de données :

**Features utilisateur :** historique d'achats, habitudes de navigation, segment démographique, catégories favorites
**Features article :** catégorie, prix, notes, embeddings de description, vélocité des ventes
**Features d'interaction :** click-through rates, temps passé à consulter, taux d'ajout au panier

Un **feature store** (ex. Feast, Tecton) fournit :
- **Offline features** - features batch précalculées pour l'entraînement du modèle
- **Online features** - récupération de features à faible latence pour le ranking en temps réel
- **Consistency** - mêmes définitions de features utilisées en entraînement et en serving

```
Feature Store Schema:
  user_features:    { user_id → [purchase_vector, category_prefs, avg_price, ...] }
  item_features:    { item_id → [embedding, category, price, popularity_score, ...] }
  interaction_features: { (user_id, item_id) → [clicks, views, last_interaction, ...] }
```

<FunFact>
Netflix estime que son système de recommandation fait économiser 1 milliard de dollars par an à l'entreprise en réduisant le churn des abonnés. Les utilisateurs qui reçoivent des suggestions pertinentes sont nettement moins susceptibles de résilier leur abonnement.
</FunFact>

## Étape 5 : Recommandations Real-Time vs Batch

| Aspect | Batch | Real-Time |
|--------|-------|-----------|
| Fraîcheur | Plusieurs heures | Quelques secondes |
| Coût de calcul | Moindre (jobs planifiés) | Plus élevé (inférence par requête) |
| Complexité | Pipeline plus simple | Infrastructure de streaming nécessaire |
| Cas d'usage | Page d'accueil « pour vous » | « Parce que vous venez de consulter X » |

**Approche hybride pratique :**
- Précalculer les listes de candidats en batch toutes les quelques heures
- Re-ranker en temps réel en utilisant le contexte de session le plus récent
- Utiliser du streaming (Kafka) pour mettre à jour les features utilisateur en quasi temps réel à mesure que les événements surviennent

<ThinkAboutIt>
Lorsque vous consultez un produit sur Amazon, les recommandations se mettent à jour presque instantanément. Pourtant, entraîner des modèles de collaborative filtering prend des heures. Comment pensez-vous qu'Amazon comble cet écart entre l'entraînement batch et la personnalisation en temps réel ?
</ThinkAboutIt>

## Étape 6 : Framework d'A/B Testing

Les systèmes de recommandation nécessitent une expérimentation rigoureuse :

- **Traffic splitting** - router un pourcentage d'utilisateurs vers le nouveau modèle
- **Métriques** - click-through rate (CTR), taux de conversion, revenu par session, diversité
- **Significativité statistique** - exécuter les tests assez longtemps pour obtenir des conclusions fiables
- **Guardrail metrics** - s'assurer que les nouveaux modèles ne dégradent pas le revenu ou l'engagement global

Un setup typique utilise un **feature flag service** pour assigner les utilisateurs aux buckets d'expérimentation de manière cohérente (un même utilisateur voit toujours la même variante durant un test).

## Étape 7 : Solutions au Cold-Start

Le problème du cold-start se manifeste tant pour les nouveaux utilisateurs que pour les nouveaux articles :

**Nouveaux utilisateurs :**
- Afficher initialement des recommandations basées sur la popularité
- Poser des questions d'onboarding (« Sélectionnez les catégories qui vous intéressent »)
- Utiliser des signaux démographiques ou géographiques comme features initiales
- Basculer vers des recommandations personnalisées après 5 à 10 interactions

**Nouveaux articles :**
- Utiliser les features content-based (description, catégorie, prix) pour placer les articles dans l'espace d'embedding
- Mettre en avant les nouveaux articles dans une section dédiée « Nouveautés »
- Appliquer des stratégies exploration/exploitation (montrer les nouveaux articles à un sous-ensemble d'utilisateurs pour collecter des données d'interaction)

## Étape 8 : Mise à l'échelle du système

**Architecture pour 50K requêtes par seconde :**
- **API gateway** - rate limiting, authentification, routage des requêtes
- **Service de recommandation** - stateless, scalé horizontalement derrière un load balancer
- **Feature store** - cluster Redis pour la récupération de features à faible latence
- **Model serving** - modèles préchargés en mémoire (TensorFlow Serving, Triton)
- **Cache précalculé** - cache Redis des meilleures recommandations par utilisateur, rafraîchi par des jobs batch
- **Fallback** - en cas de cache miss et timeout du modèle, retourner des résultats par défaut basés sur la popularité

<FunFact>
La playlist Discover Weekly de Spotify est générée chaque lundi par un pipeline de recommandation batch qui analyse les habitudes d'écoute de plus de 100 millions d'utilisateurs. Cette fonctionnalité a connu un tel succès qu'elle est devenue le principal moyen pour les utilisateurs de découvrir de nouvelle musique.
</FunFact>

## Points clés à retenir

- Utilisez une architecture en deux étapes : candidate generation (large) → ranking (précis)
- Les approches hybrides (collaborative + content-based) surpassent les systèmes à méthode unique
- Les feature stores garantissent la cohérence entre l'entraînement et le serving en temps réel
- L'A/B testing est essentiel - optimisez pour les métriques business, pas uniquement les métriques ML
- Les stratégies cold-start évitent une mauvaise expérience pour les nouveaux utilisateurs et articles
- Mettez en cache agressivement et dégradez gracieusement pour gérer les pics de trafic
