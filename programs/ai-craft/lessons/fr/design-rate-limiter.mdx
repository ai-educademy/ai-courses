# Concevoir un Rate Limiter - Protéger les APIs à grande échelle

Le rate limiting est l'un des sujets de system design les plus pratiques. Chaque API en production - de GitHub à Stripe - s'en sert pour prévenir les abus, garantir une utilisation équitable et protéger les services backend contre la surcharge.

## Étape 1 - Recueil des exigences

### Exigences fonctionnelles
- Limiter les requêtes selon des règles configurables (par utilisateur, IP ou API key)
- Retourner un message clair lorsqu'un client est limité (HTTP 429)
- Supporter plusieurs règles de rate limiting simultanément

### Exigences non fonctionnelles
- **Faible latence** - le limiter ne doit pas ajouter de surcharge significative
- **Distribué** - fonctionne sur plusieurs serveurs
- **Tolérant aux pannes** - si le limiter échoue, le trafic doit continuer (fail-open)

<ThinkAboutIt>Un rate limiter doit-il être fail-open (autoriser le trafic) ou fail-closed (bloquer le trafic) en cas de panne ? Quels sont les compromis pour un site e-commerce par rapport à une API bancaire ?</ThinkAboutIt>

## Étape 2 - Algorithmes de Rate Limiting

### Fixed Window Counter
Divisez le temps en fenêtres fixes (par ex. intervalles de 60 secondes). Incrémentez un compteur par requête. Réinitialisez aux limites de fenêtre. **Problème :** le trafic en rafale aux bords des fenêtres peut permettre 2× la limite.

### Sliding Window Log
Stockez le timestamp de chaque requête. Comptez les entrées dans la fenêtre glissante. Précis mais **gourmand en mémoire** - stocker chaque timestamp est coûteux à grande échelle.

### Sliding Window Counter
Un hybride : combinez le compteur de la fenêtre courante avec une portion pondérée de la fenêtre précédente. Équilibre entre précision et efficacité mémoire.

### Token Bucket Algorithm (le plus populaire)
Un bucket contient des tokens jusqu'à une capacité maximale. Les tokens sont ajoutés à un débit fixe. Chaque requête consomme un token. Si le bucket est vide, la requête est rejetée.

```
Bucket: capacity=10, refill_rate=2/sec

Time 0s: [🪙🪙🪙🪙🪙🪙🪙🪙🪙🪙] → 10 tokens
Burst of 8 requests → [🪙🪙] → 2 tokens remain
Time 1s: refill +2 → [🪙🪙🪙🪙] → 4 tokens
```

### Comparaison des algorithmes

| Algorithme | Mémoire | Précision | Gestion des rafales |
|-----------|---------|-----------|---------------------|
| Fixed Window | Faible | Faible (rafales aux bords) | Mauvaise |
| Sliding Log | Élevée | Exacte | Bonne |
| Sliding Counter | Moyenne | Approximative | Bonne |
| Token Bucket | Faible | Bonne | Excellente |

<Quiz question="Quel algorithme permet des rafales contrôlées tout en maintenant un débit stable à long terme ?" options="Fixed Window Counter|Sliding Window Log|Token Bucket|Sliding Window Counter" answer="2" explanation="Le Token Bucket autorise de courtes rafales jusqu'à la capacité du bucket tout en imposant un débit moyen via le mécanisme de refill. C'est idéal pour les APIs qui doivent gérer des pics de trafic." />

## Étape 3 - Architecture de haut niveau

```
Client → API Gateway (Rate Limiter Middleware) → Backend Services
                    ↕
              Redis Cluster
         (counters + token buckets)
```

<Illustration src="/images/lessons/design-rate-limiter.svg" alt="Architecture du rate limiter montrant les clients, l'API gateway avec le middleware de rate limiting, le cluster Redis pour l'état distribué et les services backend" caption="Le rate limiter se situe dans l'API gateway, utilisant Redis pour le stockage distribué des compteurs" />

### Où placer le Rate Limiter
- **Côté client** - facile à contourner, peu fiable
- **Côté serveur** - par instance uniquement, ne passe pas à l'échelle
- **API Gateway / Middleware** - la meilleure approche pour les systèmes distribués ; application centralisée sur toutes les instances

## Étape 4 - Approfondissement : Rate Limiting distribué

### Redis + Lua Scripts
Dans un système distribué, plusieurs serveurs doivent partager l'état du rate limiting. Redis est le choix standard grâce à ses opérations atomiques et sa latence sub-milliseconde.

Un Lua script garantit l'atomicité - la vérification et la décrémentation se font en une seule opération Redis, évitant les race conditions :

```
-- Token Bucket in Redis (pseudocode)
local tokens = redis.call('GET', key)
local last_refill = redis.call('GET', key..':time')
-- Calculate tokens to add based on elapsed time
-- If tokens > 0: decrement and ALLOW
-- Else: DENY
```

### En-têtes de réponse
Les rate limiters bien conçus communiquent leur état aux clients :
- `X-RateLimit-Limit: 100` - requêtes maximales par fenêtre
- `X-RateLimit-Remaining: 42` - requêtes restantes
- `X-RateLimit-Reset: 1672531200` - epoch UTC de réinitialisation de la fenêtre
- HTTP `429 Too Many Requests` avec un en-tête `Retry-After`

<Quiz question="Pourquoi utilise-t-on des Lua scripts avec Redis pour le rate limiting plutôt que des commandes GET et SET séparées ?" options="Lua est plus rapide que les commandes Redis|Les Lua scripts s'exécutent de manière atomique sur Redis, empêchant les race conditions|Redis ne supporte pas GET/SET nativement|Lua réduit la bande passante réseau" answer="1" explanation="Les Lua scripts s'exécutent de manière atomique sur le serveur Redis. Sans atomicité, deux requêtes concurrentes pourraient lire le même nombre de tokens, constater qu'il en reste, et toutes deux décrémenter - autorisant plus de requêtes que la limite ne le permet." />

## Étape 5 - Gestion des cas limites

### Race Conditions
Sans opérations atomiques, des requêtes concurrentes peuvent contourner les limites. Solutions : Lua scripts Redis ou transactions Redis `MULTI/EXEC`.

### Dimensions du Rate Limiting
Appliquez des limites à plusieurs niveaux simultanément :
- **Par utilisateur** - 100 requêtes/minute pour les utilisateurs authentifiés
- **Par IP** - 50 requêtes/minute pour le trafic anonyme
- **Par API key** - plans par paliers (free : 10/min, pro : 1000/min)

### Distributed Clock Skew
Les serveurs peuvent avoir des horloges légèrement décalées. Utilisez l'heure du serveur Redis comme source unique de vérité plutôt que les timestamps des serveurs applicatifs.

<FunFact>L'API REST de GitHub autorise 5 000 requêtes par heure pour les utilisateurs authentifiés mais seulement 60 par heure pour les requêtes non authentifiées. Stripe utilise une approche sliding window avec des limites séparées pour les API keys en mode live et test.</FunFact>

## Étape 6 - Goulots d'étranglement et optimisations

| Goulot d'étranglement | Solution |
|----------------------|----------|
| Redis point de défaillance unique | Redis Cluster avec réplication |
| Latence réseau vers Redis | Cache local en mémoire avec synchronisation périodique |
| Hot keys (API keys populaires) | Répartition sur plusieurs nœuds Redis |
| Surcharge d'évaluation des règles | Pré-compilation des règles, mise en cache des configurations |
