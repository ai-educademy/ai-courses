## Le sujet d'entretien

*"Concevez une plateforme AI API qui sert plusieurs modèles de machine learning aux développeurs via une REST API — de manière similaire à la façon dont OpenAI ou Hugging Face servent des modèles à grande échelle."*

Cette question est de plus en plus populaire en entretien car elle combine la **conception de plateforme API** avec les **défis d'infrastructure GPU** propres aux workloads d'IA.

## Étape 1 : Recueil des exigences

**Exigences fonctionnelles :**
- Servir plusieurs modèles d'IA (génération de texte, classification d'images, embeddings) via REST API
- Authentication par API key avec quotas d'utilisation par clé
- Suivi d'utilisation et facturation par appel API
- Support de plusieurs versions de modèles avec A/B testing
- Réponses en streaming pour les tâches de génération longues

**Exigences non fonctionnelles :**
- Latence : p50 inférieur à 500ms pour les modèles d'embedding, p50 inférieur à 2s pour les modèles génératifs
- Débit : 10 000 requêtes par seconde sur l'ensemble des modèles
- Disponibilité : 99,9% d'uptime avec dégradation gracieuse
- Efficacité des coûts : les GPUs sont chers — maximiser l'utilisation

<Callout>
Les plateformes d'IA ont une latence particulièrement variable. Une requête de complétion de texte peut prendre 200ms ou 30 secondes selon la longueur de la sortie. Discutez toujours de la façon dont votre conception gère cette variance.
</Callout>

## Étape 2 : Conception de l'API

```
POST /v1/completions
Headers: Authorization: Bearer sk-abc123
Body: { "model": "gpt-4", "prompt": "Explain caching", "max_tokens": 200 }

POST /v1/embeddings
Headers: Authorization: Bearer sk-abc123
Body: { "model": "text-embedding-3", "input": "Hello world" }

GET  /v1/models                    — lister les modèles disponibles
GET  /v1/usage?start=2024-01-01    — données d'utilisation et de facturation

SSE: POST /v1/completions (stream: true)
     → returns Server-Sent Events with incremental tokens
```

Le streaming via **Server-Sent Events (SSE)** est essentiel pour les modèles génératifs — les utilisateurs voient les tokens apparaître progressivement au lieu d'attendre la réponse complète.

## Étape 3 : API Gateway et authentication

L'API gateway est la porte d'entrée de la plateforme :

- **Authentication** — valider les API keys contre un key store (Redis pour la rapidité, base de données pour la persistance)
- **Rate limiting** — appliquer des limites par clé (ex. : 60 requêtes/minute pour le tier gratuit)
- **Routage des requêtes** — diriger les requêtes vers le bon cluster de model-serving
- **Validation des requêtes** — vérifier le schéma du payload, la disponibilité du modèle, les limites de tokens

**Implémentation du rate limiting :**
Utiliser un **sliding window counter** dans Redis :
```
Key:   rate:{api_key}:{window_minute}
Value: request count
TTL:   120 seconds
```

Chaque requête incrémente le compteur. Si la limite est dépassée, renvoyer HTTP 429.

## Étape 4 : Model serving et allocation GPU

C'est ici que les plateformes d'IA diffèrent fondamentalement des services web traditionnels. Les GPUs sont rares, coûteux et nécessitent une orchestration spécialisée.

**Couche de model serving :**
- Les modèles sont chargés en mémoire GPU avec des frameworks comme **Triton Inference Server** ou **vLLM**
- Chaque version de modèle s'exécute sur un pool dédié d'instances GPU
- Les large language models (LLMs) peuvent nécessiter du **model parallelism** sur plusieurs GPUs

**Batching des requêtes :**
Les requêtes d'inférence individuelles sont inefficaces sur GPU. Le **dynamic batching** regroupe plusieurs requêtes pour les traiter ensemble :
- Collecter les requêtes pendant 10ms maximum ou jusqu'à atteindre 32 requêtes
- Traiter le batch entier en un seul GPU forward pass
- Retourner les réponses individuelles à chaque appelant

Cela peut améliorer le débit de 5 à 10x par rapport au traitement individuel.

<FunFact>
Un seul GPU NVIDIA H100 coûte environ 25 000 £ et consomme 700 watts. Faire tourner un cluster de 1 000 GPUs pour le model serving coûte plus de 2 millions £ par mois rien qu'en électricité et matériel — c'est pourquoi le batching et l'utilisation efficaces comptent énormément.
</FunFact>

## Étape 5 : A/B Testing entre versions de modèles

Lors du déploiement d'une nouvelle version de modèle, un rollout contrôlé est nécessaire :

- **Traffic splitting** — router 5% des requêtes vers le nouveau modèle, 95% vers le modèle actuel
- **Shadow mode** — exécuter le nouveau modèle sur le trafic de production sans retourner ses résultats aux utilisateurs
- **Comparaison de métriques** — latence, scores de qualité, taux d'erreur, coût par requête
- **Rollout progressif** — augmenter le trafic vers la nouvelle version à mesure que la confiance grandit

La logique de routage réside dans l'API gateway :
```
model: "gpt-4"  →  95% → gpt-4-v2.1 (current)
                →   5% → gpt-4-v2.2 (candidate)
```

<ThinkAboutIt>
Évaluer si une nouvelle version de LLM est « meilleure » est étonnamment difficile — contrairement aux modèles de classification, il n'existe pas de métrique d'accuracy simple. Comment concevriez-vous un système d'évaluation automatisé de la qualité pour les modèles d'IA générative ?
</ThinkAboutIt>

## Étape 6 : Suivi d'utilisation et facturation

Les API d'IA facturent généralement au token (tokens d'entrée + de sortie pour les LLMs, par requête pour les autres modèles).

**Pipeline de facturation event-driven :**
1. Chaque réponse API inclut un header d'utilisation : `x-usage-tokens: 847`
2. Un événement de facturation est publié sur Kafka : `{ api_key, model, input_tokens, output_tokens, timestamp }`
3. Un service d'agrégation consomme les événements et met à jour les totaux cumulés
4. Les dashboards d'utilisation interrogent les données agrégées depuis une base de données time-series

**Pourquoi Kafka ?** La facturation doit être fiable mais n'a pas besoin d'être synchrone. Si le service de facturation est temporairement indisponible, les événements s'accumulent dans Kafka et sont traités lors de la reprise — aucune donnée d'utilisation n'est perdue.

## Étape 7 : Monitoring et observabilité

Les plateformes d'IA nécessitent un monitoring au-delà des services web classiques :

| Métrique | Pourquoi c'est important |
|----------|--------------------------|
| Latence d'inférence (p50, p95, p99) | Expérience utilisateur et conformité SLA |
| Utilisation GPU | Efficacité des coûts — les GPUs inactifs gaspillent de l'argent |
| Tokens par seconde | Capacité de débit et signaux de scaling |
| Taux d'erreur par modèle | Détecter les défaillances spécifiques à un modèle |
| Profondeur de la queue | Déclencheur de scaling — des queues profondes indiquent une demande supérieure à la capacité |
| Scores de qualité des sorties | Détecter la dégradation du modèle (via évaluation automatisée) |

**Seuils d'alerte :** une utilisation GPU en dessous de 40% déclenche un scale-down ; une profondeur de queue au-dessus de 100 déclenche un scale-up.

## Étape 8 : Auto-Scaling

Les workloads d'IA ont une demande très variable. L'auto-scaling doit équilibrer réactivité et coût :

- **Horizontal scaling** — ajouter des instances GPU quand la profondeur de queue ou la latence dépasse les seuils
- **Predictive scaling** — utiliser les patterns de trafic historiques pour pré-provisionner la capacité avant les pics attendus
- **Scale-to-zero** — pour les modèles rarement utilisés, arrêter toutes les instances et cold-start à la demande
- **Warm pools** — garder un petit nombre d'instances avec les modèles pré-chargés pour éviter la latence de cold-start

La stratégie scale-to-zero est particulièrement importante pour les plateformes hébergeant des centaines de modèles — on ne peut pas garder chaque modèle chargé en mémoire GPU en permanence.

Chacun a des caractéristiques de latence, des besoins GPU et une tarification différents. Comment concevriez-vous la couche de routage pour partager efficacement un cluster GPU fini entre des modèles aux profils si différents ?
</ThinkAboutIt>

<FunFact>

## Points clés à retenir

- Les plateformes AI API combinent la conception d'API traditionnelle avec les défis d'infrastructure spécifiques aux GPUs
- Le request batching améliore considérablement le débit GPU et l'efficacité des coûts
- Le rate limiting protège la plateforme et les clients contre les coûts incontrôlés
- La facturation asynchrone via Kafka assure un suivi d'utilisation fiable sans bloquer les requêtes
- L'auto-scaling doit équilibrer réactivité et coût — scale-to-zero pour les modèles à faible trafic
- Le monitoring de l'utilisation GPU et de la profondeur de queue guide les décisions de scaling intelligentes
