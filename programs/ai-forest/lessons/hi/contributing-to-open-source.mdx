# Open Source में योगदान

Open-source में योगदान सीखने, विश्वसनीयता बनाने और AI समुदाय से जुड़ने का सबसे तेज़ तरीका है। इस लेसन में हम आपको बताएंगे कि शुरुआत कैसे करें और ऐसे योगदान कैसे दें जो वाकई मायने रखते हैं।

## योगदान क्यों करें?

Open source में योगदान कोई दान नहीं है — यह एक स्मार्ट करियर मूव है।

**सीखना** — प्रोडक्शन-क्वालिटी कोड पढ़ना और उसमें बदलाव करना आपको ऐसे पैटर्न सिखाता है जो ट्यूटोरियल में कभी नहीं मिलते। आप सीखते हैं कि असली सिस्टम edge cases, testing और scale को कैसे हैंडल करते हैं।

**नेटवर्किंग** — आपके PR reviewers और co-contributors अक्सर collaborators, referees, या यहाँ तक कि co-founders बन जाते हैं। AI open-source समुदाय बेहद करीबी है।

**करियर पर असर** — टॉप AI कंपनियों के hiring managers GitHub प्रोफ़ाइल को actively देखते हैं। किसी अच्छे प्रोजेक्ट में merged PR किसी certification से कहीं ज़्यादा योग्यता दर्शाता है।

**वापस देना** — आपने लगभग निश्चित रूप से open-source टूल्स का फ़ायदा उठाया है। योगदान ही इस इकोसिस्टम को टिकाए रखता है।

<FunFact>
2024 की एक GitHub सर्वे में पाया गया कि 87% AI/ML hiring managers उम्मीदवारों का मूल्यांकन करते समय open-source योगदान को ध्यान में रखते हैं — यह relevant work experience के बाद दूसरा सबसे महत्वपूर्ण सिग्नल है।
</FunFact>

## सही प्रोजेक्ट कैसे खोजें

PyTorch या TensorFlow में योगदान से शुरुआत न करें। वहाँ से शुरू करें जहाँ आपका योगदान दिखेगा और सराहा जाएगा।

**कहाँ देखें:**
- **GitHub Topics** — `good-first-issue` + `machine-learning` या `artificial-intelligence` ब्राउज़ करें
- **Hugging Face** — उनकी `transformers`, `datasets`, और `diffusers` लाइब्रेरीज़ contributors का स्वागत करती हैं
- **LangChain / LlamaIndex** — तेज़ी से बढ़ते प्रोजेक्ट्स जिनमें कई open issues हैं
- **छोटे टूल्स** — 500–5,000 stars वाली लाइब्रेरीज़ के maintainers अक्सर responsive होते हैं और मदद की कद्र करते हैं

**प्रोजेक्ट का मूल्यांकन कैसे करें:**
1. आखिरी commit की तारीख़ देखें — क्या यह actively maintained है?
2. CONTRIBUTING.md पढ़ें — क्या यह मौजूद है?
3. Open PRs देखें — क्या उनकी review हो रही है या उन्हें ignore किया जा रहा है?
4. Issue response time चेक करें — क्या maintainers engage करते हैं?

## योगदान का वर्कफ़्लो

हर योगदान एक ही पैटर्न फ़ॉलो करता है। इसे एक बार सीख लो, फिर कहीं भी योगदान कर सकते हो।

### स्टेप 1: Fork और Clone करें

```bash
gh repo fork owner/project --clone
cd project
git remote add upstream https://github.com/owner/project.git
```

### स्टेप 2: Branch बनाएं

```bash
git checkout -b fix/improve-tokeniser-docs
```

descriptive branch names इस्तेमाल करें। `fix/`, `feat/`, या `docs/` जैसे prefixes maintainers को एक नज़र में intent समझने में मदद करते हैं।

### स्टेप 3: अपने बदलाव करें

- प्रोजेक्ट की coding style को बिल्कुल फ़ॉलो करें
- अपने बदलावों से पहले और बाद में मौजूदा tests रन करें
- नई functionality के लिए tests जोड़ें
- बदलाव focused रखें — एक PR में एक ही concern

### स्टेप 4: Commit और Push करें

```bash
git add .
git commit -m "docs: clarify tokeniser padding behaviour for batch inputs"
git push origin fix/improve-tokeniser-docs
```

### स्टेप 5: Pull Request खोलें

एक स्पष्ट PR description लिखें जो बताए कि आपने **क्या** बदला, **क्यों** बदला, और इसे **कैसे** test करें। अगर कोई issue नंबर है तो उसका reference दें।

<ThinkAboutIt>
किसी open-source AI टूल में आपने जो bug देखा है, उसके बारे में सोचें। क्या आप उसे खुद fix करके PR submit कर सकते थे? अक्सर जवाब हाँ होता है — और वह पहला योगदान सबसे बड़ी मानसिक बाधा होती है।
</ThinkAboutIt>

## बेहतरीन PRs और Commits कैसे लिखें

आपका PR आपका प्रोफ़ेशनल विज़िटिंग कार्ड है। इसे बेहतरीन बनाएं।

**Commit messages** — Conventional commits इस्तेमाल करें: `feat:`, `fix:`, `docs:`, `test:`, `refactor:`। Subject 72 characters से कम रखें। जटिल बदलावों के लिए body जोड़ें।

**PR description में शामिल करें:**
- बदलाव का एक-लाइन सारांश
- संदर्भ: यह कौन सी समस्या हल करता है?
- बदलाव को कैसे test करें
- अगर relevant हो तो screenshots या output samples
- आपने जो भी trade-offs या decisions लिए

**आम गलतियों से बचें:**
- एक PR में असंबंधित बदलाव मिलाना
- प्रोजेक्ट के linting और formatting नियमों को ignore करना
- Review feedback का जवाब देने में देरी करना
- Review के दौरान बिना चेतावनी force-push करना

## कोड से परे

कुछ सबसे मूल्यवान योगदानों में एक भी लाइन कोड नहीं होती।

**Documentation** — Typos ठीक करें, examples बेहतर बनाएं, docs का अनुवाद करें। कई प्रोजेक्ट्स को documentation मदद की सख़्त ज़रूरत है और यह शुरुआत का बेहतरीन ज़रिया है।

**Issue triage** — Bugs reproduce करें, context जोड़ें, issues को label करें। यह unprocessed issues में डूबे maintainers के लिए बेहद क़ीमती है।

**Translations** — AI टूल्स को वैश्विक स्तर पर सुलभ बनाने में मदद करें। Hugging Face जैसे प्रोजेक्ट्स translators की तलाश में रहते हैं।

**Community support** — Discord, Stack Overflow, या GitHub Discussions पर सवालों के जवाब दें। दूसरों की मदद करना भी एक योगदान है।

<Callout>
**प्रो टिप:** Documentation PRs की acceptance rate सभी योगदान प्रकारों में सबसे ज़्यादा होती है। ये codebase सीखने, maintainers का विश्वास जीतने, और बाद में बड़े code contributions की नींव रखने का सबसे बढ़िया तरीका है।
</Callout>

## पब्लिक प्रोफ़ाइल बनाना

लगातार योगदान एक शक्तिशाली प्रोफ़ेशनल एसेट में बदल जाता है।

**व्यावहारिक कदम:**
1. अपने बेहतरीन योगदानों को GitHub प्रोफ़ाइल पर pin करें
2. हर योगदान से सीखी गई बातों पर छोटे blog posts लिखें
3. प्रोजेक्ट discussions में हिस्सा लें — सोच-समझकर की गई टिप्पणियाँ ध्यान खींचती हैं
4. अपने प्रोजेक्ट्स से जुड़ी virtual meetups या conferences में भाग लें
5. Open-source काम को अपने CV में merged PRs के links के साथ जोड़ें

Andrej Karpathy के neural network लाइब्रेरीज़ में शुरुआती open-source योगदान ने Tesla या OpenAI जॉइन करने से बहुत पहले उनकी प्रतिष्ठा बनाने में मदद की। आपकी GitHub प्रोफ़ाइल एक जीवंत पोर्टफ़ोलियो है।
</FunFact>

## मुख्य बातें

- Open-source योगदान सीखने, नेटवर्किंग और करियर ग्रोथ को तेज़ करता है
- छोटे, actively maintained प्रोजेक्ट्स से शुरू करें — mega-repos से नहीं
- Fork → branch → PR वर्कफ़्लो में महारत हासिल करें और इसे हर जगह इस्तेमाल करें
- स्पष्ट, focused PRs लिखें जिनमें descriptive commit messages हों
- Documentation और issue triage मूल्यवान योगदान हैं जिनमें कोड की ज़रूरत नहीं
- Scale से ज़्यादा consistency मायने रखती है — नियमित छोटे योगदान compound होते हैं
