# MLOps और Deployment

94% accuracy वाली Jupyter notebook कोई product नहीं है। जैसे ही आप laptop बंद करते हैं, वो model किसी के काम नहीं आता। MLOps वो discipline है जो experimentation और production के बीच की खाई को पाटता है - और यही वो जगह है जहाँ ज़्यादातर AI teams सबसे ज़्यादा struggle करती हैं।

## Notebooks Production क्यों नहीं हैं

Notebooks बुरी engineering habits को बढ़ावा देती हैं: cells के गलत क्रम में चलने से hidden state, hardcoded file paths, कोई dependency management नहीं, और reproducibility बिल्कुल ज़ीरो। Notebook में train किया गया model नया data आने पर अपने-आप retrain नहीं हो सकता, टूटने पर rollback नहीं हो सकता, और degradation के लिए monitor नहीं किया जा सकता।

Production ML के लिए **versioned code, versioned data, versioned models, और automated pipelines** ज़रूरी हैं। बाकी सब सिर्फ demo है।

## MLOps Lifecycle

MLOps lifecycle एक continuous loop है, कोई linear pipeline नहीं। हर stage दूसरे stages को feedback देता है, और आपकी organisation की maturity तय करती है कि ये transitions कितने automated हैं।

Google तीन MLOps maturity levels define करता है:
- **Level 0** - Manual, script-driven process। Data scientists models को engineers को handoff करते हैं।
- **Level 1** - Automated training pipelines। नए data पर retraining अपने-आप trigger होती है।
- **Level 2** - ML के लिए पूरा CI/CD। Code, data, और model changes सब automated pipelines से गुज़रते हैं।

Loop के stages:

1. **Data ingestion और validation** - Incoming data को collect, clean, और schemas के against validate करें
2. **Feature engineering** - Raw data को feature store के ज़रिए model-ready features में transform करें
3. **Training** - Tracked hyperparameters के साथ reproducible training jobs चलाएँ
4. **Evaluation** - Held-out test sets पर candidate models की baselines से तुलना करें
5. **Registry** - Approved models को metadata, lineage, और approval status के साथ store करें
6. **Deployment** - Model को canary या blue-green rollout के साथ API के पीछे serve करें
7. **Monitoring** - Prediction quality, latency, throughput, और data drift को track करें
8. **Retraining** - Performance गिरने पर नई training runs trigger करें

## Model Registries

Model registry आपके trained models का version control system है। **MLflow**, **Weights & Biases**, और **Neptune** जैसे tools आपको model artefacts को उनकी training metrics, hyperparameters, dataset versions, और promotion status (staging → production → archived) के साथ store करने देते हैं।

Registry ज़रूरी सवालों के जवाब देती है: *अभी production में कौन-सा model है? इसे किस data पर train किया गया था? किसने approve किया? Evaluation metrics क्या थीं?*

## Machine Learning के लिए CI/CD

Traditional CI/CD code test करता है। ML CI/CD को तीन चीज़ें test करनी होती हैं: **code, data, और models**।

- **Code tests** - Feature engineering logic, data transformations, और API contracts के लिए unit tests
- **Data tests** - Schema validation, distribution checks (क्या नया data expected ranges में है?), और completeness checks (कोई missing columns तो नहीं)
- **Model tests** - Validation sets पर performance thresholds, latency benchmarks, bias metrics, और current production model से तुलना

अगर कोई pull request feature pipeline बदलता है, तो उसे अपने-आप retraining और evaluation trigger करनी चाहिए। अगर नया model baseline से ख़राब perform करता है, तो PR block हो जाता है।

## ML के लिए Containerisation

Production ML के लिए Docker ज़रूरी है। आपका training environment और serving environment एक जैसा होना चाहिए - "मेरी machine पर तो चल रहा था" वाली problem तब और बढ़ जाती है जब GPU drivers, CUDA versions, और Python dependencies सबको match करना हो।

एक typical ML Dockerfile की layers: base CUDA image → Python dependencies → model artefacts → serving framework। Multi-stage builds final image को lean रखते हैं।

ML containerisation के लिए ज़रूरी practices:
- **हर dependency pin करें** - `requirements.txt` में exact versions use करें, ranges नहीं
- **Digest-based base images use करें** - `nvidia/cuda@sha256:...` न कि `:latest`
- **Training और serving images अलग रखें** - Training images को compilers और dev tools चाहिए; serving images को सिर्फ runtime
- **Vulnerabilities के लिए scan करें** - ML images deep dependency trees से CVEs inherit करती हैं

<FunFact>अकेली NVIDIA CUDA base image 3 GB से ज़्यादा होती है। Production teams अक्सर cold-start times और container registry costs कम करने के लिए Docker images optimise करने में दिन लगा देती हैं।</FunFact>

## Serving Infrastructure

Model कैसे serve करना है, ये आपकी latency और throughput requirements पर निर्भर करता है:

| Tool | सबसे अच्छा किसके लिए | मुख्य Feature |
|------|----------------------|--------------|
| **TorchServe** | PyTorch models | Built-in batching, model versioning |
| **Triton Inference Server** | Multi-framework, GPU | Dynamic batching, concurrent model execution |
| **vLLM** | Large language models | PagedAttention, continuous batching |
| **TensorFlow Serving** | TF/Keras models | gRPC + REST, automatic model reload |
| **BentoML** | कोई भी framework | Pythonic API, built-in containerisation |

LLM serving के लिए खास तौर पर **vLLM** industry standard बन चुका है क्योंकि इसका PagedAttention mechanism naive serving की तुलना में GPU memory waste को 90% तक कम कर देता है।

## Model Drift की Monitoring

Deployment के समय accurate model समय के साथ degrade होगा। **Data drift** तब होता है जब input distribution बदलती है (जैसे, pandemic के बाद customer behaviour बदलना)। **Concept drift** तब होता है जब inputs और outputs के बीच का relationship बदलता है (जैसे, fraud patterns का evolve होना)।

इन signals को monitor करें:
- **Prediction distribution** - क्या outputs shift हो रहे हैं? सामान्य से ज़्यादा high-confidence predictions तो नहीं?
- **Feature distributions** - Current data की training data से तुलना करने वाले statistical tests (KS test, PSI)
- **Ground truth feedback** - जब labels आएँ, तो live accuracy/F1 compute करें और baseline से compare करें
- **Latency और error rates** - Infrastructure health अप्रत्यक्ष रूप से model quality को प्रभावित करती है

<ThinkAboutIt>आपका e-commerce recommendation model pandemic से पहले train हुआ था। Pandemic के बाद shopping patterns पूरी तरह बदल गए हैं। आप इस drift को automatically कैसे detect करेंगे, और आपकी retraining strategy क्या होगी?</ThinkAboutIt>

## Feature Stores

Feature store, feature definitions और values की एक centralised repository है, जो training और serving दोनों में shared होती है। **Feast**, **Tecton**, और **Hopsworks** जैसे tools **training-serving skew** की critical problem solve करते हैं - जब training और inference के दौरान features अलग-अलग तरीके से compute होते हैं।

Feature stores provide करते हैं: offline store (training के लिए batch features), online store (serving के लिए low-latency features), और registry (feature definitions, ownership, lineage)।

## Batch vs Real-Time Inference

हर prediction को real time में होने की ज़रूरत नहीं है। Cost का फ़र्क बहुत बड़ा है:

- **Batch inference** - एक schedule पर predictions चलाएँ (hourly, daily)। सस्ता, spot instances use करता है। Recommendations, risk scoring, report generation के लिए ideal।
- **Real-time inference** - API के ज़रिए sub-second predictions। महँगा, always-on GPU instances चाहिए। Chatbots, fraud detection, autonomous systems के लिए ज़रूरी।
- **Near-real-time (streaming)** - Kafka या Pub/Sub के ज़रिए events को आते ही process करें। Personalisation और anomaly detection जैसे use cases के लिए latency और cost का balance।

एक common pattern है कि 95% predictable cases के लिए batch predictions pre-compute कर लें, और सिर्फ बाकी 5% को real-time endpoint पर route करें।

Cost optimisation tip: **Inference endpoints को autoscale करें** request queue depth के आधार पर, CPU utilisation पर नहीं। ML workloads bursty होते हैं - queue depth पर scaling over-provisioning और traffic spikes में request timeouts दोनों से बचाता है।
