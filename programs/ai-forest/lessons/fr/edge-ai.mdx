# Edge AI

Chaque prédiction n'a pas besoin de faire l'aller-retour vers le cloud. L'Edge AI exécute les modèles directement sur l'appareil — votre téléphone, votre navigateur, une caméra, une voiture. Les avantages sont considérables : latence réseau nulle, confidentialité totale (les données ne quittent jamais l'appareil), fonctionnement hors ligne et coûts d'exploitation radicalement réduits.

Le défi ? Il faut un modèle qui tient en mégaoctets, tourne sur un matériel limité et fournit quand même une précision exploitable.

## Pourquoi l'Edge plutôt que le Cloud ?

Le choix entre Edge et Cloud n'est pas binaire — la plupart des systèmes en production utilisent une approche hybride. Mais comprendre les compromis est essentiel :

| Facteur | Inférence Cloud | Inférence Edge |
|---------|----------------|----------------|
| **Latence** | 100-500ms (aller-retour réseau) | 5-50ms (sur l'appareil) |
| **Confidentialité** | Données envoyées au serveur | Données restent sur l'appareil |
| **Coût** | Coût GPU par requête | Coût marginal nul |
| **Hors ligne** | Connectivité requise | Fonctionne partout |
| **Scalabilité** | Goulot d'étranglement serveur | Monte en charge avec les appareils |
| **Taille du modèle** | Illimitée | Limitée par la mémoire de l'appareil |

Pour des applications comme la traduction en temps réel, la prédiction au clavier, le déverrouillage facial ou la surveillance de santé, l'Edge n'est pas seulement préférable — c'est la seule option viable.

## Techniques de compression de modèles

Les modèles de production sont trop volumineux pour les appareils Edge. Un modèle de classe GPT en précision FP32 consommerait des gigaoctets de mémoire. Les techniques de compression rendent le déploiement possible.

### Quantification

La quantification réduit la précision numérique des poids du modèle, du virgule flottante 32 bits (FP32) vers des représentations plus compactes :

- **FP16** — Demi-précision. Réduction mémoire de 2×, perte de précision minimale. Standard pour l'inférence GPU.
- **INT8** — Réduction mémoire de 4×. Le meilleur compromis pour la plupart des déploiements Edge.
- **INT4** — Réduction mémoire de 8×. Agressif, mais viable pour les LLMs avec un calibrage soigneux.

La quantification post-entraînement (PTQ) convertit un modèle déjà entraîné. L'entraînement sensible à la quantification (QAT) simule la faible précision pendant l'entraînement pour mieux préserver la précision.

<FunFact>Le Apple Neural Engine de l'iPhone 15 Pro traite les opérations INT8 à 35 TOPS (mille milliards d'opérations par seconde) — soit à peu près l'équivalent d'un GPU de data center milieu de gamme d'il y a seulement cinq ans.</FunFact>

### Élagage (Pruning)

L'élagage supprime les poids (ou des neurones entiers) qui contribuent le moins à la sortie du modèle. L'élagage non structuré met à zéro des poids individuels ; l'élagage structuré supprime des canaux entiers ou des têtes d'attention pour obtenir de véritables gains de vitesse sur le matériel.

Un modèle bien élagué peut supprimer 50 à 90 % des paramètres avec moins de 1 % de perte de précision — mais le modèle élagué doit être affiné ensuite pour récupérer ses performances.

### Distillation de connaissances

Un petit modèle « élève » est entraîné à imiter les sorties d'un grand modèle « enseignant ». L'élève apprend non seulement les bonnes réponses, mais aussi la distribution de probabilité complète de l'enseignant — y compris quelles mauvaises réponses sont « presque correctes ». Cette information souple se transfère étonnamment bien.

DistilBERT, par exemple, conserve 97 % de la précision de BERT avec 60 % de sa taille et 2× la vitesse.

## ONNX : le format universel

**Open Neural Network Exchange (ONNX)** est un format de modèle indépendant du framework. Entraînez dans PyTorch, exportez en ONNX, déployez partout — TensorFlow Lite, Core ML, ONNX Runtime ou le navigateur.

ONNX Runtime fournit une inférence optimisée sur CPUs, GPUs et NPUs avec une seule API. Il gère automatiquement la fusion d'opérateurs, la planification mémoire et l'accélération spécifique au matériel.

## Environnements d'exécution par plateforme

### TensorFlow Lite (Mobile / IoT)

L'environnement d'exécution de Google pour Android, iOS et microcontrôleurs. Les modèles TFLite sont des flat buffers — petits, rapides à charger, sans allocation mémoire dynamique. L'interpréteur fait moins de 1 Mo. Les API Delegate déchargent les calculs vers le GPU, le DSP ou le NPU.

### Core ML (écosystème Apple)

Le framework d'Apple route automatiquement les calculs entre CPU, GPU et le Neural Engine dédié. Si vous publiez une app iOS avec des fonctionnalités ML, Core ML est le chemin le plus simple. Il supporte le chiffrement de modèles pour la protection de la propriété intellectuelle.

### WebAssembly + ONNX Runtime Web

Exécutez des modèles ML dans le navigateur avec des performances quasi natives. ONNX Runtime Web compile les modèles en WebAssembly et utilise WebGL ou WebGPU pour l'accélération GPU. Pas de serveur, pas d'installation, aucune donnée ne quitte le navigateur.

Cas d'usage : suppression d'arrière-plan en temps réel dans les appels vidéo, modération de contenu sur la page, et OCR de documents dans le navigateur.

## LLMs embarqués sur l'appareil

Faire tourner des LLMs sur du matériel grand public semblait impossible il y a deux ans. Aujourd'hui, des modèles quantifiés de moins de 4 milliards de paramètres tournent confortablement sur les smartphones haut de gamme :

- **Gemma 2B/7B** (Google) — Optimisé pour le mobile via TFLite et MediaPipe
- **Phi-3 Mini** (Microsoft) — 3,8 milliards de paramètres, raisonnement solide pour sa taille
- **Llama 3.2 1B/3B** (Meta) — Conçu spécifiquement pour le déploiement embarqué
- **Gemini Nano** (Google) — Intégré dans Android 14+ via AICore

Ces modèles gèrent la synthèse, la traduction, les réponses intelligentes et la complétion de code sans aucun appel réseau.

## Accélérateurs matériels en périphérie

Les appareils modernes embarquent du silicium dédié à l'inférence de réseaux de neurones :

- **Apple Neural Engine** — NPU 16 cœurs dans les puces M-series et A-series. Intégration étroite avec Core ML.
- **Qualcomm Hexagon NPU** — Propulse l'IA Android sur les puces Snapdragon. Supporte nativement INT4.
- **Google Tensor TPU** — Puce personnalisée dans les téléphones Pixel pour la reconnaissance vocale, la traduction et les photos embarquées.
- **Intel Movidius** — VPU basse consommation pour les caméras IoT et les drones.

La tendance est claire : chaque grand fabricant de puces traite désormais l'accélération IA comme une fonctionnalité de premier plan, pas un ajout après coup.

## Applications concrètes de l'Edge AI

- **Prédiction au clavier** (GBoard, SwiftKey) — La prédiction du mot suivant fonctionne entièrement sur l'appareil avec des modèles personnalisés
- **Traduction en temps réel** (Google Translate hors ligne) — Traduction neuronale complète sans connectivité
- **Reconnaissance faciale** (Face ID) — Le système TrueDepth d'Apple traite les cartes faciales 3D sur le Neural Engine
- **Caméras intelligentes** — Analytique retail, surveillance de la faune et gestion du trafic en périphérie
- **Appareils auditifs** — Annulation de bruit et amélioration de la parole sur l'appareil avec une latence inférieure à la milliseconde
