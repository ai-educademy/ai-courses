# MLOps et Deployment

Un notebook Jupyter avec 94 % de précision n'est pas un produit. Dès que vous fermez le laptop, ce modèle ne sert plus personne. Le MLOps est la discipline qui comble le fossé entre l'expérimentation et la production — et c'est là que la plupart des équipes AI rencontrent le plus de difficultés.

## Pourquoi les Notebooks ne sont pas de la Production

Les notebooks encouragent de terribles habitudes d'ingénierie : état caché dû à l'exécution des cellules dans le désordre, chemins de fichiers codés en dur, aucune gestion de dépendances et zéro reproductibilité. Un modèle entraîné dans un notebook ne peut pas être ré-entraîné automatiquement lorsque de nouvelles données arrivent, ne peut pas être restauré lorsqu'il casse, et ne peut pas être surveillé contre la dégradation.

Le ML en production exige **du code versionné, des données versionnées, des modèles versionnés et des pipelines automatisés**. Tout le reste n'est qu'une démo.

## Le Cycle de Vie MLOps

Le cycle de vie MLOps est une boucle continue, pas un pipeline linéaire. Chaque étape alimente les autres, et la maturité de votre organisation détermine le degré d'automatisation de ces transitions.

Google définit trois niveaux de maturité MLOps :
- **Niveau 0** — Processus manuel, piloté par scripts. Les data scientists transmettent les modèles aux ingénieurs.
- **Niveau 1** — Pipelines d'entraînement automatisés. Le ré-entraînement se déclenche automatiquement sur de nouvelles données.
- **Niveau 2** — CI/CD complet pour le ML. Les changements de code, de données et de modèles passent tous par des pipelines automatisés.

Les étapes de la boucle :

1. **Ingestion et validation des données** — Collecter, nettoyer et valider les données entrantes par rapport aux schémas
2. **Feature engineering** — Transformer les données brutes en features prêtes pour le modèle via un feature store
3. **Entraînement** — Exécuter des jobs d'entraînement reproductibles avec des hyperparamètres suivis
4. **Évaluation** — Comparer les modèles candidats aux baselines sur des jeux de test réservés
5. **Registre** — Stocker les modèles approuvés avec métadonnées, lignage et statut d'approbation
6. **Deployment** — Servir le modèle derrière une API avec un rollout canary ou blue-green
7. **Monitoring** — Suivre la qualité des prédictions, la latence, le débit et le data drift
8. **Ré-entraînement** — Déclencher de nouvelles sessions d'entraînement lorsque les performances se dégradent

## Registres de Modèles

Un registre de modèles est le système de contrôle de versions pour vos modèles entraînés. Des outils comme **MLflow**, **Weights & Biases** et **Neptune** vous permettent de stocker les artefacts de modèles avec leurs métriques d'entraînement, hyperparamètres, versions de datasets et statut de promotion (staging → production → archivé).

Le registre répond à des questions critiques : *Quel modèle est actuellement en production ? Sur quelles données a-t-il été entraîné ? Qui l'a approuvé ? Quelles étaient ses métriques d'évaluation ?*

## CI/CD pour le Machine Learning

Le CI/CD traditionnel teste le code. Le CI/CD pour le ML doit tester trois choses : **le code, les données et les modèles**.

- **Tests de code** — Tests unitaires pour la logique de feature engineering, les transformations de données et les contrats API
- **Tests de données** — Validation de schéma, vérifications de distribution (les nouvelles données sont-elles dans les plages attendues ?) et vérifications de complétude (pas de colonnes manquantes)
- **Tests de modèles** — Seuils de performance sur les jeux de validation, benchmarks de latence, métriques de biais et comparaison avec le modèle actuellement en production

Une pull request qui modifie un pipeline de features devrait déclencher automatiquement un ré-entraînement et une évaluation. Si le nouveau modèle sous-performe par rapport à la baseline, la PR est bloquée.

## Conteneurisation pour le ML

Docker est incontournable pour le ML en production. Votre environnement d'entraînement et votre environnement de serving doivent être identiques — le problème du « ça marche sur ma machine » est amplifié quand les drivers GPU, les versions CUDA et les dépendances Python doivent tous correspondre.

Un Dockerfile ML typique superpose : image CUDA de base → dépendances Python → artefacts de modèle → framework de serving. Les builds multi-stage gardent l'image finale légère en séparant l'environnement de build de celui d'exécution.

Pratiques clés pour la conteneurisation ML :
- **Figer chaque dépendance** — Utilisez des versions exactes dans `requirements.txt`, pas des plages
- **Utiliser des images de base par digest** — `nvidia/cuda@sha256:...` au lieu de `:latest`
- **Séparer les images de training et de serving** — Les images de training ont besoin de compilateurs et d'outils de dev ; les images de serving n'ont besoin que du runtime
- **Scanner les vulnérabilités** — Les images ML héritent des CVE d'arbres de dépendances profonds

<FunFact>L'image de base NVIDIA CUDA seule fait plus de 3 Go. Les équipes de production passent régulièrement des jours à optimiser les images Docker pour réduire les temps de démarrage à froid et les coûts de registre de conteneurs.</FunFact>

## Infrastructure de Serving

La façon de servir un modèle dépend de vos exigences en latence et en débit :

| Outil | Idéal pour | Fonctionnalité clé |
|-------|-----------|-------------------|
| **TorchServe** | Modèles PyTorch | Batching intégré, versioning de modèles |
| **Triton Inference Server** | Multi-framework, GPU | Batching dynamique, exécution concurrente de modèles |
| **vLLM** | Large language models | PagedAttention, batching continu |
| **TensorFlow Serving** | Modèles TF/Keras | gRPC + REST, rechargement automatique des modèles |
| **BentoML** | Tout framework | API Pythonic, conteneurisation intégrée |

Pour le serving de LLM en particulier, **vLLM** est devenu le standard de l'industrie car son mécanisme PagedAttention réduit le gaspillage de mémoire GPU jusqu'à 90 % par rapport au serving naïf.

## Surveillance du Drift des Modèles

Un modèle précis au moment du deployment se dégradera avec le temps. Le **data drift** survient lorsque la distribution des entrées change (ex. : le comportement client évolue après une pandémie). Le **concept drift** survient lorsque la relation entre entrées et sorties change (ex. : les schémas de fraude évoluent).

Surveillez ces signaux :
- **Distribution des prédictions** — Les sorties dérivent-elles ? Plus de prédictions à haute confiance que d'habitude ?
- **Distributions des features** — Tests statistiques (KS test, PSI) comparant les données actuelles aux données d'entraînement
- **Retour de vérité terrain** — Lorsque les labels arrivent, calculez l'accuracy/F1 en direct et comparez à la baseline
- **Latence et taux d'erreurs** — La santé de l'infrastructure affecte indirectement la qualité du modèle

<ThinkAboutIt>Votre modèle de recommandation e-commerce a été entraîné avant la pandémie. Les habitudes d'achat post-pandémie sont complètement différentes. Comment détecteriez-vous ce drift automatiquement, et quelle serait votre stratégie de ré-entraînement ?</ThinkAboutIt>

## Feature Stores

Un feature store est un dépôt centralisé pour les définitions et valeurs de features, partagé entre l'entraînement et le serving. Des outils comme **Feast**, **Tecton** et **Hopsworks** résolvent le problème critique du **training-serving skew** — quand les features sont calculées différemment pendant l'entraînement et pendant l'inférence.

Les feature stores fournissent : un store offline (features par batch pour l'entraînement), un store online (features à faible latence pour le serving) et un registre (définitions des features, propriété, lignage).

## Inférence Batch vs Temps Réel

Chaque prédiction n'a pas besoin d'être en temps réel. La différence de coût est énorme :

- **Inférence batch** — Exécuter les prédictions selon un planning (horaire, quotidien). Peu coûteux, utilise des instances spot. Idéal pour les recommandations, le scoring de risque, la génération de rapports.
- **Inférence temps réel** — Prédictions en moins d'une seconde via API. Coûteux, nécessite des instances GPU toujours actives. Requis pour les chatbots, la détection de fraude, les systèmes autonomes.
- **Quasi temps réel (streaming)** — Traiter les événements à leur arrivée via Kafka ou Pub/Sub. Équilibre entre latence et coût pour des cas d'usage comme la personnalisation et la détection d'anomalies.

Un pattern courant consiste à pré-calculer les prédictions batch pour les 95 % de cas prévisibles, et router seulement les 5 % restants vers un endpoint temps réel.

Conseil d'optimisation des coûts : **auto-scalez les endpoints d'inférence** en fonction de la profondeur de la file d'attente, pas de l'utilisation CPU. Les workloads ML sont irréguliers — le scaling sur la profondeur de file évite à la fois le sur-provisionnement et les timeouts lors des pics de trafic.
