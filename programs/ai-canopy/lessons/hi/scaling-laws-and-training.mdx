# Scaling Laws और Training at Scale - बड़े Models क्यों ज़्यादा Smart होते हैं

Modern AI revolution सिर्फ algorithms की कहानी नहीं - यह **engineering और scaling** की कहानी है। Researchers ने discover किया कि model performance parameters, data और compute के across predictable power laws follow करती है, और उन laws को exploit करने के लिए बिल्कुल नई distributed training paradigms invent करनी पड़ीं।

## Kaplan Scaling Laws

2020 में OpenAI researchers (Kaplan et al.) ने एक landmark finding publish की: Transformer loss तीन variables के साथ **smooth power-law relationships** follow करता है:

- **N** - Model parameters की संख्या
- **D** - Dataset size (tokens)
- **C** - Compute budget (FLOPs)

Key result: `L(N) ∝ N^(-0.076)` - loss parameter count के power law के रूप में decrease होता है। Parameters double करो, और loss predictably drop होता है। ये curves छह orders of magnitude में कोई plateauing नहीं दिखाती थीं, suggest करते हुए कि scaling better performance का reliable path है।

Kaplan की recommendation: जब compute limited हो, तो **बड़ा model कम steps के लिए train** करना छोटे model को ज़्यादा steps के लिए train करने से efficient है। इसी ने GPT-3 को 175 billion parameters तक scale करने का decision guide किया।

## Chinchilla और Compute-Optimal Training

2022 में DeepMind के **Chinchilla paper** ने Kaplan की conclusions challenge कीं। उनकी key insight: parameters और training tokens के बीच optimal allocation लगभग **1:20** है - हर parameter के लिए, लगभग 20 tokens पर train करना चाहिए।

| Model | Parameters | Training Tokens | Ratio |
|-------|-----------|----------------|-------|
| GPT-3 | 175B | 300B | 1.7:1 |
| Chinchilla | 70B | 1.4T | 20:1 |
| LLaMA 2 70B | 70B | 2T | 28:1 |

Chinchilla (70B parameters) ने 4× ज़्यादा data पर train करके बहुत बड़े Gopher (280B) को **outperform** किया। सबक गहरा था: ज़्यादातर large models massively **over-parameterised और under-trained** थे।

## Parameter Count ही सब कुछ क्यों नहीं है

Model की capability architecture, data और training के interplay पर depend करती है:

- **Data quality** raw volume से ज़्यादा matter करती है। Curated, deduplicated, high-quality text पर trained models consistently larger लेकिन noisier datasets पर trained models को outperform करते हैं। Meta के LLaMA ने दिखाया कि carefully curated data पर trained 13B model कई benchmarks पर GPT-3 (175B) से match कर सकता है।
- **Architecture choices** - attention patterns, activation functions (SwiGLU vs ReLU), normalisation placement (pre-norm vs post-norm) - सब scaling curves shift करते हैं।
- **Tokenisation** effective data volume affect करता है। Better tokeniser per token ज़्यादा information compress करता है, parameter-to-token ratio बदलते हुए।

<FunFact>
GPT-4 train करने में अकेले compute पर $100 million से ज़्यादा खर्च हुआ अनुमानतः। एक single frontier model training run में consume होने वाली electricity एक छोटे शहर की annual consumption से ज़्यादा हो सकती है - scaling की environmental sustainability पर गंभीर सवाल उठाते हुए।
</FunFact>

## Distributed Training Strategies

कोई single GPU frontier model hold नहीं कर सकता। Training के लिए हज़ारों accelerators में काम distribute करना पड़ता है, कई complementary strategies use करके:

**Data Parallelism** - सबसे simple approach: पूरा model हर GPU पर replicate करो, training batch devices में split करो, और हर step के बाद gradients average करो। तब अच्छा काम करता है जब model single GPU की memory में fit हो।

**Model Parallelism (Tensor Parallelism)** - Individual layers GPUs में split करो। एक single matrix multiplication divide होता है ताकि हर GPU output का एक slice compute करे। High-bandwidth interconnects (NVLink, InfiniBand) ज़रूरी हैं क्योंकि GPUs को हर forward/backward pass में communicate करना पड़ता है।

**Pipeline Parallelism** - अलग-अलग layers अलग-अलग GPUs को assign करो, activations forward और gradients backward pipeline से pass करो। **Pipeline bubbles** (दूसरे stages का इंतज़ार करते समय idle time) introduce होते हैं, जिन्हें micro-batching से mitigate किया जाता है।

Modern systems तीनों combine करते हैं: node के अंदर tensor parallelism (fast NVLink), nodes के across pipeline parallelism, और node groups के across data parallelism।

<ThinkAboutIt>
Pipeline parallelism idle "bubble" time introduce करता है जब GPUs दूसरे stages से activations का इंतज़ार करते हैं। Pipeline जितना बड़ा, bubble overhead उतना ज़्यादा। आप इस waste को कैसे minimise करेंगे - और आपके solution से क्या trade-offs आएँगे?
</ThinkAboutIt>

## Mixed Precision और Gradient Accumulation

**Mixed precision training** ज़्यादातर computations के लिए half-precision floats (FP16 या BF16) use करता है जबकि weights की master copy FP32 में रखता है:

- Activations और gradients के लिए **2× memory reduction**
- Tensor cores वाले modern GPUs पर **2-8× throughput increase**
- BF16 (Brain Float 16) FP16 से preferred है क्योंकि इसकी wider exponent range FP16 training में common overflow issues avoid करती है

**Gradient accumulation** proportional memory cost बिना large batch sizes simulate करता है: multiple forward/backward passes run करो, gradients accumulate करो, और N micro-steps के बाद ही weights update करो।

## DeepSpeed और FSDP

Distributed training में दो frameworks dominate करते हैं:

- **DeepSpeed (Microsoft)** - ZeRO (Zero Redundancy Optimiser) stages 1-3 implement करता है, progressively optimiser states, gradients और parameters GPUs में shard करता है। ZeRO-3 trillion-parameter models train कर सकता है यह ensure करके कि कोई single GPU कुछ भी full copy hold न करे।
- **FSDP (PyTorch)** - Fully Sharded Data Parallel, PyTorch का native equivalent ZeRO-3 का। PyTorch ecosystem में directly integrated, researchers के लिए ज़्यादा accessible।

## Emergent Abilities और Diminishing Returns

जैसे-जैसे models scale होते हैं, कुछ capabilities gradually improve होने की बजाय **अचानक emerge** होती दिखती हैं:

- Multi-step arithmetic लगभग 100B parameters पर appear होना
- Chain-of-thought reasoning scale पर dramatically improve होना
- Few-shot learning certain thresholds से ऊपर ही reliable बनना

हालाँकि, recent research debate करती है कि emergence real है या **discontinuous evaluation metrics** का artefact। Continuous metrics (जैसे per-token log-probability) use करने पर performance अक्सर smoothly improve होती है - "sudden jump" measurement का illusion है, model का नहीं।

**Diminishing returns debate** भी intensify हो रही है। Compute में हर order of magnitude benchmarks में diminishing improvements yield करती है, कुछ researchers को argue करवाते हुए कि fundamental architectural innovations बिना scaling alone artificial general intelligence achieve नहीं करेगी।
