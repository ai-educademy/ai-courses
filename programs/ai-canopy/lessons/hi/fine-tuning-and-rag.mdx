# Fine-Tuning और RAG - Companies अपने Data के लिए AI कैसे Customise करती हैं

एक general-purpose LLM दुनिया के बारे में बहुत कुछ जानता है, लेकिन आपकी company के internal documents, proprietary data या domain-specific terminology के बारे में कुछ नहीं। इस lesson में हम उस gap को पाटने के दो मुख्य approaches cover करेंगे: **fine-tuning** और **retrieval-augmented generation (RAG)**।

## Generic Models काफ़ी क्यों नहीं हैं

सोचिए आप GPT-4 से अपनी company की specific return policy, internal product codes, या medical treatment protocols के बारे में customer queries का जवाब माँगें। वह नहीं दे सकता - यह information उसके training data में नहीं है।

Businesses को ऐसा AI चाहिए जो large model की **general intelligence** को उनके domain की **specific knowledge** के साथ combine करे। इसके दो रास्ते हैं, और सही रास्ता चुनना बहुत matter करता है।

<Callout>
एक common misconception यह है कि आप बस अपना data LLM को "upload" कर सकते हैं। असल में, model customise करने के लिए या तो उसके weights retrain करने (fine-tuning) या retrieval pipeline बनाने (RAG) की ज़रूरत है - दोनों engineering efforts हैं जिनमें careful planning चाहिए।
</Callout>

## Fine-Tuning: Model को नई चीज़ें सिखाना

Fine-tuning एक pre-trained model लेता है और उसे आपके dataset पर further train करता है। Model के weights update होते हैं ताकि आपके domain की language, style और knowledge reflect हो।

**कब fine-tune करें:**
- आपको model से consistently specific tone या writing style चाहिए।
- आपके domain में specialised terminology है जिसे base model ठीक से handle नहीं करता।
- आपको faster inference चाहिए (knowledge retrieve करने की बजाय "baked in" है)।

**Process:**
1. अपने domain में high-quality examples का dataset तैयार करो (आमतौर पर सैकड़ों से हज़ारों input-output pairs)।
2. Base model चुनो (जैसे Llama 3, Mistral)।
3. Low learning rate के साथ कुछ epochs train करो ताकि catastrophic forgetting से बचा जा सके।
4. Held-out test data पर evaluate करो।

<FunFact>
Bloomberg ने 363 billion tokens of financial data पर fine-tune करके BloombergGPT बनाया। इस model ने financial NLP tasks जैसे earnings calls की sentiment analysis और SEC filings में named entity recognition पर general-purpose models को significantly outperform किया।
</FunFact>

## LoRA और QLoRA: Efficient Fine-Tuning

Traditional fine-tuning **सभी** model parameters update करती है, जिसके लिए enormous GPU memory चाहिए। 70-billion parameter model को सिर्फ gradient storage के लिए सैकड़ों gigabytes चाहिए।

**LoRA (Low-Rank Adaptation)** इसे elegantly solve करता है:
- सभी weights update करने की बजाय, LoRA original model freeze करता है और हर layer में छोटे **trainable adapter matrices** add करता है।
- ये adapters आमतौर पर सिर्फ 0.1-1% extra parameters add करते हैं।
- Result लगभग identical performance है cost के fraction पर।

**QLoRA** frozen model को 4-bit precision में quantise करके memory requirements dramatically कम करता है। आप 48GB VRAM वाले single consumer GPU पर 70B parameter model fine-tune कर सकते हैं।

## RAG: Retrieval-Augmented Generation

RAG बिल्कुल अलग approach लेता है। Model बदलने की बजाय, आप उसे **query time पर external documents का access** देते हैं।

**RAG कैसे काम करता है:**
1. **Index** - आपके documents chunks में split होते हैं और embedding model से numerical vectors (embeddings) में convert होते हैं।
2. **Store** - ये vectors fast similarity search के लिए vector database में store होते हैं।
3. **Retrieve** - जब user सवाल पूछता है, query भी embed होती है, और सबसे relevant document chunks retrieve होते हैं।
4. **Generate** - Retrieved chunks prompt में context के रूप में insert होते हैं, और LLM आपके actual documents पर grounded answer generate करता है।

**कब RAG use करें:**
- आपका knowledge base frequently change होता है (नए documents, updated policies)।
- आपको source citations और traceability चाहिए।
- Fine-tuning की cost और complexity avoid करनी है।
- Factual accuracy critical है और hallucinations minimise करने हैं।

<ThinkAboutIt>
RAG model को query time पर relevant documents provide करता है, लेकिन final answer तो model ही generate करता है। आप कैसे verify करेंगे कि model का response actually retrieved documents को reflect करता है, न कि उनसे आगे hallucinate कर रहा है?
</ThinkAboutIt>

## Vector Databases

Vector databases high-dimensional embeddings store और search करने के लिए purpose-built हैं:

| Database | Type | Key Feature |
|----------|------|-------------|
| **Pinecone** | Managed cloud | Fully hosted, simple API |
| **Weaviate** | Open-source | Hybrid search (vector + keyword) |
| **pgvector** | PostgreSQL extension | अपने existing Postgres database का use करो |
| **Chroma** | Open-source | Lightweight, prototyping के लिए बढ़िया |
| **Qdrant** | Open-source | High performance, Rust-based |

Vector database चुनना आपके scale, existing infrastructure, और managed hosting vs self-hosted preference पर depend करता है।

## कब Fine-Tune करें vs कब RAG Use करें

| Consideration | Fine-Tuning | RAG |
|---------------|-------------|-----|
| **Data freshness** | Static (update के लिए retrain) | Dynamic (कभी भी documents update करो) |
| **Cost** | Higher upfront (training) | Lower upfront, ongoing retrieval cost |
| **Traceability** | Low (knowledge implicit है) | High (source documents cite कर सकते हैं) |
| **Style/behaviour** | Tone और format के लिए excellent | Prompt instructions तक limited |
| **Hallucination risk** | Moderate | Lower (documents में grounded) |

**Best production systems अक्सर दोनों combine करते हैं।** Domain-specific language और style के लिए fine-tune करो, फिर query time पर up-to-date factual content inject करने के लिए RAG use करो।

<FunFact>
UK की National Health Service ने RAG systems pilot किए हैं जो LLMs को clinical guidelines और drug interaction databases से connect करते हैं। Doctors natural language में सवाल पूछ सकते हैं और latest NICE guidelines पर grounded answers पा सकते हैं, page references के साथ।
</FunFact>

## Real-World Examples

- **Customer support** - एक retailer past support tickets पर tone के लिए model fine-tune करता है, फिर product catalogue और return policy के factual answers के लिए RAG use करता है।
- **Legal document analysis** - Law firms लाखों case documents search करने के लिए RAG use करती हैं, LLM हर नए case के लिए relevant precedents summarise करता है।
- **Medical Q&A** - Healthcare providers peer-reviewed literature के against RAG use करते हैं ताकि clinicians natural language में सवाल पूछकर evidence-based answers पा सकें।

## Production RAG Pipeline बनाना

एक robust RAG system के लिए सिर्फ vector database और LLM से ज़्यादा चाहिए:

- **Chunking strategy** - Documents कैसे split करते हैं, बहुत matter करता है। बहुत बड़े chunks में irrelevant content signal dilute करता है; बहुत छोटे में context खो जाता है। 10-20% overlap वाले overlapping chunks अक्सर अच्छे काम करते हैं।
- **Re-ranking** - Initial vector retrieval के बाद, cross-encoder re-ranker results को relevance से reorder कर सकता है, answer quality significantly improve करते हुए।
- **Evaluation** - Retrieval precision (क्या सही chunks मिले?) और generation faithfulness (क्या answer chunks reflect करता है?) अलग-अलग measure करो।
- **Hybrid search** - Vector similarity को traditional keyword search (BM25) के साथ combine करना उन cases पकड़ता है जहाँ semantic search alone exact terms या acronyms miss करता है।
