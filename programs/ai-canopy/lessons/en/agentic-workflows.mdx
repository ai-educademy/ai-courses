# Agentic Workflows — Building Multi-Agent AI Systems

A language model answers questions. An **agent** accomplishes goals. The difference is the loop: agents observe their environment, reason about what to do, take actions using tools, and repeat until the task is complete. This lesson covers the architecture, patterns, and failure modes of modern agentic AI systems.

## What Makes an AI Agent

An AI agent is defined by three capabilities operating in a continuous loop:

1. **Observe** — Perceive the current state: user instructions, tool outputs, environment data, previous actions
2. **Reason** — Decide what to do next: decompose the goal, plan steps, evaluate options
3. **Act** — Execute an action: call a tool, write code, send an API request, return a result

The critical distinction from simple prompting is **autonomy over multiple steps**. An agent doesn't just respond — it iterates. It might search the web, read the results, decide it needs more information, search again with a refined query, synthesise findings, and then present an answer.

## Tool Use and Function Calling

Without tools, an agent is limited to text generation. **Tool use** gives agents the ability to interact with the world:

- **Function calling** — The LLM outputs structured JSON specifying which function to call and with what arguments. The runtime executes the function and feeds results back as context.
- **API integration** — Agents can query databases, call REST endpoints, send emails, or manipulate files.
- **Code execution** — Some agents write and run code in a sandboxed environment, using the output to inform next steps.

Modern frameworks expose tools as typed function signatures with descriptions. The model learns which tool to invoke based on the task — choosing a calculator for maths, a search engine for current events, or a code interpreter for data analysis.

## Memory: Short-Term and Long-Term

Agents need memory to handle complex tasks that exceed a single context window:

**Short-term memory** is the conversation context — the sliding window of recent observations, actions, and results. It is limited by the model's context length and degrades for information far back in the window.

**Long-term memory** uses external storage, typically **vector databases**, to persist and retrieve information across sessions:
- Embed important observations and store them in a vector store
- At each step, query the store for relevant past experiences
- This gives agents access to knowledge accumulated over many interactions

The combination of short-term context and long-term retrieval mirrors human working memory and episodic memory — keeping immediate context readily available while drawing on a larger knowledge base when needed.

## Planning Strategies

How an agent decides **what to do next** determines its effectiveness:

**Chain-of-Thought (CoT)** — The model reasons step by step before acting. Simple but effective for straightforward problems.

**ReAct (Reason + Act)** — Interleaves explicit reasoning traces with tool actions in a structured format: `Thought → Action → Observation → Thought → ...` This makes the agent's decision process transparent and debuggable.

**Tree-of-Thought (ToT)** — The agent explores multiple reasoning paths simultaneously, evaluates each branch, and selects the most promising one. Useful for problems where the first approach might be wrong (e.g., puzzle-solving, complex debugging).

**Plan-then-Execute** — The agent first generates a complete plan, then executes each step. This separates high-level strategy from low-level execution, reducing the chance of losing track of the overall goal mid-execution.

## Multi-Agent Architectures

Complex tasks often benefit from multiple specialised agents working together:

**Supervisor pattern** — A central "manager" agent decomposes the task and delegates subtasks to specialised worker agents (a researcher, a coder, a reviewer). The supervisor synthesises their outputs and manages coordination.

**Peer-to-peer** — Agents communicate directly with each other without a central coordinator. Each agent has a defined role and passes results to the next agent in a chain or network.

**Hierarchical** — Multiple layers of supervision: a top-level planner delegates to mid-level coordinators, who manage specialist workers. This mirrors organisational structures and scales to very complex tasks.

## Modern Agent Frameworks

The ecosystem has matured rapidly from early experiments:

- **LangGraph** — Builds agents as stateful graphs with explicit control flow, cycles, and persistence. Favours predictability and debuggability over autonomy.
- **CrewAI** — Focuses on role-based multi-agent collaboration with defined tasks, tools, and delegation patterns.
- **AutoGen (Microsoft)** — Enables conversational multi-agent workflows where agents chat with each other to solve problems.
- **OpenAI Agents SDK** — First-party framework with built-in function calling, handoffs between agents, and guardrails.

## Agentic Coding

Perhaps the most transformative application: AI agents that write, test, and debug code autonomously:

- **GitHub Copilot** — In-editor agent mode that plans changes, edits multiple files, and runs terminal commands.
- **Cursor** — IDE built around an agent that understands entire codebases and makes multi-file edits.
- **Devin (Cognition)** — Autonomous software engineering agent that can handle full development tasks from issue to pull request.

These tools represent a shift from **autocomplete to autonomy** — the agent doesn't just suggest the next line, it understands the goal and works towards it across files, terminals, and browsers.

## When Agents Fail

Understanding failure modes is essential for building reliable agents:

- **Hallucination loops** — The agent fabricates a tool result, reasons from it, and digs deeper into fiction with each iteration.
- **Infinite recursion** — The agent repeatedly retries a failing action without changing its approach, burning tokens and compute.
- **Goal drift** — Over many steps, the agent gradually loses sight of the original objective, pursuing tangential subtasks.
- **Reward hacking** — The agent finds shortcuts that satisfy its success criteria without actually solving the problem.

## Production Agent Patterns

Building agents that work reliably in production requires defensive engineering:

- **Guardrails** — Input/output validation, content filtering, and boundary checks that prevent agents from taking dangerous actions.
- **Human-in-the-loop** — For high-stakes actions (sending emails, modifying databases, deploying code), require human approval before execution.
- **Structured outputs** — Force agents to produce machine-parseable responses (JSON schemas) rather than free-form text, reducing parsing errors.
- **Observability** — Log every thought, action, and tool call. Without comprehensive traces, debugging a failed 20-step agent run is nearly impossible.
- **Circuit breakers** — Set maximum step counts, token budgets, and time limits. If an agent exceeds these bounds, halt and escalate to a human.

<FunFact>
Browser agents — AI systems that navigate websites by clicking, typing, and reading screens — are among the hardest agentic applications to build reliably. Benchmarks like WebArena show that even the best agents complete only 15-30% of realistic web tasks successfully, compared to near-perfect human performance.
</FunFact>
