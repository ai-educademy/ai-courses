# Fine-Tuning మరియు RAG — కంపెనీలు వారి Data కోసం AI ని ఎలా Customise చేస్తాయి

General-purpose LLM ప్రపంచం గురించి చాలా తెలుసు, కానీ మీ company internal documents, proprietary data, లేదా domain-specific terminology గురించి ఏమీ తెలియదు. ఈ gap bridge చేయడానికి రెండు ప్రధాన approaches ఈ lesson cover చేస్తుంది: **fine-tuning** మరియు **retrieval-augmented generation (RAG)**.

## Generic Models ఎందుకు సరిపోవు

GPT-4 ని మీ company specific return policy, internal product codes, లేదా medical treatment protocols గురించి customer queries answer చేయమని అడగడం imagine చేయండి. అది చేయలేదు — ఆ information దాని training data లో లేదు.

Businesses కి large model యొక్క **general intelligence** మరియు వారి domain **specific knowledge** combine చేసే AI అవసరం. అక్కడికి చేరుకోవడానికి రెండు paths ఉన్నాయి, సరైనది choose చేయడం చాలా ముఖ్యం.

<Callout>
ఒక common misconception: మీ data ని LLM కి "upload" చేస్తే చాలు అని. నిజంలో, model customise చేయడానికి weights retrain చేయడం (fine-tuning) లేదా retrieval pipeline build చేయడం (RAG) అవసరం — రెండూ careful planning demand చేసే engineering efforts.
</Callout>

## Fine-Tuning: Model కి కొత్త Tricks నేర్పించడం

Fine-tuning pre-trained model తీసుకుని మీ own dataset పై training continue చేస్తుంది. Model weights మీ domain language, style, మరియు knowledge reflect చేయడానికి update అవుతాయి.

**Fine-tune ఎప్పుడు చేయాలి:**
- Model consistently specific tone లేదా writing style adopt చేయాలంటే.
- మీ domain లో base model poorly handle చేసే specialised terminology ఉంటే.
- Faster inference కావాలంటే (knowledge retrieve చేయకుండా "baked in" ఉంటుంది).

**Process:**
1. మీ domain లో high-quality examples dataset prepare చేయడం (సాధారణంగా hundreds నుండి thousands input-output pairs).
2. Base model choose చేయడం (ఉదా., Llama 3, Mistral).
3. Catastrophic forgetting avoid చేయడానికి low learning rate తో few epochs train చేయడం.
4. Held-out test data పై evaluate చేయడం.

<FunFact>
Bloomberg 363 billion tokens financial data పై fine-tune చేసి BloombergGPT build చేసింది. ఫలితంగా వచ్చిన model earnings calls sentiment analysis మరియు SEC filings లో named entity recognition వంటి financial NLP tasks లో general-purpose models ని significantly outperform చేసింది.
</FunFact>

## LoRA మరియు QLoRA: Efficient Fine-Tuning

Traditional fine-tuning **అన్ని** model parameters update చేస్తుంది, enormous GPU memory అవసరం. 70-billion parameter model కి gradient storage కోసం మాత్రమే hundreds of gigabytes కావాలి.

**LoRA (Low-Rank Adaptation)** దీన్ని elegantly solve చేస్తుంది:
- అన్ని weights update చేయడం బదులు, LoRA original model freeze చేసి ప్రతి layer కి చిన్న **trainable adapter matrices** add చేస్తుంది.
- ఈ adapters సాధారణంగా 0.1–1% extra parameters మాత్రమే add చేస్తాయి.
- Cost fraction లో దాదాపు identical performance.

**QLoRA** ఇంకా ముందుకు వెళ్ళి frozen model ని 4-bit precision కి quantise చేస్తుంది, memory requirements dramatically reduce చేస్తుంది. 48GB VRAM ఉన్న single consumer GPU పై 70B parameter model fine-tune చేయవచ్చు.

## RAG: Retrieval-Augmented Generation

RAG పూర్తిగా different approach తీసుకుంటుంది. Model మార్చడం బదులు, **query time లో external documents access** ఇస్తారు.

**RAG ఎలా work చేస్తుంది:**
1. **Index** — Documents ని chunks గా split చేసి embedding model ఉపయోగించి numerical vectors (embeddings) గా convert చేయడం.
2. **Store** — Fast similarity search కోసం ఈ vectors ని vector database లో store చేయడం.
3. **Retrieve** — User question అడిగినప్పుడు, query కూడా embed అవుతుంది, most relevant document chunks retrieve అవుతాయి.
4. **Generate** — Retrieved chunks prompt లో context గా insert చేయబడతాయి, LLM మీ actual documents లో grounded answer generate చేస్తుంది.

**RAG ఎప్పుడు use చేయాలి:**
- Knowledge base frequently change అవుతుంటే (new documents, updated policies).
- Source citations మరియు traceability అవసరమైతే.
- Fine-tuning cost మరియు complexity avoid చేయాలంటే.
- Factual accuracy critical మరియు hallucinations minimise చేయాలంటే.

<ThinkAboutIt>
RAG model కి query time లో relevant documents provide చేస్తుంది, కానీ final answer model generate చేస్తుంది. Model response retrieved documents లో చెప్పింది reflect చేస్తుందా, వాటి beyond hallucinate చేస్తుందా — ఇది ఎలా verify చేయవచ్చు?
</ThinkAboutIt>

## Vector Databases

Vector databases high-dimensional embeddings store మరియు search చేయడానికి purpose-built:

| Database | Type | Key Feature |
|----------|------|-------------|
| **Pinecone** | Managed cloud | Fully hosted, simple API |
| **Weaviate** | Open-source | Hybrid search (vector + keyword) |
| **pgvector** | PostgreSQL extension | Existing Postgres database use చేయవచ్చు |
| **Chroma** | Open-source | Lightweight, prototyping కి great |
| **Qdrant** | Open-source | High performance, Rust-based |

Vector database choose చేయడం మీ scale, existing infrastructure, managed hosting లేదా self-hosted preference పై depend అవుతుంది.

## Fine-Tune vs RAG ఎప్పుడు

| Consideration | Fine-Tuning | RAG |
|---------------|-------------|-----|
| **Data freshness** | Static (update కోసం retrain) | Dynamic (ఎప్పుడైనా documents update) |
| **Cost** | Higher upfront (training) | Lower upfront, ongoing retrieval cost |
| **Traceability** | Low (knowledge implicit) | High (source documents cite చేయవచ్చు) |
| **Style/behaviour** | Tone మరియు format కి excellent | Prompt instructions కి limited |
| **Hallucination risk** | Moderate | Lower (documents లో grounded) |

**Best production systems తరచుగా రెండూ combine చేస్తాయి.** Domain-specific language మరియు style కోసం fine-tune చేసి, query time లో up-to-date factual content inject చేయడానికి RAG use చేయడం.

<FunFact>
UK National Health Service LLMs ని clinical guidelines మరియు drug interaction databases కి connect చేసే RAG systems pilot చేసింది. Doctors natural language questions అడిగి latest NICE guidelines లో grounded answers receive చేయగలరు, page references తో సహా.
</FunFact>

## Real-World Examples

- **Customer support** — Retailer past support tickets పై tone కోసం model fine-tune చేసి, factual answers కోసం product catalogue మరియు return policy against RAG use చేస్తుంది.
- **Legal document analysis** — Law firms millions of case documents search చేయడానికి RAG use చేస్తాయి, ప్రతి new case కి relevant precedents LLM summarise చేస్తుంది.
- **Medical Q&A** — Healthcare providers peer-reviewed literature against RAG use చేసి clinicians natural language లో questions అడిగి citations తో evidence-based answers receive చేయగలరు.

## Production RAG Pipeline Build చేయడం

Robust RAG system కి vector database మరియు LLM కంటే ఎక్కువ అవసరం. Key considerations:

- **Chunking strategy** — Documents ఎలా split చేస్తారో చాలా ముఖ్యం. Too large అయితే irrelevant content signal dilute చేస్తుంది; too small అయితే context lose అవుతుంది. 10–20% overlap తో overlapping chunks తరచుగా బాగా work చేస్తాయి.
- **Re-ranking** — Initial vector retrieval తర్వాత, cross-encoder re-ranker results ని relevance ప్రకారం reorder చేయగలదు, answer quality significantly improve చేస్తుంది.
- **Evaluation** — Retrieval precision (right chunks find అయ్యాయా?) మరియు generation faithfulness (answer chunks reflect చేస్తుందా?) separately measure చేయడం.
- **Hybrid search** — Vector similarity తో traditional keyword search (BM25) combine చేయడం — semantic search alone miss చేసే exact terms లేదా acronyms catch చేస్తుంది.
