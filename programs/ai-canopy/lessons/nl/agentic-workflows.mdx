# Agentic Workflows - Multi-Agent AI-systemen Bouwen

Een taalmodel beantwoordt vragen. Een **agent** bereikt doelen. Het verschil zit in de lus: agents observeren hun omgeving, redeneren over wat ze moeten doen, ondernemen acties met tools, en herhalen dit tot de taak voltooid is. In deze les behandelen we de architectuur, patronen en faalwijzen van moderne agentische AI-systemen.

## Wat Maakt een AI-agent

Een AI-agent wordt gedefinieerd door drie capaciteiten die in een continue lus opereren:

1. **Waarnemen** - De huidige toestand observeren: gebruikersinstructies, tool-output, omgevingsdata, eerdere acties
2. **Redeneren** - Beslissen wat de volgende stap is: het doel ontleden, stappen plannen, opties evalueren
3. **Handelen** - Een actie uitvoeren: een tool aanroepen, code schrijven, een API-verzoek versturen, een resultaat retourneren

Het cruciale onderscheid met eenvoudig prompten is **autonomie over meerdere stappen**. Een agent reageert niet alleen - hij itereert. Hij zoekt bijvoorbeeld op het web, leest de resultaten, besluit dat hij meer informatie nodig heeft, zoekt opnieuw met een verfijnde query, synthetiseert bevindingen, en presenteert dan een antwoord.

## Tool Use en Function Calling

Zonder tools is een agent beperkt tot tekstgeneratie. **Tool use** geeft agents het vermogen om met de wereld te interacteren:

- **Function calling** - Het LLM genereert gestructureerde JSON die specificeert welke functie aangeroepen moet worden en met welke argumenten. De runtime voert de functie uit en stuurt de resultaten terug als context.
- **API-integratie** - Agents kunnen databases bevragen, REST-endpoints aanroepen, e-mails versturen of bestanden bewerken.
- **Code-uitvoering** - Sommige agents schrijven en draaien code in een sandboxed omgeving, en gebruiken de output om volgende stappen te bepalen.

Moderne frameworks bieden tools aan als getypeerde functiesignaturen met beschrijvingen. Het model leert welke tool het moet inzetten op basis van de taak - een rekenmachine voor wiskunde, een zoekmachine voor actualiteiten, of een code-interpreter voor data-analyse.

## Geheugen: Korte en Lange Termijn

Agents hebben geheugen nodig voor complexe taken die één context window overstijgen:

**Kortetermijngeheugen** is de gesprekscontext - het schuivende venster van recente observaties, acties en resultaten. Dit is beperkt door de contextlengte van het model en degradeert voor informatie ver terug in het venster.

**Langetermijngeheugen** maakt gebruik van externe opslag, doorgaans **vectordatabases**, om informatie over sessies heen op te slaan en op te halen:
- Sla belangrijke observaties op als embeddings in een vectorstore
- Bevraag bij elke stap de store voor relevante eerdere ervaringen
- Dit geeft agents toegang tot kennis opgebouwd over vele interacties

De combinatie van kortetermijncontext en langetermijnopvraging weerspiegelt menselijk werkgeheugen en episodisch geheugen - directe context blijft beschikbaar terwijl een grotere kennisbasis geraadpleegd wordt wanneer nodig.

## Planningsstrategieën

Hoe een agent beslist **wat de volgende stap is** bepaalt zijn effectiviteit:

**Chain-of-Thought (CoT)** - Het model redeneert stap voor stap voordat het handelt. Eenvoudig maar effectief voor rechttoe-rechtaan problemen.

**ReAct (Reason + Act)** - Verweft expliciete redeneringstappen met tool-acties in een gestructureerd format: `Thought → Action → Observation → Thought → ...` Dit maakt het beslissingsproces van de agent transparant en debugbaar.

**Tree-of-Thought (ToT)** - De agent verkent meerdere redeneerpaden tegelijkertijd, evalueert elke tak, en selecteert de meest veelbelovende. Nuttig voor problemen waarbij de eerste aanpak fout kan zijn (bijv. puzzels, complexe debugging).

**Plan-then-Execute** - De agent genereert eerst een volledig plan en voert vervolgens elke stap uit. Dit scheidt de strategie op hoog niveau van de uitvoering op laag niveau, waardoor het risico kleiner is dat het overkoepelende doel halverwege uit het oog wordt verloren.

## Multi-Agent-architecturen

Complexe taken profiteren vaak van meerdere gespecialiseerde agents die samenwerken:

**Supervisor-patroon** - Een centrale "manager"-agent ontleedt de taak en delegeert deeltaken aan gespecialiseerde worker-agents (een onderzoeker, een programmeur, een reviewer). De supervisor synthetiseert hun output en coördineert het geheel.

**Peer-to-peer** - Agents communiceren rechtstreeks met elkaar zonder centrale coördinator. Elke agent heeft een gedefinieerde rol en geeft resultaten door aan de volgende agent in een keten of netwerk.

**Hiërarchisch** - Meerdere lagen van supervisie: een planner op topniveau delegeert aan coördinatoren op middenniveau, die specialistische workers aansturen. Dit weerspiegelt organisatiestructuren en schaalt naar zeer complexe taken.

## Moderne Agent-frameworks

Het ecosysteem is snel gerijpt sinds de vroege experimenten:

- **LangGraph** - Bouwt agents als stateful graphs met expliciete control flow, cycli en persistentie. Geeft voorrang aan voorspelbaarheid en debugbaarheid boven autonomie.
- **CrewAI** - Focust op rolgebaseerde multi-agent-samenwerking met gedefinieerde taken, tools en delegatiepatronen.
- **AutoGen (Microsoft)** - Maakt conversationele multi-agent-workflows mogelijk waarbij agents met elkaar chatten om problemen op te lossen.
- **OpenAI Agents SDK** - First-party framework met ingebouwde function calling, handoffs tussen agents en guardrails.

## Agentic Coding

Misschien wel de meest transformatieve toepassing: AI-agents die autonoom code schrijven, testen en debuggen:

- **GitHub Copilot** - In-editor agent-modus die wijzigingen plant, meerdere bestanden bewerkt en terminalcommando's uitvoert.
- **Cursor** - IDE gebouwd rond een agent die volledige codebases begrijpt en multi-file bewerkingen uitvoert.
- **Devin (Cognition)** - Autonome software engineering-agent die volledige ontwikkeltaken afhandelt, van issue tot pull request.

Deze tools vertegenwoordigen een verschuiving van **autocomplete naar autonomie** - de agent suggereert niet alleen de volgende regel, maar begrijpt het doel en werkt ernaartoe over bestanden, terminals en browsers heen.

## Wanneer Agents Falen

Inzicht in faalwijzen is essentieel voor het bouwen van betrouwbare agents:

- **Hallucinatielussen** - De agent verzint een tool-resultaat, redeneert op basis daarvan, en graaft zich bij elke iteratie dieper in fictie.
- **Oneindige recursie** - De agent probeert herhaaldelijk een falende actie opnieuw zonder zijn aanpak te wijzigen, terwijl tokens en rekenkracht verspild worden.
- **Doeldrift** - Over vele stappen verliest de agent geleidelijk het oorspronkelijke doel uit het oog en jaagt tangentiële deeltaken na.
- **Reward hacking** - De agent vindt sluiproutes die aan zijn succescriteria voldoen zonder het probleem daadwerkelijk op te lossen.

## Productie-agent-patronen

Het bouwen van agents die betrouwbaar werken in productie vereist defensieve engineering:

- **Guardrails** - Input/output-validatie, contentfiltering en grenscontroles die voorkomen dat agents gevaarlijke acties ondernemen.
- **Human-in-the-loop** - Voor acties met hoge impact (e-mails versturen, databases wijzigen, code deployen) is menselijke goedkeuring vereist vóór uitvoering.
- **Gestructureerde outputs** - Dwing agents om machine-parseerbare antwoorden te produceren (JSON-schema's) in plaats van vrije tekst, waardoor parseerfouten verminderen.
- **Observability** - Log elke gedachte, actie en tool-aanroep. Zonder uitgebreide traces is het debuggen van een mislukte 20-staps agent-run vrijwel onmogelijk.
- **Circuit breakers** - Stel maximale staptellers, tokenbudgetten en tijdslimieten in. Als een agent deze grenzen overschrijdt, stop dan en escaleer naar een mens.
