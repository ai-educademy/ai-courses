## Hashing क्यों ज़रूरी है

जब एक recommendation engine को ये चेक करना हो कि आपने कोई फ़िल्म पहले देखी है या नहीं - लाखों फ़िल्मों के catalogue में - तो वो एक-एक करके list scan नहीं कर सकता। एक **hash table** ये सवाल O(1) average time में answer कर देता है।

Hash tables production software में सबसे ज़्यादा इस्तेमाल होने वाला data structure है, और AI systems इन्हें feature lookups, caching, और deduplication के लिए काफ़ी use करते हैं।

## Hashing कैसे काम करता है

एक **hash function** कोई input लेता है (string, number, या object) और एक integer return करता है - यही **hash code** है। इस code को modulo के ज़रिए **bucket index** में map किया जाता है:

```
index = hashFunction(key) % numberOfBuckets
```

एक अच्छे hash function की खासियत:
- **Deterministic** - एक ही input हमेशा same output देगा।
- **Uniform** - keys को buckets में बराबर distribute करे।
- **Fast** - key size के हिसाब से O(1) में compute हो।

### उदाहरण

```
function simpleHash(key, size):
    total = 0
    for char in key:
        total += charCode(char)
    return total % size
```

ये जानबूझकर simple रखा गया है - production में MurmurHash3 या xxHash जैसे hash functions bit-mixing use करते हैं जो बहुत बेहतर distribution देते हैं।

## Collision Handling

Collisions अटल हैं (pigeonhole principle)। दो मुख्य strategies:

### Chaining

हर bucket में एक linked list (या छोटा array) होता है। Collide करने वाली keys को list में append कर दिया जाता है।

```
function insert(table, key, value):
    idx = hash(key) % table.size
    for entry in table[idx]:
        if entry.key == key:
            entry.value = value
            return
    table[idx].append((key, value))
```

**फ़ायदे:** Simple है, moderate load पर अच्छा काम करता है।
**नुकसान:** Pointer chasing की वजह से cache-unfriendly है।

### Open Addressing

सभी entries सीधे array में ही रहती हैं। Collision होने पर, अगली खाली slot **probe** की जाती है।

- **Linear probing:** index+1, index+2, … चेक करो।
- **Quadratic probing:** index+1², index+2², … चेक करो।
- **Double hashing:** Step size तय करने के लिए दूसरा hash function use करो।

**फ़ायदे:** बेहतर cache performance, कोई extra pointers नहीं।
**नुकसान:** Clustering से performance गिर सकती है; deletion मुश्किल होती है (tombstones चाहिए)।

<FunFact>Python dictionaries open addressing use करती हैं एक custom probing sequence के साथ। CPython की dict implementation इतनी optimised है कि modern hardware पर 5 करोड़ से ज़्यादा lookups per second handle कर सकती है।</FunFact>

## Load Factor और Rehashing

**Load factor** α = entries की संख्या / buckets की संख्या। जैसे-जैसे α बढ़ता है, collisions बढ़ती हैं और performance गिरती है।

| Load Factor | Expected Lookups (chaining) |
|------------|----------------------------|
| 0.5 | ~1.25 |
| 0.75 | ~1.375 |
| 1.0 | ~1.5 |
| 2.0 | ~2.0 |

ज़्यादातर implementations α एक threshold (आमतौर पर 0.75) से ज़्यादा होने पर **rehash** करते हैं: एक बड़ा array allocate करो, हर key का index दोबारा compute करो, और reinsert करो।

Rehashing O(n) है लेकिन बहुत कम होती है, इसलिए insertion की **amortised** cost O(1) ही रहती है।

<ThinkAboutIt>Rehashing table size double करता है और हर element को दोबारा insert करता है। अगर आपका AI system एक live request stream handle कर रहा है, तो ये pause एक latency spike पैदा कर सकता है। आप rehashing cost को एक बार में करने की बजाय कई insertions में कैसे बाँट सकते हैं?</ThinkAboutIt>

## AI में असली Applications

| Use Case | Hash Tables कैसे मदद करते हैं |
|----------|-------------------------------|
| **Embedding tables** | Token IDs को O(1) में dense vector representations से map करें |
| **Feature hashing** | High-cardinality categorical features को fixed-size vector में hash करें ("hashing trick") |
| **Deduplication** | Content fingerprints hash करके duplicate documents या images detect करें |
| **Caching layers** | Input hash से keyed precomputed model predictions store करें |
| **Count-min sketch** | Multiple hash functions से streaming data में approximate frequency counts निकालें |

## Bloom Filters: Probabilistic Membership

एक **Bloom filter** ये बताता है कि "क्या ये element set में है?" इसके दो जवाब:
- **पक्का नहीं** - element निश्चित रूप से नहीं है।
- **शायद हाँ** - element शायद है (छोटा false-positive rate)।

ये _k_ hash functions और एक bit array use करता है। Insert करने पर _k_ bits set होते हैं; lookup पर चेक होता है कि सभी _k_ bits set हैं या नहीं।

```
function bloomInsert(filter, item):
    for i in 1..k:
        idx = hashFunctions[i](item) % filter.size
        filter.bits[idx] = 1

function bloomContains(filter, item):
    for i in 1..k:
        idx = hashFunctions[i](item) % filter.size
        if filter.bits[idx] == 0: return false
    return true   // probably present
```

**AI में use:** ये check करना कि कोई URL पहले crawl हो चुका है, spam filtering, और caching layers में redundant database queries से बचना।

<FunFact>Google Chrome एक Bloom filter use करता है जिसमें लाखों known malicious URLs हैं। कोई भी page load करने से पहले, आपका browser filter check करता है - अगर वो "शायद हाँ" कहे, तो Chrome confirm करने के लिए full list fetch करता है।</FunFact>

## Sets vs Maps: कब क्या Use करें

| Structure | क्या Store करता है | कब Use करें |
|-----------|-------------------|-------------|
| **Set** | सिर्फ़ Keys | Membership checks चाहिए - "क्या मैंने ये पहले देखा है?" |
| **Map** | Key–value pairs | हर key के साथ data associate करना है - "इस key की value क्या है?" |

दोनों hash tables पर बने हैं और O(1) average operations देते हैं। **Set** चुनिए deduplication, visited tracking, और uniqueness constraints के लिए। **Map** चुनिए lookups, caches, और configuration stores के लिए।

<ThinkAboutIt>एक AI content moderation system को incoming messages को 1 करोड़ banned phrases की blocklist से check करना है। क्या आप hash set use करेंगे, Bloom filter, या दोनों? Memory, speed, और accuracy में क्या trade-offs हैं?</ThinkAboutIt>

## मुख्य बातें

1. Hash tables O(1) average-case lookups देते हैं - caching, indexing, और feature engineering की रीढ़।
2. Collisions को chaining या open addressing से manage किया जाता है; load factor बताता है कब rehash करना है।
3. Bloom filters थोड़ा false-positive rate accept करके membership queries में बहुत ज़्यादा memory बचाते हैं।
4. Hashing internals समझने से आप हर AI workload के लिए सही tool - set, map, या probabilistic filter - चुन सकते हैं।
