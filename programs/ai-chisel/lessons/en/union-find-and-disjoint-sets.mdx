## The Connectivity Problem

Given a network of nodes, you constantly receive two types of queries: *connect node A to node B* and *are A and B connected?* Doing this with BFS/DFS means traversing the graph for every query — far too slow for millions of operations.

**Union-Find** answers both in near-constant time.

## Naive Approach — Quick Find

Store a component ID for each node. To check connectivity, compare IDs in O(1). But merging two components forces you to update every member of one group — O(n) per union.

```
IDs:  [0, 0, 0, 3, 3]
       └─group 0─┘ └group 3┘
union(0, 3) → update all 3s to 0 → O(n)
```

We need something better.

## The Union-Find Data Structure

Each node points to a **parent**. The root of each tree is the component representative. Two nodes are connected if and only if they share the same root.

```python
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
```

## Find with Path Compression

Walking up parent pointers can be slow in a deep tree. **Path compression** flattens the tree by pointing every visited node directly to the root.

```python
def find(self, x):
    if self.parent[x] != x:
        self.parent[x] = self.find(self.parent[x])
    return self.parent[x]
```

Before and after finding node 4:

```
Before:  0←1←2←3←4     After:  0←1
                                0←2
                                0←3
                                0←4
```

## Union by Rank

Always attach the shorter tree under the taller one. This keeps tree height logarithmic and prevents degenerate chains.

```python
def union(self, x, y):
    rx, ry = self.find(x), self.find(y)
    if rx == ry:
        return False
    if self.rank[rx] < self.rank[ry]:
        rx, ry = ry, rx
    self.parent[ry] = rx
    if self.rank[rx] == self.rank[ry]:
        self.rank[rx] += 1
    return True
```

## Kruskal's MST Algorithm

Union-Find powers **Kruskal's algorithm** for Minimum Spanning Trees:

1. Sort all edges by weight
2. For each edge (u, v), if `find(u) ≠ find(v)`, add it and `union(u, v)`
3. Stop when you have n − 1 edges

The Union-Find prevents cycles by rejecting edges whose endpoints already share a root. Total time: **O(E log E)** dominated by sorting.

## Classic Problems

### Number of Connected Components

Initialise Union-Find with n nodes. Process each edge with `union`. Count distinct roots at the end.

### Accounts Merge (LeetCode 721)

Each account has emails. If two accounts share an email, they belong to the same person. Union accounts by shared emails, then group and sort.

### Redundant Connection (LeetCode 684)

Given a tree plus one extra edge, find the redundant edge. Process edges in order — the first edge where `union` returns `False` (already connected) is the answer.

<FunFact>The proof that Union-Find with both optimisations runs in O(α(n)) was published by Tarjan in 1975 and is considered one of the most elegant amortised analyses in computer science.</FunFact>

## When to Choose Union-Find vs BFS/DFS

| Scenario | Best choice |
|----------|-------------|
| Repeated connectivity queries | Union-Find |
| Single traversal / shortest path | BFS/DFS |
| Incremental edge additions | Union-Find |
| Edge removals (dynamic graph) | BFS/DFS (or offline UF) |
| Kruskal's MST | Union-Find |
| Topological sort / cycle in directed graph | DFS |

Union-Find shines when you process edges *incrementally* and need fast connectivity checks. If you only traverse once or need path information, stick with BFS/DFS.
