## Le problème de connectivité

Étant donné un réseau de nœuds, vous recevez constamment deux types de requêtes : *connecter le nœud A au nœud B* et *A et B sont-ils connectés ?* Utiliser BFS/DFS signifierait parcourir le graphe pour chaque requête — bien trop lent pour des millions d'opérations.

**Union-Find** répond aux deux en temps quasi-constant.

## Approche naïve — Quick Find

On stocke un identifiant de composante pour chaque nœud. Pour vérifier la connectivité, on compare les identifiants en O(1). Mais fusionner deux composantes oblige à mettre à jour chaque membre d'un groupe — O(n) par union.

```
IDs:  [0, 0, 0, 3, 3]
       └─group 0─┘ └group 3┘
union(0, 3) → update all 3s to 0 → O(n)
```

Il nous faut quelque chose de mieux.

## La structure de données Union-Find

Chaque nœud pointe vers un **parent**. La racine de chaque arbre est le représentant de la composante. Deux nœuds sont connectés si et seulement s'ils partagent la même racine.

```python
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
```

## Find avec Path Compression

Remonter les pointeurs parents peut être lent dans un arbre profond. La **path compression** aplatit l'arbre en faisant pointer chaque nœud visité directement vers la racine.

```python
def find(self, x):
    if self.parent[x] != x:
        self.parent[x] = self.find(self.parent[x])
    return self.parent[x]
```

Avant et après la recherche du nœud 4 :

```
Before:  0←1←2←3←4     After:  0←1
                                0←2
                                0←3
                                0←4
```

## Union by Rank

On attache toujours l'arbre le plus court sous le plus grand. Cela maintient la hauteur logarithmique et empêche les chaînes dégénérées.

```python
def union(self, x, y):
    rx, ry = self.find(x), self.find(y)
    if rx == ry:
        return False
    if self.rank[rx] < self.rank[ry]:
        rx, ry = ry, rx
    self.parent[ry] = rx
    if self.rank[rx] == self.rank[ry]:
        self.rank[rx] += 1
    return True
```

## Algorithme MST de Kruskal

Union-Find est au cœur de l'**algorithme de Kruskal** pour les arbres couvrants minimaux :

1. Trier toutes les arêtes par poids
2. Pour chaque arête (u, v), si `find(u) ≠ find(v)`, l'ajouter et faire `union(u, v)`
3. S'arrêter lorsqu'on a n − 1 arêtes

Union-Find empêche les cycles en rejetant les arêtes dont les extrémités partagent déjà une racine. Temps total : **O(E log E)** dominé par le tri.

## Problèmes classiques

### Nombre de composantes connexes

Initialiser Union-Find avec n nœuds. Traiter chaque arête avec `union`. Compter les racines distinctes à la fin.

### Accounts Merge (LeetCode 721)

Chaque compte possède des emails. Si deux comptes partagent un email, ils appartiennent à la même personne. Unir les comptes par emails communs, puis grouper et trier.

### Redundant Connection (LeetCode 684)

Étant donné un arbre plus une arête supplémentaire, trouver l'arête redondante. Traiter les arêtes dans l'ordre — la première arête où `union` retourne `False` (déjà connectés) est la réponse.

<FunFact>La preuve que Union-Find avec les deux optimisations s'exécute en O(α(n)) a été publiée par Tarjan en 1975 et est considérée comme l'une des analyses amorties les plus élégantes en informatique.</FunFact>

## Quand choisir Union-Find vs BFS/DFS

| Scénario | Meilleur choix |
|----------|---------------|
| Requêtes de connectivité répétées | Union-Find |
| Parcours unique / plus court chemin | BFS/DFS |
| Ajouts d'arêtes incrémentaux | Union-Find |
| Suppressions d'arêtes (graphe dynamique) | BFS/DFS (ou UF offline) |
| MST de Kruskal | Union-Find |
| Tri topologique / cycle dans un graphe orienté | DFS |

Union-Find excelle lorsque vous traitez les arêtes *de manière incrémentale* et avez besoin de vérifications de connectivité rapides. Si vous ne parcourez qu'une seule fois ou avez besoin d'informations de chemin, préférez BFS/DFS.
