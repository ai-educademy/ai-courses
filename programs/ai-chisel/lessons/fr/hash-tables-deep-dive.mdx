## Pourquoi le hachage est important

Quand un moteur de recommandation doit vérifier si vous avez déjà vu un film — parmi des millions dans son catalogue — il ne peut pas se permettre de parcourir une liste. Une **table de hachage** répond à cette question en temps moyen O(1).

Les tables de hachage sont la structure de données la plus utilisée en production, et les systèmes d'IA s'en servent massivement pour les recherches de features, la mise en cache et la déduplication.

## Comment fonctionne le hachage

Une **fonction de hachage** prend une entrée (chaîne, nombre ou objet) et renvoie un entier — le **hash code**. Ce code est associé à un **indice de bucket** via modulo :

```
index = hashFunction(key) % numberOfBuckets
```

Une bonne fonction de hachage est :
- **Déterministe** — la même entrée produit toujours la même sortie.
- **Uniforme** — distribue les clés de manière égale entre les buckets.
- **Rapide** — calculable en O(1) par rapport à la taille de la clé.

### Exemple

```
function simpleHash(key, size):
    total = 0
    for char in key:
        total += charCode(char)
    return total % size
```

Cet exemple est volontairement naïf — les fonctions de hachage en production comme MurmurHash3 ou xxHash utilisent le bit-mixing pour obtenir une distribution bien meilleure.

## Gestion des collisions

Les collisions sont inévitables (principe des tiroirs). Deux stratégies principales :

### Chaînage

Chaque bucket contient une liste chaînée (ou un petit tableau) d'entrées. Les clés en collision sont ajoutées à la liste.

```
function insert(table, key, value):
    idx = hash(key) % table.size
    for entry in table[idx]:
        if entry.key == key:
            entry.value = value
            return
    table[idx].append((key, value))
```

**Avantages :** Simple, fonctionne bien sous charge modérée.
**Inconvénients :** Peu adapté au cache à cause du parcours de pointeurs.

### Adressage ouvert

Toutes les entrées résident directement dans le tableau. En cas de collision, on **sonde** le prochain emplacement libre.

- **Sondage linéaire :** vérifier index+1, index+2, …
- **Sondage quadratique :** vérifier index+1², index+2², …
- **Double hachage :** utiliser une seconde fonction de hachage pour déterminer le pas.

**Avantages :** Meilleure performance du cache, pas de pointeurs supplémentaires.
**Inconvénients :** Le clustering peut dégrader les performances ; la suppression est délicate (nécessite des tombstones).

<FunFact>Les dictionnaires Python utilisent l'adressage ouvert avec une séquence de sondage personnalisée. L'implémentation dict de CPython est si optimisée qu'elle traite plus de 50 millions de recherches par seconde sur du matériel moderne.</FunFact>

## Facteur de charge et rehachage

Le **facteur de charge** α = nombre d'entrées / nombre de buckets. À mesure que α augmente, les collisions se multiplient et les performances se dégradent.

| Facteur de charge | Recherches attendues (chaînage) |
|-------------------|-------------------------------|
| 0.5 | ~1.25 |
| 0.75 | ~1.375 |
| 1.0 | ~1.5 |
| 2.0 | ~2.0 |

La plupart des implémentations effectuent un **rehachage** lorsque α dépasse un seuil (généralement 0.75) : allouer un tableau plus grand, recalculer l'indice de chaque clé, et réinsérer.

Le rehachage est en O(n) mais survient rarement, donc le coût **amorti** de l'insertion reste O(1).

<ThinkAboutIt>Le rehachage double la taille de la table et réinsère chaque élément. Si votre système d'IA gère un flux de requêtes en temps réel, cette pause pourrait provoquer un pic de latence. Comment pourriez-vous répartir le coût du rehachage sur plusieurs insertions au lieu de tout faire d'un coup ?</ThinkAboutIt>

## Applications concrètes en IA

| Cas d'usage | Comment les tables de hachage aident |
|-------------|--------------------------------------|
| **Tables d'embeddings** | Associer les IDs de tokens à des représentations vectorielles denses en O(1) |
| **Feature hashing** | Hacher des features catégorielles de haute cardinalité dans un vecteur de taille fixe (le « hashing trick ») |
| **Déduplication** | Détecter des documents ou images en double en hachant leurs empreintes de contenu |
| **Couches de cache** | Stocker des prédictions de modèle précalculées indexées par le hash de l'entrée |
| **Count-min sketch** | Compter approximativement les fréquences dans des flux de données à l'aide de plusieurs fonctions de hachage |

## Bloom Filters : appartenance probabiliste

Un **Bloom filter** répond à la question « Cet élément est-il dans l'ensemble ? » avec :
- **Non certain** — l'élément est assurément absent.
- **Oui probable** — l'élément est probablement présent (faible taux de faux positifs).

Il utilise _k_ fonctions de hachage et un tableau de bits. L'insertion positionne _k_ bits ; la recherche vérifie si tous les _k_ bits sont positionnés.

```
function bloomInsert(filter, item):
    for i in 1..k:
        idx = hashFunctions[i](item) % filter.size
        filter.bits[idx] = 1

function bloomContains(filter, item):
    for i in 1..k:
        idx = hashFunctions[i](item) % filter.size
        if filter.bits[idx] == 0: return false
    return true   // probably present
```

**Usages en IA :** vérifier si une URL a déjà été explorée, filtrer le spam, et éviter les requêtes redondantes en base de données dans les couches de cache.

<FunFact>Google Chrome utilise un Bloom filter contenant des millions d'URLs malveillantes connues. Avant de charger une page, votre navigateur consulte le filtre — s'il dit « probablement oui », Chrome récupère la liste complète pour confirmer.</FunFact>

## Sets vs Maps : quand utiliser lequel

| Structure | Stocke | À utiliser quand |
|-----------|--------|-----------------|
| **Set** | Clés uniquement | Vous avez besoin de vérifications d'appartenance — « l'ai-je déjà vu ? » |
| **Map** | Paires clé–valeur | Vous devez associer des données à chaque clé — « quelle est la valeur pour cette clé ? » |

Les deux reposent sur des tables de hachage et offrent des opérations en O(1) en moyenne. Choisissez un **set** pour la déduplication, le suivi des visites et les contraintes d'unicité. Choisissez une **map** pour les recherches, les caches et les configurations.

## Points clés à retenir

1. Les tables de hachage offrent des recherches en O(1) en moyenne — la colonne vertébrale de la mise en cache, de l'indexation et du feature engineering.
2. Les collisions sont gérées par chaînage ou adressage ouvert ; le facteur de charge détermine quand rehacher.
3. Les Bloom filters échangent un faible taux de faux positifs contre d'énormes économies de mémoire pour les requêtes d'appartenance.
4. Comprendre les mécanismes internes du hachage vous aide à choisir le bon outil — set, map ou filtre probabiliste — pour chaque charge de travail IA.
