## Connectivity సమస్య

మీ దగ్గర nodes తో ఒక network ఉందని అనుకోండి, మీకు రెండు రకాల queries వస్తూ ఉంటాయి: *node A ని node B కి connect చేయండి* మరియు *A మరియు B connected గా ఉన్నాయా?* ప్రతి query కి BFS/DFS చేస్తే మొత్తం graph traverse చేయాల్సి వస్తుంది — లక్షల operations కి ఇది చాలా slow.

**Union-Find** రెండు queries ని దాదాపు constant time లో answer చేస్తుంది.

## Naive Approach — Quick Find

ప్రతి node కి ఒక component ID store చేయండి. Connectivity check చేయడానికి IDs ని O(1) లో compare చేయండి. కానీ రెండు components merge చేయడానికి ఒక group లోని ప్రతి member ని update చేయాలి — ప్రతి union కి O(n).

```
IDs:  [0, 0, 0, 3, 3]
       └─group 0─┘ └group 3┘
union(0, 3) → update all 3s to 0 → O(n)
```

మనకు ఏదైనా మెరుగైనది కావాలి.

## Union-Find Data Structure

ప్రతి node ఒక **parent** వైపు point చేస్తుంది. ప్రతి tree యొక్క root ఆ component యొక్క representative. రెండు nodes ఒకే root share చేస్తే మాత్రమే అవి connected.

```python
class UnionFind:
    def __init__(self, n):
        self.parent = list(range(n))
        self.rank = [0] * n
```

## Find with Path Compression

Deep tree లో parent pointers follow చేయడం slow గా ఉంటుంది. **Path compression** tree ని flat చేస్తుంది — ప్రతి visited node ని directly root కి point చేయిస్తుంది.

```python
def find(self, x):
    if self.parent[x] != x:
        self.parent[x] = self.find(self.parent[x])
    return self.parent[x]
```

Node 4 ని find చేయడానికి ముందు మరియు తర్వాత:

```
Before:  0←1←2←3←4     After:  0←1
                                0←2
                                0←3
                                0←4
```

## Union by Rank

ఎల్లప్పుడూ చిన్న tree ని పెద్ద tree కింద attach చేయండి. ఇది tree height ని logarithmic గా ఉంచుతుంది మరియు degenerate chains ఏర్పడకుండా చేస్తుంది.

```python
def union(self, x, y):
    rx, ry = self.find(x), self.find(y)
    if rx == ry:
        return False
    if self.rank[rx] < self.rank[ry]:
        rx, ry = ry, rx
    self.parent[ry] = rx
    if self.rank[rx] == self.rank[ry]:
        self.rank[rx] += 1
    return True
```

## Kruskal's MST Algorithm

Union-Find **Kruskal's algorithm** ని power చేస్తుంది — Minimum Spanning Trees కోసం:

1. అన్ని edges ని weight ప్రకారం sort చేయండి
2. ప్రతి edge (u, v) కి, `find(u) ≠ find(v)` అయితే, దాన్ని add చేసి `union(u, v)` చేయండి
3. n − 1 edges వచ్చినప్పుడు ఆపండి

Union-Find cycles ని prevent చేస్తుంది — endpoints ఇప్పటికే ఒకే root share చేస్తుంటే edge reject అవుతుంది. Total time: **O(E log E)** — sorting dominate చేస్తుంది.

<ThinkAboutIt>Kruskal's algorithm cycles detect చేయడానికి Union-Find use చేస్తుంది. DFS తో cycle ఎలా detect చేస్తారు, మరియు ఇక్కడ Union-Find ఎందుకు మరింత natural?</ThinkAboutIt>

## Classic Problems

### Number of Connected Components

Union-Find ని n nodes తో initialise చేయండి. ప్రతి edge ని `union` తో process చేయండి. చివరలో distinct roots count చేయండి.

### Accounts Merge (LeetCode 721)

ప్రతి account లో emails ఉంటాయి. రెండు accounts ఒక email share చేస్తే, అవి ఒకే వ్యక్తికి చెందినవి. Shared emails ఆధారంగా accounts ని union చేసి, group మరియు sort చేయండి.

### Redundant Connection (LeetCode 684)

ఒక tree ఇవ్వబడింది దానిలో ఒక extra edge ఉంది — redundant edge కనుగొనండి. Edges ని order లో process చేయండి — `union` మొదటిసారి `False` return చేసే edge (already connected) అదే answer.

<FunFact>Union-Find రెండు optimisations తో O(α(n)) లో run అవుతుందని proof ని Tarjan 1975 లో publish చేశారు — ఇది computer science లో అత్యంత elegant amortised analyses లో ఒకటిగా పరిగణించబడుతుంది.</FunFact>

## Union-Find vs BFS/DFS — ఎప్పుడు ఏది ఎంచుకోవాలి

| Scenario | ఉత్తమ ఎంపిక |
|----------|-------------|
| మళ్ళీ మళ్ళీ connectivity queries | Union-Find |
| Single traversal / shortest path | BFS/DFS |
| Incremental edge additions | Union-Find |
| Edge removals (dynamic graph) | BFS/DFS (లేదా offline UF) |
| Kruskal's MST | Union-Find |
| Topological sort / directed graph లో cycle | DFS |

Edges ని *incrementally* process చేస్తూ fast connectivity checks కావాలంటే Union-Find బాగా పని చేస్తుంది. ఒకే సారి traverse చేయాల్సి వస్తే లేదా path information కావాలంటే BFS/DFS వాడండి.
