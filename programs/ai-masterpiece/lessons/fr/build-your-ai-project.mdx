# Construisez votre projet IA

Vous avez passé des mois à étudier la théorie. Il est temps de **construire quelque chose de concret**, de le mettre en ligne et de prouver que vous maîtrisez le processus de bout en bout.

Rien sur votre CV ne parlera plus fort qu'un produit déployé que les gens peuvent réellement utiliser.

## Choisir le bon projet

Le choix du projet est la décision la plus importante de ce module. Si vous visez juste, tout le reste en découle naturellement.

Votre projet doit se situer à l'intersection de trois éléments :

- **La passion** - vous y passerez vos soirées ; choisissez un sujet qui vous tient vraiment à cœur.
- **Le défi technique** - il doit vous pousser dans au moins un domaine que vous ne maîtrisez pas encore.
- **La valeur portfolio** - un recruteur doit le voir et penser : « Cette personne sait livrer. »

<Callout>
Évitez les jeux de données jouets et les clones de tutoriels. Les recruteurs ont vu des milliers de classifieurs MNIST. Résolvez un problème qui compte pour un vrai public, même s'il est restreint.
</Callout>

## Planifier votre MVP

L'erreur la plus fréquente est de viser trop large. Définissez votre **Minimum Viable Product** sans pitié :

1. **Une seule fonctionnalité centrale** - un parcours utilisateur, un modèle, une interface.
2. **Stack technique** - choisissez des outils que vous connaissez plus une nouveauté. Pour la plupart des projets : Python + FastAPI en backend, React ou Next.js en frontend, une base de données managée.
3. **Calendrier** - accordez-vous deux semaines pour le MVP. Si ça prend plus longtemps, votre périmètre est trop large.
4. **Métrique de succès** - définissez un chiffre qui prouve que le projet fonctionne (ex. « précision de récupération supérieure à 80 % sur mon jeu de test »).

<ThinkAboutIt>
Décrivez votre idée de projet en une phrase. Maintenant, supprimez la moitié des fonctionnalités. Voilà votre MVP.
</ThinkAboutIt>

## Construire en public

Partager votre avancement sur les réseaux sociaux n'est pas de la vanité - c'est une stratégie de carrière.

- **Jour 1** : Publiez votre idée et demandez des retours.
- **Chaque semaine** : Partagez une courte mise à jour - ce que vous avez construit, ce qui a cassé, ce que vous avez appris.
- **Jour du lancement** : Rédigez un thread ou un article de blog décrivant l'architecture.

Construire en public crée de la responsabilité, attire des collaborateurs et met votre nom sous les yeux des recruteurs avant même que vous ne postuliez.

<FunFact>
Une enquête Stack Overflow de 2024 a révélé que 68 % des recruteurs consultent les projets publics et publications d'un candidat avant de décider de le convoquer en entretien.
</FunFact>

## Déployer votre projet

Un projet qui ne tourne que sur `localhost` est invisible. Personne ne peut l'essayer, personne ne peut le partager, et aucun recruteur ne le verra jamais.

Le déploiement n'est pas un bonus - c'est ce qui distingue un projet hobby d'une pièce de portfolio. Voici trois options solides :

| Plateforme | Idéal pour | Offre gratuite |
|------------|-----------|----------------|
| **Vercel** | Frontends Next.js, API serverless | Généreuse |
| **Railway** | Backends Python, bases de données | 500 heures/mois |
| **HuggingFace Spaces** | Démos ML avec Gradio ou Streamlit | Illimitée pour CPU |

Quel que soit votre choix, assurez-vous que votre README contient :

- Une description en une ligne de ce que fait le projet.
- Un lien live ou un GIF de démonstration.
- Des instructions claires pour l'installation locale.

<Quiz question="Pourquoi le déploiement est-il important pour votre portfolio ?" options="Il prouve que vous savez écrire des tests unitaires|Il montre que vous maîtrisez le cycle complet, du code à la production|Il garantit que votre code est sans bugs|Il remplace entièrement le besoin d'un CV" answer="1" explanation="Le déploiement démontre que vous maîtrisez le cycle complet - pas seulement l'écriture de code, mais aussi la mise en production et la maintenance d'un produit en ligne." />

## Recueillir des retours et itérer

Une fois votre MVP en ligne, le vrai apprentissage commence. La mise en ligne n'est pas la ligne d'arrivée - c'est la ligne de départ.

Sollicitez activement des retours de personnes qui seront honnêtes avec vous :

- Partagez-le dans les communautés Discord et Slack pertinentes.
- Publiez sur Reddit dans des sous-forums spécialisés.
- Demandez à deux ou trois ingénieurs que vous respectez de l'essayer et de donner un avis franc.

Traitez chaque retour comme un cadeau. Corrigez les problèmes critiques, notez le reste pour la version deux, et poussez des mises à jour rapidement.

<ThinkAboutIt>
Pensez à une communauté où vos utilisateurs cibles se retrouvent. Prévoyez d'y partager votre projet dans les 48 heures suivant le lancement.
</ThinkAboutIt>

## L'ajouter à votre portfolio

Votre projet n'a de valeur que par sa présentation. Quand vous l'ajoutez à votre portfolio :

1. **Rédigez une étude de cas** - problème, approche, diagramme d'architecture, résultats, leçons apprises.
2. **Épinglez-le sur GitHub** - historique de commits propre, bon README, issues ouvertes pour les travaux futurs.
3. **Enregistrez une vidéo de démonstration de deux minutes** - montrez le produit et expliquez un choix technique.
4. **Liez-le partout** - CV, LinkedIn, site personnel, README de profil GitHub.

<Quiz question="Quelle est la manière la plus efficace de présenter un projet dans votre portfolio ?" options="Simplement mettre un lien vers le dépôt GitHub|Rédiger une étude de cas avec le problème, l'approche, l'architecture et les résultats|Inclure uniquement une capture d'écran|Lister les technologies utilisées sans contexte" answer="1" explanation="Une étude de cas donne aux recruteurs le tableau complet - ce que vous avez construit, pourquoi, comment et ce que vous avez appris - bien plus convaincant qu'un simple lien vers un dépôt." />

## Votre prochaine étape

Arrêtez de lire. Ouvrez un nouveau dépôt maintenant. Nommez-le d'après votre idée de projet.

Rédigez le README en premier - décrivez ce qu'il fera, à qui il s'adresse et quel est le périmètre de votre MVP. Poussez ce README ce soir.

L'écart entre « je veux construire quelque chose » et « j'ai livré quelque chose » tient à un seul commit.

<FunFact>
Les données GitHub montrent que les dépôts avec un README bien rédigé reçoivent quatre fois plus d'étoiles et de forks que ceux qui n'en ont pas - la première impression compte, même en open source.
</FunFact>
