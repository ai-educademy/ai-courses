# Pipeline ML de bout en bout - Des données brutes au modèle déployé

La plupart des tutoriels s'arrêtent à `model.fit()`. En production, cela ne représente que 20 % du travail. Cette leçon parcourt le **cycle de vie ML complet** - le même pipeline qui alimente les moteurs de recommandation de Spotify, la détection de fraude chez Stripe et le classement de recherche chez Google.

## 🗺️ Le cycle de vie ML en un coup d'œil

<Illustration src="/images/lessons/end-to-end-ml-pipeline.svg" alt="Schéma des étapes du pipeline ML, de l'ingestion de données au monitoring" caption="Les huit étapes d'un pipeline ML en production - chacune est un point de défaillance potentiel." />

Tout système ML en production passe par ces étapes :

1. **Collecte de données** - API, bases de données, flux d'événements, web scraping
2. **Nettoyage de données** - gestion des valeurs nulles, déduplication, conversion de types
3. **Ingénierie de features** - transformer les signaux bruts en features exploitables par le modèle
4. **Sélection de modèle** - choisir les architectures adaptées à votre problème
5. **Entraînement** - orchestration GPU/CPU, optimisation des hyperparamètres
6. **Évaluation** - métriques hors ligne, A/B testing, audits d'équité
7. **Déploiement** - infrastructure de service, budgets de latence
8. **Monitoring** - détection de drift, alertes, déclenchement du réentraînement

## 📦 Collecte et versionnement des données

Les données brutes sont votre fondation. Traitez-les comme du code source - **versionnez tout**.

```bash
# Initialiser DVC dans votre dépôt
dvc init
dvc remote add -d storage s3://my-ml-data-bucket

# Suivre un jeu de données
dvc add data/transactions.csv
git add data/transactions.csv.dvc .gitignore
git commit -m "Track raw transaction dataset v1"
dvc push
```

**DVC** (Data Version Control) vous permet de faire `git checkout` sur n'importe quel jeu de données historique. Combiné avec les tags Git, vous pouvez reproduire n'importe quelle expérience à n'importe quel moment.

<FunFact>Netflix traite plus de 1,5 billion d'événements par jour dans ses pipelines de données - environ 17 millions d'événements par seconde alimentant ses modèles de recommandation.</FunFact>

## 🛠️ Ingénierie de features - Là où se gagnent les compétitions

Les features comptent plus que les algorithmes. Un arbre à gradient boosting avec d'excellentes features bat un réseau neuronal profond avec de mauvaises features.

```python
import pandas as pd

df = pd.read_csv("data/transactions.csv", parse_dates=["timestamp"])

# Features temporelles
df["hour"] = df["timestamp"].dt.hour
df["is_weekend"] = df["timestamp"].dt.dayofweek >= 5

# Features d'agrégation
user_stats = df.groupby("user_id")["amount"].agg(["mean", "std", "count"])
user_stats.columns = ["user_avg_amount", "user_std_amount", "user_txn_count"]
df = df.merge(user_stats, on="user_id")

# Features de ratio
df["amount_vs_avg"] = df["amount"] / df["user_avg_amount"]
```

## 🧪 Suivi d'expériences avec MLflow

Ne perdez jamais la trace de ce que vous avez essayé. MLflow enregistre automatiquement les paramètres, métriques et artefacts.

```python
import mlflow
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.metrics import f1_score

mlflow.set_experiment("fraud-detection-v2")

with mlflow.start_run(run_name="gbm-baseline"):
    params = {"n_estimators": 500, "max_depth": 6, "learning_rate": 0.05}
    mlflow.log_params(params)

    model = GradientBoostingClassifier(**params)
    model.fit(X_train, y_train)

    score = f1_score(y_test, model.predict(X_test))
    mlflow.log_metric("f1_score", score)
    mlflow.sklearn.log_model(model, "model")
```

## 🌐 Service de modèle et déploiement

Votre modèle a besoin d'une API. Deux patterns dominants :

**REST API** - simple, basé sur HTTP, idéal pour un débit modéré :
```python
# Serveur de modèle FastAPI
from fastapi import FastAPI
import joblib

app = FastAPI()
model = joblib.load("model.pkl")

@app.post("/predict")
async def predict(features: dict):
    prediction = model.predict([list(features.values())])
    return {"prediction": int(prediction[0])}
```

**gRPC** - protocole binaire, 2 à 10× plus rapide que REST, idéal pour les microservices internes et les scénarios à haut débit.

### Déploiements canari

N'envoyez jamais 100 % du trafic vers un nouveau modèle. Utilisez les **déploiements canari** : routez 5 % → 25 % → 50 % → 100 %, en surveillant les taux d'erreur à chaque étape. Faites un rollback instantané si les métriques se dégradent.

## 📊 Monitoring et boucle de rétroaction

Le déploiement n'est pas la ligne d'arrivée - c'est le coup de départ. Surveillez :

- **Le drift de données** - les distributions d'entrée s'éloignent des données d'entraînement
- **Le drift de concept** - la relation entre features et cible change
- **La dégradation de performance** - les métriques de précision/latence déclinent

Des outils comme **Evidently AI**, **WhyLabs** et **Arize** fournissent des tableaux de bord de drift. Quand le drift dépasse les seuils, déclenchez un réentraînement automatique avec des données de production fraîches - bouclant la boucle de rétroaction.

## 🎯 Points clés à retenir

- Versionnez **tout** : code, données, modèles et configurations de pipeline
- L'ingénierie de features offre un meilleur retour sur investissement que les changements d'architecture
- Encodez les pipelines sous forme de code - les étapes manuelles sont de la dette technique
- Déployez avec des releases canari et surveillez sans relâche
- La boucle de rétroaction (données de production → réentraînement) est ce qui sépare les démos des systèmes de production
