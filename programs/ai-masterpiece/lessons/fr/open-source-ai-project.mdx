# Publier un projet IA open source - Votre chef-d'œuvre

C'est la dernière leçon. Tout ce que vous avez appris converge ici : pipelines ML, implémentation de recherche, conception système et rigueur d'ingénierie. Vous allez **publier un vrai projet IA open source** qui résout un problème concret et qui perdurera au-delà de ce cours.

## 💡 Choisir votre idée de projet

Les meilleurs projets open source naissent d'une **frustration personnelle**. Ne construisez pas une énième application TODO avec de l'IA - trouvez un manque :

- **Résolvez votre propre problème** - un outil que vous aimeriez avoir dans votre workflow quotidien
- **Améliorez un outil existant** - trouvez la limitation d'une bibliothèque populaire et résolvez-la
- **Implémentez un article de recherche** - transformez une avancée en bibliothèque utilisable
- **Comblez un fossé** - connectez deux écosystèmes qui ne communiquent pas entre eux

<Illustration src="/images/lessons/open-source-ai-project.svg" alt="Cycle de vie d'un projet open source, de l'idée au lancement puis à la croissance de la communauté" caption="Le cycle de vie open source - la publication est le milieu du parcours, pas la ligne d'arrivée." />

**Validez avant de construire** : cherchez sur GitHub, PyPI et npm. Si quelque chose de similaire existe, votre projet a besoin d'un différenciateur clair - plus rapide, API plus simple, meilleure documentation ou approche innovante.

<FunFact>Hugging Face a démarré comme une entreprise de chatbot pour adolescents. Quand ils ont mis en open source leur bibliothèque Transformers comme projet annexe, elle est devenue si populaire qu'ils ont pivoté toute l'entreprise autour - désormais valorisée à plus de 4,5 milliards de dollars.</FunFact>

## 📋 Planification du projet - Périmètre MVP

Définissez votre **Minimum Viable Project** sans concession :

```markdown
## Périmètre MVP (Semaine 1-2)
- Fonctionnalité principale : UNE seule chose, bien faite
- Interface CLI ou API Python
- 3 exemples fonctionnels
- README de base avec installation et quickstart

## Périmètre V1.0 (Semaine 3-4)
- Documentation complète
- Couverture de tests > 80 %
- Pipeline CI/CD
- Publié sur PyPI/npm
- Démo déployée
```

**L'anti-pattern** : passer des mois à perfectionner du code que personne n'a vu. Publiez tôt, itérez sur la base des retours.

<ThinkAboutIt>Pensez aux trois derniers outils que vous avez installés. Qu'est-ce qui vous a fait les choisir par rapport aux alternatives ? Le README, le design de l'API, le nombre d'étoiles, ou autre chose ? Appliquez ces enseignements à votre propre projet.</ThinkAboutIt>

## 🏗️ Configuration du dépôt

La première impression compte. Un dépôt professionnel nécessite : un répertoire `src/` pour le code, `tests/`, `examples/`, `docs/`, `.github/workflows/ci.yml`, plus les fichiers racine - `LICENSE` (MIT ou Apache 2.0), `README.md`, `CONTRIBUTING.md`, `CODE_OF_CONDUCT.md` et `pyproject.toml`.

### Le README - Votre fichier le plus important

Votre README est une page de vente. Il doit contenir : **description en une ligne**, **installation** (`pip install mon-projet`), **quickstart** (code fonctionnel en moins de 10 lignes), **fonctionnalités clés** (liste à puces), **tableau comparatif** avec les alternatives et un **lien vers la documentation**.

Un quickstart que l'on peut copier-coller et qui fonctionne immédiatement convertit les visiteurs en utilisateurs. Tout le reste est secondaire par rapport à cette première expérience réussie." />

## 📦 Publication sur PyPI

Packaging Python moderne avec `pyproject.toml` :

```toml
[build-system]
requires = ["hatchling"]
build-backend = "hatchling.build"

[project]
name = "my-ai-project"
version = "0.1.0"
description = "One-line description of your project"
requires-python = ">=3.9"
license = "MIT"
dependencies = ["torch>=2.0", "numpy>=1.24"]

[project.urls]
Documentation = "https://my-ai-project.readthedocs.io"
Repository = "https://github.com/username/my-ai-project"
```

```bash
# Build and publish
pip install build twine
python -m build
twine upload dist/*
```

**Releases automatisées** : utilisez GitHub Actions pour publier sur PyPI à chaque release taguée - ne publiez plus jamais manuellement après la première fois.

## 🎨 Créer des démos

Une démo convertit les sceptiques en utilisateurs. Trois excellentes plateformes :

| Plateforme | Idéale pour | Déploiement |
|------------|-------------|-------------|
| **Hugging Face Spaces** | Démos de modèles ML | Git push, tier GPU gratuit |
| **Streamlit** | Applications de données, dashboards | Streamlit Cloud, tier gratuit |
| **Gradio** | Interfaces ML rapides | Hugging Face Spaces ou autonome |

```python
# Démo Gradio - 8 lignes pour une interface partageable
import gradio as gr
from my_ai_project import predict

demo = gr.Interface(
    fn=predict,
    inputs=gr.Textbox(label="Enter text"),
    outputs=gr.Label(label="Prediction"),
    title="My AI Project Demo"
)
demo.launch()
```

## 📖 Rédiger la documentation

Suivez le **framework Diátaxis** - quatre types de documentation : **Tutoriels** (orientés apprentissage, pas à pas), **Guides pratiques** (orientés tâches, concrets), **Référence** (documentation API complète) et **Explications** (compréhension conceptuelle). Utilisez **MkDocs** ou **Docusaurus** pour générer des sites depuis du Markdown. Déployez gratuitement sur GitHub Pages ou Read the Docs.

## 📣 Promouvoir votre projet

Le construire, c'est la moitié du travail. Amener les gens à l'utiliser, c'est l'autre moitié :

1. **Hacker News** - post « Show HN : [description] » le jour du lancement
2. **Reddit** - r/MachineLearning, r/Python, subreddits pertinents
3. **Twitter/X** - thread expliquant le problème et votre solution, avec un GIF de démo
4. **Product Hunt** - si votre projet a une interface web
5. **Article de blog** - write-up technique détaillé sur votre site personnel ou Medium/Dev.to

**Le timing compte** : lancez du mardi au jeudi matin (GMT). Évitez les week-ends et jours fériés.

<FunFact>Les créateurs de FastAPI, LangChain et Streamlit attribuent tous à un post « Show HN » bien minuté sur Hacker News le rôle de principal moteur de leur vague d'adoption initiale.</FunFact>

## 🔧 Maintenir votre projet

La pérennité compte plus que la viralité :

- **Étiquetez les issues** avec `good-first-issue`, `help-wanted`, `bug`, `enhancement`
- **Répondez aux issues sous 48 heures** - même un « Merci, je vais regarder ça » construit la confiance
- **Rédigez un CONTRIBUTING.md** avec les instructions d'installation, les standards de code et les guidelines pour les PR
- **Utilisez le versionnement sémantique** - les changements cassants incrémentent la version majeure
- **Automatisez tout** - CI, releases, génération du changelog, mises à jour des dépendances

## 💰 Pistes de monétisation

L'open source ne signifie pas travailler gratuitement pour toujours. Modèles durables : **GitHub Sponsors** (faible effort, dons récurrents), **consulting** (installation et formation payantes), **wrapper SaaS** (version hébergée avec fonctionnalités entreprise - effort maximal mais le plus durable), **double licence** (open source + commerciale) et **sponsoring d'entreprise** (fonctionnalités prioritaires pour les entreprises payantes).

## 🎯 Checklist de votre chef-d'œuvre

Avant de considérer le projet comme livré :

- [ ] La fonctionnalité principale fonctionne et est testée
- [ ] Le README contient l'installation, le quickstart et un aperçu des fonctionnalités
- [ ] Publié sur PyPI ou npm
- [ ] Démo en ligne sur Hugging Face Spaces ou Streamlit
- [ ] Le pipeline CI/CD exécute les tests sur chaque PR
- [ ] La documentation couvre les tutoriels et la référence API
- [ ] LICENSE, CONTRIBUTING.md et CODE_OF_CONDUCT.md présents
- [ ] Lancé sur au moins deux plateformes (HN, Reddit, Twitter)

**Ce projet EST votre portfolio.** Quand un recruteur demandera « Qu'avez-vous construit ? », vous aurez une réponse en ligne, documentée et publiée.
