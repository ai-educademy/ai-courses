# अपना AI Project बनाएँ

आपने महीनों theory सीखी है। अब वक़्त है **कुछ असली बनाने का**, उसे दुनिया के सामने लाने का, और ये साबित करने का कि आप end-to-end deliver कर सकते हैं।

आपके CV पर कोई भी चीज़ एक deployed product से ज़्यादा ज़ोर से नहीं बोलेगी — जिसे लोग सच में इस्तेमाल कर सकें।

## सही Project कैसे चुनें

सही project चुनना इस capstone का सबसे अहम फ़ैसला है। अगर ये सही हुआ तो बाकी सब आसान हो जाएगा।

आपका project तीन चीज़ों के intersection पर होना चाहिए:

- **Passion** — आप इस पर शामें बिताएँगे; कुछ ऐसा चुनें जिसकी आपको सच में परवाह है।
- **Skill gap** — ये आपको कम से कम एक ऐसे area में खींचे जो आपने अभी तक master नहीं किया।
- **Portfolio value** — hiring managers देखकर सोचें, "ये इंसान ship कर सकता है।"

<Callout>
Toy datasets और tutorial clones से बचें। Recruiters ने हज़ारों MNIST classifiers देखे हैं। एक ऐसी problem solve करें जो किसी real audience के लिए मायने रखती हो, भले ही वो audience छोटी हो।
</Callout>

## अपना MVP Plan करें

Builders की सबसे बड़ी ग़लती होती है scope बहुत बड़ा रखना। अपना **Minimum Viable Product** बेरहमी से define करें:

1. **सिर्फ़ core feature** — एक user flow, एक model, एक interface।
2. **Tech stack** — वो tools चुनें जो आप जानते हैं plus एक नई चीज़। ज़्यादातर projects के लिए: Python + FastAPI backend, React या Next.js frontend, एक managed database।
3. **Timeline** — MVP के लिए दो हफ़्ते दें। अगर ज़्यादा लगे, तो scope बहुत बड़ा है।
4. **Success metric** — एक number define करें जो बताए कि project काम कर रहा है (जैसे, "मेरे test set पर retrieval accuracy 80% से ऊपर")।

<ThinkAboutIt>
अपना project idea एक sentence में लिखें। अब आधे features काट दें। बस यही आपका MVP है।
</ThinkAboutIt>

## Building in Public

Social media पर अपनी progress share करना vanity नहीं है — ये एक career strategy है।

- **Day 1**: अपना idea post करें और feedback माँगें।
- **हर हफ़्ते**: एक छोटा update share करें — क्या बनाया, क्या टूटा, क्या सीखा।
- **Launch day**: एक thread या blog post लिखें जिसमें architecture explain करें।

Building in public से accountability बनती है, collaborators आते हैं, और apply करने से पहले ही आपका नाम hiring managers के सामने आ जाता है।

<FunFact>
Stack Overflow के 2024 survey में पता चला कि 68% hiring managers interview का फ़ैसला लेने से पहले candidate के public projects और posts चेक करते हैं।
</FunFact>

## अपना Project Deploy करें

जो project सिर्फ़ `localhost` पर चलता है, वो invisible है। कोई try नहीं कर सकता, कोई share नहीं कर सकता, और कोई hiring manager कभी देखेगा नहीं।

Deployment nice-to-have नहीं है — ये hobby project और portfolio piece के बीच का फ़र्क़ है। तीन बढ़िया options:

| Platform | किसके लिए बेस्ट | Free Tier |
|----------|----------------|-----------|
| **Vercel** | Next.js frontends, serverless APIs | काफ़ी generous |
| **Railway** | Python backends, databases | 500 hours/month |
| **HuggingFace Spaces** | Gradio या Streamlit के साथ ML demos | CPU के लिए unlimited |

जो भी चुनें, README में ये ज़रूर शामिल करें:

- Project क्या करता है इसकी एक-line description।
- एक live link या demo GIF।
- Local development के लिए clear setup instructions।

## Feedback लें और Iterate करें

जब आपका MVP live हो जाए, तब असली सीखना शुरू होता है। Shipping finish line नहीं है — ये starting line है।

ऐसे लोगों से actively feedback माँगें जो honest होंगे:

- Related Discord और Slack communities में share करें।
- Reddit पर domain-specific subreddits में post करें।
- दो-तीन engineers से request करें कि वो try करें और honest criticism दें।

हर feedback को तोहफ़ा समझें। Critical issues fix करें, बाकी version two के लिए note करें, और जल्दी updates push करें।

<ThinkAboutIt>
ऐसी एक community सोचें जहाँ आपके target users रहते हैं। Launch के 48 घंटों के अंदर वहाँ share करने का plan बनाएँ।
</ThinkAboutIt>

## इसे अपने Portfolio में जोड़ें

आपका project उतना ही valuable है जितनी उसकी presentation। Portfolio में जोड़ते वक़्त:

1. **Case study लिखें** — problem, approach, architecture diagram, results, सीखे गए lessons।
2. **GitHub पर pin करें** — clean commit history, अच्छा README, future work के लिए open issues।
3. **दो-minute का demo video record करें** — product दिखाएँ और एक technical decision explain करें।
4. **हर जगह link करें** — CV, LinkedIn, personal site, GitHub profile README।

## आपका अगला कदम

पढ़ना बंद करें। अभी एक नया repository खोलें। उसे अपने project idea के नाम पर रखें।

पहले README लिखें — बताएँ कि ये क्या करेगा, किसके लिए है, और आपका MVP scope क्या है। आज रात वो README push कर दें।

"मैं कुछ बनाना चाहता हूँ" और "मैंने कुछ ship किया" के बीच का फ़ासला बस एक commit का है।

<FunFact>
GitHub data दिखाता है कि अच्छी-तरह लिखे README वाले repositories को बिना README वालों से चार गुना ज़्यादा stars और forks मिलते हैं — first impressions open source में भी मायने रखते हैं।
</FunFact>
