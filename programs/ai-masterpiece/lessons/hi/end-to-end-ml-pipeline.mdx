# End-to-End ML Pipeline — Raw Data से Deployed Model तक

ज़्यादातर tutorials `model.fit()` पर रुक जाते हैं। Production में, ये बमुश्किल 20% काम है। ये lesson **पूरे ML lifecycle** को cover करता है — वही pipeline जो Spotify के recommendation engines, Stripe के fraud detection, और Google की search ranking को power करता है।

## 🗺️ ML Lifecycle एक नज़र में

<Illustration src="/images/lessons/end-to-end-ml-pipeline.svg" alt="Data ingestion से monitoring तक ML pipeline stages का diagram" caption="Production ML pipeline के आठ stages — हर एक potential failure point है।" />

हर production ML system इन stages से गुज़रता है:

1. **Data Collection** — APIs, databases, event streams, web scraping
2. **Data Cleaning** — nulls handle करना, deduplication, type coercion
3. **Feature Engineering** — raw signals को model-ready features में बदलना
4. **Model Selection** — अपनी problem से match करने वाले architectures चुनना
5. **Training** — GPU/CPU orchestration, hyperparameter tuning
6. **Evaluation** — offline metrics, A/B testing, fairness audits
7. **Deployment** — serving infrastructure, latency budgets
8. **Monitoring** — drift detection, alerting, retraining trigger

## 📦 Data Collection और Versioning

Raw data आपकी foundation है। इसे source code की तरह treat करें — **सब कुछ version करें**।

```bash
# Initialise DVC in your repo
dvc init
dvc remote add -d storage s3://my-ml-data-bucket

# Track a dataset
dvc add data/transactions.csv
git add data/transactions.csv.dvc .gitignore
git commit -m "Track raw transaction dataset v1"
dvc push
```

**DVC** (Data Version Control) आपको किसी भी historical dataset को `git checkout` करने देता है। Git tags के साथ मिलाकर, आप किसी भी point in time से कोई भी experiment reproduce कर सकते हैं।

<FunFact>Netflix अपनी data pipelines के ज़रिए रोज़ाना 1.5 trillion से ज़्यादा events process करता है — हर second लगभग 17 million events जो उनके recommendation models को feed करते हैं।</FunFact>

## 🛠️ Feature Engineering — जहाँ Competitions जीती जाती हैं

Features algorithms से ज़्यादा matter करते हैं। शानदार features वाला gradient-boosted tree ख़राब features वाले deep neural network को हरा देता है।

```python
import pandas as pd

df = pd.read_csv("data/transactions.csv", parse_dates=["timestamp"])

# Temporal features
df["hour"] = df["timestamp"].dt.hour
df["is_weekend"] = df["timestamp"].dt.dayofweek >= 5

# Aggregation features
user_stats = df.groupby("user_id")["amount"].agg(["mean", "std", "count"])
user_stats.columns = ["user_avg_amount", "user_std_amount", "user_txn_count"]
df = df.merge(user_stats, on="user_id")

# Ratio features
df["amount_vs_avg"] = df["amount"] / df["user_avg_amount"]
```

## 🧪 MLflow के साथ Experiment Tracking

कभी track मत खोइए कि आपने क्या try किया। MLflow parameters, metrics, और artefacts automatically log करता है।

```python

from sklearn.ensemble import GradientBoostingClassifier
from sklearn.metrics import f1_score

    params = {"n_estimators": 500, "max_depth": 6, "learning_rate": 0.05}
    mlflow.log_params(params)
```

## 🌐 Model Serving और Deployment

आपके model को एक API चाहिए। दो dominant patterns:

**REST API** — simple, HTTP-based, moderate throughput के लिए बढ़िया:
```python
# FastAPI model server
from fastapi import FastAPI
import joblib

app = FastAPI()
model = joblib.load("model.pkl")

@app.post("/predict")
async def predict(features: dict):
    prediction = model.predict([list(features.values())])
    return {"prediction": int(prediction[0])}
```

**gRPC** — binary protocol, REST से 2–10× faster, internal microservices और high-throughput scenarios के लिए ideal।

### Canary Deployments

कभी भी 100% traffic नए model पर मत भेजें। **Canary releases** इस्तेमाल करें: 5% → 25% → 50% → 100% route करें, हर step पर error rates monitor करें। Metrics degrade हों तो instantly roll back करें।

## 📊 Monitoring और Feedback Loop

Deployment finish line नहीं है — ये starting gun है। Monitor करें:

- **Data drift** — input distributions training data से shift हो रही हैं
- **Concept drift** — features और target के बीच relationship बदल रहा है
- **Performance degradation** — accuracy/latency metrics गिर रहे हैं

**Evidently AI**, **WhyLabs**, और **Arize** जैसे tools drift dashboards provide करते हैं। जब drift thresholds से ज़्यादा हो, fresh production data से automatic retraining trigger करें — feedback loop बंद करें।

<ThinkAboutIt>आपकी monitoring Friday शाम को significant drift detect करती है। क्या आप automatic retrain trigger करेंगे, Monday की manual review का इंतज़ार करेंगे, या previous model version पर roll back करेंगे? कौन से factors आपके decision को influence करेंगे?</ThinkAboutIt>

## 🎯 मुख्य Takeaways

- **सब कुछ** version करें: code, data, models, और pipeline configs
- Feature engineering architecture changes से ज़्यादा ROI deliver करती है
- Pipelines को code में encode करें — manual steps technical debt हैं
- Canary releases से deploy करें और relentlessly monitor करें
- Feedback loop (production data → retraining) वो है जो demos को production systems से अलग करता है
