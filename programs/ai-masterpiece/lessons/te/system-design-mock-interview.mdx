# System Design Mock Interview — 45 నిమిషాల సీనియర్ ఇంజనీర్ సిమ్యులేషన్

System design ఇంటర్వ్యూలు సీనియర్ ఇంజనీర్లను మిగతా వారి నుండి వేరు చేస్తాయి. ఒకే సరైన సమాధానం లేదు — ఇంటర్వ్యూయర్లు మీ **structured thinking**, trade-off analysis, మరియు ambiguity ను navigate చేసే సామర్థ్యాన్ని evaluate చేస్తారు. ఈ పాఠం పూర్తి 45 నిమిషాల simulation.

## 🎯 సమస్య: ఫుడ్ డెలివరీ System డిజైన్ చేయండి

*"కస్టమర్లు, రెస్టారెంట్లు, మరియు డెలివరీ drivers ను కనెక్ట్ చేసే Uber Eats లేదా Deliveroo వంటి ఫుడ్ డెలివరీ ప్లాట్‌ఫారమ్ డిజైన్ చేయండి."*

## దశ 1: అవసరాల సేకరణ (5 నిమిషాలు)

**వెంటనే boxes గీయడం మొదలుపెట్టకండి.** ముందుగా స్పష్టీకరణ ప్రశ్నలు అడగండి:

**Functional అవసరాలు:**
- కస్టమర్లు రెస్టారెంట్లు browse చేయడం, menus చూడడం, orders పెట్టడం, real-time లో delivery track చేయడం
- రెస్టారెంట్లు menus manage చేయడం, orders accept/reject చేయడం, preparation status update చేయడం
- Drivers డెలివరీ requests స్వీకరించడం, pickup/dropoff కు navigate చేయడం, delivery confirm చేయడం
- Order lifecycle అంతటా real-time estimated delivery time (EDT) చూపించడం

**Non-functional అవసరాలు:**
- **Availability**: 99.99% uptime (సంవత్సరానికి ≈52 నిమిషాల downtime)
- **Latency**: search < 200ms, order placement < 500ms, location updates < 1s
- **Scale**: 50M నెలవారీ active users, రోజుకు 5M orders, peak లో 500K concurrent users
- **Consistency**: search కోసం eventual consistency సరిపోతుంది; payments కోసం strong consistency

## దశ 2: API Design (5 నిమిషాలు)

Architecture కంటే ముందు core contracts నిర్వచించండి:

```
# Customer APIs
GET    /api/v1/restaurants?lat={}&lng={}&cuisine={}  → Restaurant[]
GET    /api/v1/restaurants/{id}/menu                  → Menu
POST   /api/v1/orders                                 → Order
GET    /api/v1/orders/{id}/track                      → OrderStatus + DriverLocation

# Restaurant APIs
PATCH  /api/v1/restaurants/{id}/menu/items/{itemId}   → MenuItem
POST   /api/v1/orders/{id}/accept                     → Order
POST   /api/v1/orders/{id}/ready                      → Order

# Driver APIs
POST   /api/v1/drivers/{id}/location                  → void (fire-and-forget)
POST   /api/v1/deliveries/{id}/accept                 → Delivery
POST   /api/v1/deliveries/{id}/complete               → Delivery
```

**కీలక నిర్ణయం**: real-time driver tracking కోసం WebSockets, మిగతా అన్నింటికీ REST ఉపయోగించండి.

## దశ 3: ఉన్నత-స్థాయి Architecture (10 నిమిషాలు)

### Core Services

| Service | బాధ్యత | Database |
|---------|--------|----------|
| **User Service** | Authentication, profiles, addresses | PostgreSQL |
| **Restaurant Service** | Menus, availability, hours | PostgreSQL + Elasticsearch |
| **Order Service** | Order lifecycle, state machine | PostgreSQL (strong consistency) |
| **Driver Service** | Driver location, availability, matching | Redis (location) + PostgreSQL |
| **Payment Service** | Charges, refunds, restaurant payouts | PostgreSQL (ACID అవసరం) |
| **Notification Service** | Push, SMS, email notifications | Message queue consumer |

### Communication Patterns

- **Synchronous**: API Gateway → individual services (REST/gRPC)
- **Asynchronous**: Order events **Kafka** కు publish — Payment, Notification, మరియు Analytics services consume చేస్తాయి
- **Real-time**: Driver location updates WebSocket ద్వారా → geospatial indexing తో **Redis** లో store

<FunFact>Uber ప్రపంచవ్యాప్తంగా సెకనుకు 1 మిలియన్ కంటే ఎక్కువ location updates process చేస్తుంది. సాంప్రదాయ latitude/longitude queries scale లో తట్టుకోలేక వారు hexagonal hierarchical grids ఉపయోగించి సొంత geospatial indexing system (H3) నిర్మించారు.</FunFact>

## దశ 4: Deep Dive — కీలక భాగాలు (15 నిమిషాలు)

### Driver Matching Algorithm

Order pickup కోసం ready అయినప్పుడు, matching service ఉత్తమ driver ను assign చేయాలి:

```
Score(driver) = w1 × (1 / distance_to_restaurant)
              + w2 × driver_rating
              + w3 × (1 / current_active_orders)
              + w4 × acceptance_rate
```

**Implementation**: restaurant నుండి 5km లోపల drivers కోసం Redis `GEORADIUS` query, ప్రతి candidate ను score చేయండి, అత్యధిక-scoring available driver ను assign చేయండి. Decline చేస్తే, 30-సెకన్ల timeout తో తదుపరి driver కు offer చేయండి.

### Real-Time Order Tracking

```
Client ←→ WebSocket Gateway ←→ Driver Service
                                    ↑
                              Redis GEO Store
                            (GEOADD, GEOPOS)
```

Drivers ప్రతి 4 సెకన్లకు GPS coordinates publish చేస్తారు. WebSocket gateway active orders కోసం location updates కు subscribe అయి customer device కు push చేస్తుంది. **Redis Pub/Sub** fan-out ను సమర్థవంతంగా handle చేస్తుంది.

### Estimated Delivery Time

EDT **multi-stage prediction**:

```
EDT = restaurant_prep_time + driver_pickup_travel + dropoff_travel + buffer

ఇక్కడ:
- restaurant_prep_time  → ప్రతి restaurant కు historical order data పై train చేసిన ML model
- driver_pickup_travel  → distance/traffic API (Google Maps, Mapbox)
- dropoff_travel        → అదే routing API
- buffer                → dynamic, వర్షం/peak hours లో పెరుగుతుంది
```

<ThinkAboutIt>పై matching algorithm fixed weights (w1–w4) ఉపయోగిస్తుంది. delivery speed మరియు driver satisfaction రెండింటినీ ఏకకాలంలో optimize చేసే learning system గా దీన్ని ఎలా evolve చేస్తారు?</ThinkAboutIt>

### Payment Processing

Payments **two-phase pattern** అనుసరిస్తాయి:

1. **Authorise** — order పెట్టినప్పుడు funds reserve చేయండి (hold, charge చేయకండి)
2. **Capture** — delivery confirmation తర్వాత వాస్తవ amount charge చేయండి

ఇది order modifications, cancellations, మరియు partial refunds ను సునాయాసంగా handle చేస్తుంది. Double-charging నివారించడానికి ప్రతి payment request పై **idempotency key** ఉపయోగించండి.

## దశ 5: Scaling మరియు Bottlenecks (10 నిమిషాలు)

### Database Scaling

- **Order Service**: `order_id` ద్వారా shard — orders స్వతంత్రం, horizontal scaling సులభం
- **Restaurant Service**: search queries కోసం read replicas, full-text మరియు geospatial search కోసం Elasticsearch
- **Driver Location**: Geospatial indexing తో Redis Cluster — పూర్తి in-memory, sub-millisecond reads

### Peak Load Handle చేయడం (Surge)

Peak times (శుక్రవారం 7pm, పెద్ద క్రీడా events) లో, order volume 10× spike అవ్వవచ్చు:

- CPU కాదు, queue depth ఆధారంగా application pods **auto-scaling**
- Supply మరియు demand balance చేయడానికి **surge pricing** — driver supply తక్కువగా ఉన్నప్పుడు delivery fees పెంచండి
- Peak సమయంలో abuse నివారించడానికి user కు **rate limiting**
- Downstream services పై **circuit breakers** — cascading failure కంటే graceful degradation

### Single Points of Failure

| భాగం | నివారణ |
|------|--------|
| API Gateway | Load balancer వెనుక బహుళ instances, health checks |
| Kafka | Replication factor 3 తో multi-broker cluster |
| Redis | Automatic failover కోసం Redis Sentinel |
| Payment provider | Automatic fallback తో secondary provider |

## 💡 ఇంటర్వ్యూ Tips

1. **మీ సమయాన్ని structure చేయండి** — requirements పై 30 నిమిషాలు ఖర్చు చేయకండి
2. **Trade-offs ను స్పష్టంగా చెప్పండి** — "Y కంటే X ఎంచుకున్నాను ఎందుకంటే..."
3. **సరళంగా మొదలుపెట్టి, తర్వాత scale చేయండి** — మొదట 1× load కోసం design, తర్వాత 100× address చేయండి
4. **మాట్లాడుతూ draw చేయండి** — visual communication evaluation లో సగం
5. **తెలియని విషయాలు acknowledge చేయండి** — "దీన్ని benchmark చేయాలి, కానీ నా hypothesis ఏమిటంటే..."
