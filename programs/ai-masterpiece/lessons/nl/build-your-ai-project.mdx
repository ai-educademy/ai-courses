# Bouw Je Eigen AI-Project

Je hebt maanden theorie gestudeerd. Nu is het tijd om **iets echts te bouwen**, het de wereld in te sturen en te bewijzen dat je een project van begin tot eind kunt opleveren.

Niets op je cv spreekt luider dan een werkend product dat mensen daadwerkelijk kunnen gebruiken.

<Illustration src="/images/lessons/build-your-ai-project.svg" alt="Diagram dat de reis toont van idee via bouwen, deployen en itereren" caption="De volledige levenscyclus van een AI-capstoneproject" />

## Het Juiste Project Kiezen

Het kiezen van het juiste project is de belangrijkste beslissing in dit capstone-traject. Maak de juiste keuze en de rest volgt vanzelf.

Je project moet op het snijvlak liggen van drie dingen:

- **Passie** — je gaat hier je avonden aan besteden; kies iets waar je oprecht om geeft.
- **Kennishiaat** — het moet je uitdagen op minstens één gebied dat je nog niet beheerst.
- **Portfoliowaarde** — een hiring manager moet denken: "Deze persoon kan opleveren."

<Callout>
Vermijd speelgoeddatasets en tutorial-klonen. Recruiters hebben al duizend MNIST-classifiers gezien. Los een probleem op dat ertoe doet voor een echt publiek, ook al is dat publiek klein.
</Callout>

## Je MVP Plannen

De grootste fout die bouwers maken is te breed scopen. Definieer je **Minimum Viable Product** meedogenloos:

1. **Alleen de kernfunctie** — één gebruikersstroom, één model, één interface.
2. **Techstack** — kies tools die je kent plus één nieuw ding. Voor de meeste projecten: Python + FastAPI backend, React of Next.js frontend, een managed database.
3. **Tijdlijn** — geef jezelf twee weken voor de MVP. Duurt het langer, dan is je scope te groot.
4. **Succesmetriek** — definieer één getal dat je vertelt of het project werkt (bijv. "retrieval-nauwkeurigheid boven 80 procent op mijn testset").

<ThinkAboutIt>
Schrijf je projectidee op in één zin. Schrap nu de helft van de features. Dat is je MVP.
</ThinkAboutIt>

## Bouwen in het Openbaar

Je voortgang delen op social media is geen ijdelheid — het is een carrièrestrategie.

- **Dag 1**: Post je idee en vraag om feedback.
- **Wekelijks**: Deel een korte update — wat je gebouwd hebt, wat er misging, wat je geleerd hebt.
- **Lanceringdag**: Schrijf een thread of blogpost waarin je de architectuur doorloopt.

In het openbaar bouwen creëert verantwoordelijkheid, trekt medewerkers aan en brengt je naam onder de aandacht van hiring managers nog vóórdat je solliciteert.

<FunFact>
Een onderzoek uit 2024 van Stack Overflow toonde aan dat 68 procent van de hiring managers de openbare projecten en posts van een kandidaat bekijkt voordat ze beslissen over een uitnodiging voor een gesprek.
</FunFact>

## Je Project Deployen

Een project dat alleen op `localhost` draait is onzichtbaar. Niemand kan het uitproberen, niemand kan het delen, en geen enkele hiring manager zal het ooit zien.

Deployment is geen leuke extra — het is het verschil tussen een hobbyproject en een portfoliostuk. Hier zijn drie solide opties:

| Platform | Ideaal Voor | Gratis Tier |
|----------|------------|-------------|
| **Vercel** | Next.js frontends, serverless API's | Ruim |
| **Railway** | Python backends, databases | 500 uur/maand |
| **HuggingFace Spaces** | ML-demo's met Gradio of Streamlit | Onbeperkt voor CPU |

Welke je ook kiest, zorg dat je README het volgende bevat:

- Een omschrijving in één zin van wat het project doet.
- Een live link of demo-GIF.
- Duidelijke installatie-instructies voor lokale ontwikkeling.

## Feedback Krijgen en Itereren

Zodra je MVP live staat, begint het echte leren. Opleveren is niet de finish — het is de start.

Zoek actief feedback van mensen die eerlijk tegen je zijn:

- Deel het in relevante Discord- en Slack-community's.
- Post op Reddit in domeinspecifieke subreddits.
- Vraag twee of drie engineers die je respecteert om het uit te proberen en eerlijke kritiek te geven.

Behandel elk stukje feedback als een cadeau. Los de kritieke problemen op, noteer de rest voor versie twee, en push updates snel.

<ThinkAboutIt>
Bedenk één community waar je doelgebruikers actief zijn. Plan om je project daar te delen binnen 48 uur na lancering.
</ThinkAboutIt>

## Toevoegen aan Je Portfolio

Je project is slechts zo waardevol als de presentatie ervan. Als je het aan je portfolio toevoegt:

1. **Schrijf een casestudy** — probleem, aanpak, architectuurdiagram, resultaten, geleerde lessen.
2. **Pin het op GitHub** — schone commitgeschiedenis, goede README, open issues voor toekomstig werk.
3. **Neem een demovideo van twee minuten op** — loop door het product en leg één technische beslissing uit.
4. **Link het overal** — cv, LinkedIn, persoonlijke site, GitHub-profiel-README.

## Je Volgende Stap

Stop met lezen. Open nu een nieuwe repository. Geef het de naam van je projectidee.

Schrijf eerst de README — beschrijf wat het gaat doen, voor wie het bedoeld is en wat je MVP-scope is. Push die README vanavond nog.

Het verschil tussen "Ik wil iets bouwen" en "Ik heb iets opgeleverd" is één commit.

<FunFact>
Gegevens van GitHub laten zien dat repositories met een goed geschreven README vier keer meer stars en forks ontvangen dan repositories zonder — eerste indrukken tellen, ook in open source.
</FunFact>
