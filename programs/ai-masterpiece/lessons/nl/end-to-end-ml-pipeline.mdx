# End-to-End ML Pipeline — Van Ruwe Data tot Gedeployd Model

De meeste tutorials stoppen bij `model.fit()`. In productie is dat amper 20% van het werk. Deze les doorloopt de **volledige ML-levenscyclus** — dezelfde pipeline die aanbevelingssystemen bij Spotify, fraudedetectie bij Stripe en zoekranking bij Google aandrijft.

## 🗺️ De ML-Levenscyclus in Vogelvlucht

<Illustration src="/images/lessons/end-to-end-ml-pipeline.svg" alt="Diagram van de ML-pipelinefases van data-ingestie tot monitoring" caption="De acht fases van een productie-ML-pipeline — elke fase is een potentieel faalpunt." />

Elk productie-ML-systeem doorloopt deze fases:

1. **Dataverzameling** — API's, databases, event streams, web scraping
2. **Data opschonen** — null-waarden afhandelen, deduplicatie, type-conversie
3. **Feature engineering** — ruwe signalen transformeren naar modelklare features
4. **Modelselectie** — architecturen kiezen die bij je probleem passen
5. **Training** — GPU/CPU-orkestratie, hyperparametertuning
6. **Evaluatie** — offline metrics, A/B-testing, fairness-audits
7. **Deployment** — serving-infrastructuur, latencybudgetten
8. **Monitoring** — driftdetectie, alerting, de retraining-trigger

## 📦 Dataverzameling en Versiebeheer

Ruwe data is je fundament. Behandel het als broncode — **versieer alles**.

```bash
# Initialiseer DVC in je repo
dvc init
dvc remote add -d storage s3://my-ml-data-bucket

# Track een dataset
dvc add data/transactions.csv
git add data/transactions.csv.dvc .gitignore
git commit -m "Track raw transaction dataset v1"
dvc push
```

**DVC** (Data Version Control) laat je elke historische dataset `git checkout`-en. Gecombineerd met Git-tags kun je elk experiment vanuit elk moment in de tijd reproduceren.

<FunFact>Netflix verwerkt dagelijks meer dan 1,5 biljoen events via hun datapipelines — zo'n 17 miljoen events per seconde die hun aanbevelingsmodellen voeden.</FunFact>

## 🛠️ Feature Engineering — Waar Wedstrijden Gewonnen Worden

Features zijn belangrijker dan algoritmes. Een gradient-boosted tree met briljante features verslaat een diep neuraal netwerk met slechte features.

```python
import pandas as pd

df = pd.read_csv("data/transactions.csv", parse_dates=["timestamp"])

# Temporal features
df["hour"] = df["timestamp"].dt.hour
df["is_weekend"] = df["timestamp"].dt.dayofweek >= 5

# Aggregation features
user_stats = df.groupby("user_id")["amount"].agg(["mean", "std", "count"])
user_stats.columns = ["user_avg_amount", "user_std_amount", "user_txn_count"]
df = df.merge(user_stats, on="user_id")

# Ratio features
df["amount_vs_avg"] = df["amount"] / df["user_avg_amount"]
```

## 🧪 Experiment Tracking met MLflow

Verlies nooit uit het oog wat je geprobeerd hebt. MLflow logt parameters, metrics en artefacten automatisch.

```python
import mlflow
from sklearn.ensemble import GradientBoostingClassifier
from sklearn.metrics import f1_score

mlflow.set_experiment("fraud-detection-v2")

with mlflow.start_run(run_name="gbm-baseline"):
    params = {"n_estimators": 500, "max_depth": 6, "learning_rate": 0.05}
    mlflow.log_params(params)

    model = GradientBoostingClassifier(**params)
    model.fit(X_train, y_train)

    score = f1_score(y_test, model.predict(X_test))
    mlflow.log_metric("f1_score", score)
    mlflow.sklearn.log_model(model, "model")
```

Vergelijk runs in de MLflow-UI en promoot het beste model naar de **Model Registry** met staging → productie lifecycle-tags.

## 🌐 Model Serving en Deployment

Je model heeft een API nodig. Twee dominante patronen:

**REST API** — eenvoudig, HTTP-gebaseerd, prima voor gemiddelde doorvoer:
```python
# FastAPI model server
from fastapi import FastAPI
import joblib

app = FastAPI()
model = joblib.load("model.pkl")

@app.post("/predict")
async def predict(features: dict):
    prediction = model.predict([list(features.values())])
    return {"prediction": int(prediction[0])}
```

**gRPC** — binair protocol, 2–10× sneller dan REST, ideaal voor interne microservices en scenario's met hoge doorvoer.

### Canary Deployments

Stuur nooit 100% van het verkeer naar een nieuw model. Gebruik **canary releases**: routeer 5% → 25% → 50% → 100%, en monitor foutpercentages bij elke stap. Rol direct terug als metrics verslechteren.

<Quiz question="Wat is het BELANGRIJKSTE voordeel van canary deployments voor ML-modellen?" options="Ze verkorten de modeltrainingstijd|Ze maken geleidelijke uitrol mogelijk met automatische rollback bij verslechterde metrics|Ze verbeteren modelnauwkeurigheid via A/B-testing|Ze elimineren de noodzaak van modelversiebeheer" answer="1" explanation="Canary deployments beperken de schaderadius. Als je nieuwe model slecht presteert, wordt slechts een klein percentage gebruikers getroffen en kun je terugdraaien voordat er bredere schade ontstaat." />

## 📊 Monitoring en de Feedbackloop

Deployment is niet de finish — het is het startschot. Monitor op:

- **Datadrift** — inputdistributies die verschuiven ten opzichte van trainingsdata
- **Conceptdrift** — de relatie tussen features en target die verandert
- **Prestatieverslechtering** — nauwkeurigheids-/latencymetrics die afnemen

Tools zoals **Evidently AI**, **WhyLabs** en **Arize** bieden driftdashboards. Wanneer drift de drempelwaarden overschrijdt, trigger je automatische hertraining met verse productiedata — en sluit je de feedbackloop.
