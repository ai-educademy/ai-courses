# ⚙️ MLOps and Production ML

Here's a statistic that should shape how you prepare for interviews: **87% of ML models never make it to production.** The gap between a Jupyter notebook and a reliable, scalable ML service is enormous — and companies are desperate for engineers who can bridge it. MLOps interview questions test whether you can take a model from "it works on my machine" to "it works for millions of users, reliably, at 3 AM on a Saturday."

---

## 🧭 What MLOps Actually Covers

MLOps is the set of practices that bring DevOps discipline to machine learning. It spans the entire ML lifecycle:

```
Data → Feature Engineering → Training → Evaluation → Deployment → Monitoring → Retraining
  ↑                                                                              |
  └──────────────────────────── Feedback Loop ────────────────────────────────────┘
```

Key pillars include:
- **CI/CD for ML**: automated testing of data, features, and models — not just code
- **Experiment tracking**: systematic recording of hyperparameters, metrics, and artefacts
- **Model versioning**: tracking which model version is serving which traffic
- **Model serving**: getting predictions to users reliably and fast
- **Monitoring**: detecting when models degrade in production

<Callout>
The biggest difference between MLOps and traditional DevOps: in software, code changes break things. In ML, *data* changes break things — often silently, without any error logs.
</Callout>

## 🚀 Model Serving Patterns

How you serve predictions depends on your latency requirements, scale, and how often predictions need to change.

### Batch Inference

Pre-compute predictions on a schedule (hourly, daily) and store them for lookup.

**Best for**: email recommendations, daily risk scores, weekly reports

```
Schedule → Load data → Run model → Write predictions to DB → Serve from DB
```

**Pros**: simple infrastructure, can use large models, cost-effective
**Cons**: predictions can be stale, can't respond to real-time context

### Real-Time Inference (Model-as-a-Service)

The model runs behind an API. Each request triggers a prediction.

**Best for**: search ranking, fraud detection, chatbots

```
User request → API Gateway → Feature lookup → Model inference → Response
```

**Pros**: fresh predictions, can use real-time context
**Cons**: latency-sensitive, need to optimise model size and serving infrastructure

### Embedded Model

The model is bundled directly into the application or device.

**Best for**: mobile apps, IoT devices, edge computing

**Pros**: no network latency, works offline
**Cons**: harder to update, limited by device resources

## 📊 Experiment Tracking

Running ML experiments without tracking is like doing science without a lab notebook. You need to record:

- **Hyperparameters**: learning rate, batch size, architecture choices
- **Metrics**: training loss, validation metrics at each epoch
- **Artefacts**: model checkpoints, feature importance plots, confusion matrices
- **Environment**: library versions, hardware used, random seeds

### Popular Tools

| Tool | Strengths | Deployment |
|---|---|---|
| **MLflow** | Open-source, model registry, wide adoption | Self-hosted or managed |
| **Weights & Biases (W&B)** | Beautiful UI, collaboration features, sweeps | Cloud-hosted |
| **Neptune.ai** | Lightweight, good for teams | Cloud-hosted |
| **TensorBoard** | Free, tightly integrated with TensorFlow | Local or embedded |

```python
import mlflow

mlflow.set_experiment("fraud-detection-v2")

with mlflow.start_run():
    mlflow.log_param("learning_rate", 0.001)
    mlflow.log_param("n_estimators", 500)

    model = train_model(X_train, y_train)
    metrics = evaluate_model(model, X_test, y_test)

    mlflow.log_metric("auc_roc", metrics["auc_roc"])
    mlflow.log_metric("precision", metrics["precision"])
    mlflow.sklearn.log_model(model, "model")
```

## 🔍 Monitoring ML in Production

Traditional software monitoring tracks errors and latency. ML monitoring adds a critical layer: **is the model still making good predictions?**

### Types of Drift

**Data drift** (covariate shift): the distribution of input features changes. Example: an e-commerce model trained on pre-pandemic data sees a surge in home office equipment searches.

**Concept drift**: the relationship between features and target changes. Example: a credit scoring model's "employment status" feature means something different during a recession.

**Prediction drift**: the distribution of model outputs changes, even if inputs look similar.

### What to Monitor

| Signal | How to Detect | Action |
|---|---|---|
| Input feature distributions | Statistical tests (KS test, PSI) | Investigate data pipeline |
| Prediction distribution | Track mean, variance, percentiles over time | Compare to baseline period |
| Model performance (when labels available) | Track accuracy, F1, etc. against ground truth | Trigger retraining if below threshold |
| Latency | P50, P95, P99 response times | Scale infrastructure or optimise model |
| Data quality | Null rates, schema violations, volume anomalies | Alert data engineering team |

```python
from scipy.stats import ks_2samp

# Kolmogorov-Smirnov test for distribution shift
def detect_drift(reference_data, current_data, threshold=0.05):
    statistic, p_value = ks_2samp(reference_data, current_data)
    return p_value < threshold  # True = drift detected
```

<FunFact>
When Zillow's "Zestimate" home-pricing model experienced concept drift during 2021's unusual housing market, the company lost over $500 million on home purchases guided by its own model's predictions — and ultimately shut down its home-flipping business.
</FunFact>

## 🔄 CI/CD for Machine Learning

ML CI/CD goes beyond code testing. You need to validate data, features, and models too.

### The ML Testing Pyramid

```
              ┌──────────────┐
              │  Model Tests  │  ← Does the model meet quality thresholds?
             ┌┴──────────────┴┐
             │  Feature Tests  │  ← Are features computed correctly?
            ┌┴────────────────┴┐
            │  Data Validation  │  ← Is the data schema correct? Missing values?
           ┌┴──────────────────┴┐
           │    Unit Tests       │  ← Does the code work?
           └────────────────────┘
```

**Data validation**: check schema, null rates, value ranges, row counts
**Feature tests**: verify feature transformations produce expected outputs
**Model tests**: ensure the trained model exceeds minimum quality thresholds before deployment

## 💰 Infrastructure Considerations

Production ML has real costs. Interviewers want to see cost awareness:

### GPU Management

- **Training**: use spot/preemptible instances for cost savings (with checkpointing!)
- **Serving**: consider model distillation or quantisation to reduce GPU requirements
- **Autoscaling**: scale GPU instances based on traffic patterns, not peak capacity

### Cost Optimisation Strategies

- **Model compression**: pruning, quantisation, knowledge distillation
- **Caching**: cache predictions for repeated inputs
- **Tiered serving**: use a cheap model for most requests, expensive model only when confidence is low
- **Right-sizing**: don't serve a 175B parameter model when a fine-tuned 7B model achieves 95% of the quality

```
Cost per prediction ≈ (GPU cost per hour) / (predictions per hour)

Example:
- GPU instance: $3/hour
- Model throughput: 1,000 predictions/second
- Cost per prediction: $3 / 3,600,000 ≈ $0.0000008
```

## 🏗️ Common Interview Questions

Here are questions you should be prepared to discuss:

1. "Walk me through how you'd deploy a new model to production" — describe the full pipeline from training to canary deployment to full rollout
2. "How would you handle a model that's degrading in production?" — monitoring, root cause analysis, rollback, retraining
3. "What's the difference between model retraining and model updating?" — full retraining on new data vs incremental/online learning
4. "How do you ensure reproducibility in ML experiments?" — version control for code, data, and config; fixed seeds; containerisation
5. "Describe your ideal ML platform" — feature store, experiment tracker, model registry, serving infrastructure, monitoring dashboard

<Callout>
When answering MLOps interview questions, always connect technical decisions to business impact. "We use canary deployments because rolling back a bad model costs X in lost revenue" is more compelling than describing the technical setup alone.
</Callout>

## 🔑 Key Takeaways

- **MLOps bridges the gap** between model development and production — this is the most in-demand ML skill set today
- **Choose serving patterns wisely**: batch for cost, real-time for freshness, embedded for edge
- **Monitor beyond latency**: data drift, concept drift, and model degradation are the silent killers
- **Track every experiment**: parameters, metrics, artefacts, and environment
- **Test the ML pipeline**, not just the code: data validation, feature tests, and model quality gates
- **Think about cost**: model compression, caching, and tiered serving reduce infrastructure bills
